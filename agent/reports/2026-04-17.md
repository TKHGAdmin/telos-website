## BUG-2026-04-17-01 - Deleting a client orphans the `client_email:` lookup and blocks reuse

- **Severity:** P1
- **File:** `api/dashboard/clients.js:143-154`
- **Related:** `api/dashboard/client-portal.js:115-124` (where the index is written), `api/dashboard/clients.js:70-73` (duplicate check that then fails)

### What's wrong

The admin `DELETE /api/dashboard/clients?id=N` handler removes only two keys:

```js
await redis('DEL', 'client:' + id);
await redis('ZREM', 'clients_index', String(id));
```

It never removes the `client_email:{normalizedEmail}` lookup record that `client-portal.js` POST created when the portal was enabled (line 117). The record points at a now-nonexistent client id.

### Reproduction

1. Admin creates a client with email `foo@example.com`.
2. Admin enables the portal in the Client Portal tab → `client-portal.js` writes `client_email:foo@example.com = <id>`.
3. Admin deletes the client from the Clients tab.
4. Admin tries to re-create (or create a new) client with email `foo@example.com`.
5. `clients.js:70` finds the orphan lookup and returns `400 "Email already in use by another client"` - but there is no such client anymore.

Separately, `client_dailylog:*`, `client_training_log:*`, `client_545_*`, `client_nutrition_plan:*`, `client_sidemenu:*`, `client_supplement_*`, `client_push_sub:*`, etc. also leak on delete, but those do not block any user-facing flow the way the email index does.

### Impact

Admin-visible. The admin cannot re-create a client with a recycled email without manually deleting the Redis key. Error message is misleading (implies a duplicate that does not exist). This is likely to recur any time a client is deleted and later rejoins, or whenever the admin deletes a test record.

### Suggested fix

In the DELETE branch, read the client record before deletion, normalize the email, and `DEL client_email:{normalizedEmail}` if portal was enabled. Same shape as the disable-portal flow in `client-portal.js:121-124`.

## BUG-2026-04-17-02 - `engagement-check` cron spams inactive clients with a reminder every day

- **Severity:** P1
- **File:** `api/cron/engagement-check.js:61-93`
- **Related:** `vercel.json:23-26` (schedule)

### What's wrong

The cron runs **daily** at 15:00 UTC and emails any active client whose last daily log is 3+ days old (or missing across the 5-day lookback). There is no "last reminder sent" marker and no cooldown. A client who stops logging on Monday will receive:

- Thursday (day 3) - reminder email
- Friday (day 4) - reminder email
- Saturday (day 5) - reminder email
- Sunday, Monday, Tuesday... - reminder email every single day

There is no unsubscribe link in the email body, only the global `client.emailEnabled === false` flag (line 39) which the admin has to toggle manually.

### Reproduction

1. Ensure a client has `status: 'active'`, `portalEnabled: true`, no `emailEnabled: false` override.
2. Do not log in or submit a daily check-in for 5+ days.
3. Inspect `RESEND_API_KEY` send logs (or add a local `console.log`) - one send per day once the 3-day threshold is crossed, indefinitely.

### Impact

- Real risk of clients marking the address as spam (hurts `noreply@telosathleticclub.com` deliverability for everyone).
- Bad UX: a disengaged client getting 7 "we miss you" emails in a week is likely to unsubscribe or churn faster, not re-engage.
- Quiet cost risk on Resend volume if the client roster grows.

### Suggested fix

Add a per-client cooldown. Cheapest: `SETEX engagement_reminder:{clientId} 345600 1` (4-day TTL) after a send, and skip if the key exists. Or write `client.lastEngagementReminderAt` into the client record and skip if within the last N days. Tier the reminders (e.g., day 3, day 7, day 14) instead of daily.

## BUG-2026-04-17-03 - Admin HMAC signature comparison is not timing-safe (regression vs client auth)

- **Severity:** P2
- **File:** `api/lib/auth.js:27`
- **Related:** `api/lib/client-auth.js:46-49` (the correct pattern)

### What's wrong

```js
// api/lib/auth.js
const expected = crypto.createHmac('sha256', SESSION_SECRET).update(payload).digest('hex');
if (signature !== expected) return false;   // <-- plain string compare
```

vs. the client equivalent:

```js
// api/lib/client-auth.js
var a = Buffer.from(signature, 'hex');
var b = Buffer.from(expected, 'hex');
if (a.length !== b.length) return null;
if (!crypto.timingSafeEqual(a, b)) return null;
```

The admin path is the only gate between a stolen/guessed session token and every `/api/dashboard/*` endpoint (clients, revenue, ad spend, Charleston applications, etc.). CLAUDE.md explicitly advertises "timing-safe password check" for the admin dashboard; the HMAC check is the other half of that guarantee and is inconsistent.

### Why it's worth fixing even though HMAC is unpredictable

- Exploiting a 256-char hex HMAC over the network is extremely hard, so this is not an urgent P0. But the admin password check (`verifyPassword` at line 52) already does a `crypto.timingSafeEqual` - the HMAC compare is the odd one out.
- The client auth module got this right; the admin module was presumably written earlier and never updated. Easy fix that aligns the two.

### Suggested fix

Mirror `client-auth.js`: convert both strings to Buffers, length-check, then `crypto.timingSafeEqual`.
