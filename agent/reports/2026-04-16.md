## Summary

First run of the Telos Bug Hunter. Today's focus is Functional. I walked the critical paths in the admin client-management flow and the client portal auth stack: `api/dashboard/clients.js`, `api/dashboard/client-portal.js`, `api/lib/auth.js`, `api/lib/client-auth.js`, `api/client/login.js`, `api/client/reset-password.js`, plus the public form endpoints (`submit-quiz`, `submit-email`, `submit-chs-application`) and the logged-data endpoints (`daily-log`, `training-log`, `nutrition-log`, `five-four-five`, `activity-log`, `module-progress`). Two real functional bugs found, both clustered in the client-email lookup: one allows an admin action to silently lock an active client out of their own portal, the other blocks re-using an email after deleting a client.

## BUG-20260416-01 - Enabling portal on a duplicate-email client hijacks the other client's login lookup

- **Severity**: P1
- **Area**: api/dashboard/clients.js, api/dashboard/client-portal.js
- **File(s)**:
  - api/dashboard/clients.js (POST lines 60-96, PUT lines 100-140)
  - api/dashboard/client-portal.js (POST lines 89-136)
  - api/lib/client-auth.js (normalizeEmail, verifyClientToken)
- **Description**: Email uniqueness across clients is enforced only through the `client_email:{normalizedEmail}` Redis lookup, which is created/refreshed exclusively when `portalEnabled` is true. Both the POST (create) and PUT (edit) paths in `clients.js` therefore accept a duplicate email whenever the colliding client has their portal disabled. When the admin later enables the portal for the duplicate, `client-portal.js` POST does a blind `SET client_email:{email} = {newClientId}` on line 117, overwriting the lookup that points at the original, active client. The original client immediately loses the ability to log in - their email resolves to the wrong client record and `verifyPassword` returns false because the password hashes differ - producing a generic "Invalid email or password" response with no indicator of what actually happened.
- **Reproduction**:
  1. Create Client A with email `alice@example.com` via `POST /api/dashboard/clients`, then enable portal via `POST /api/dashboard/client-portal` with `{ clientId: A.id, enabled: true, password: "Secret123!" }`. Verify Client A can log in.
  2. Create Client B with the same email `alice@example.com` via `POST /api/dashboard/clients` - it succeeds because `clients.js:70` only checks the `client_email` lookup, and B's portal is not yet enabled, so no uniqueness collision is detected during creation.
  3. Enable portal for Client B via `POST /api/dashboard/client-portal` with `{ clientId: B.id, enabled: true, password: "Different456!" }`. This hits `client-portal.js:115-118` which runs `SET client_email:alice@example.com = B.id`, silently overwriting the lookup.
  4. Client A now cannot log in. `POST /api/client/login` with Client A's email+password returns 401 "Invalid email or password" because the email maps to B's record, which has a different password hash.
  - The PUT flow has the same failure mode: changing Client B's email to `alice@example.com` in `clients.js:100` only enforces uniqueness inside the `if (... && client.portalEnabled)` branch at line 121, so the duplicate sails through whenever portal is disabled at the moment of the edit.
- **Expected**: Email uniqueness is enforced on every write path that sets `client.email`, independent of `portalEnabled`. Enabling a portal for a client whose email already belongs to another client returns a 400 rather than silently overwriting the lookup.
- **Actual**: Uniqueness is enforced only when `portalEnabled` is true, and the portal-enable handler performs an unconditional `SET` on the lookup. An admin action can silently deactivate another client's login.
- **Suggested fix**: In `clients.js` POST/PUT, check `client_email:{normalized}` for any existing owner whenever `body.email` is set or changes, regardless of `portalEnabled`. In `client-portal.js:115-118`, before the `SET`, fetch the current lookup value and refuse (return 400) if it is set to a different `clientId`.

## BUG-20260416-02 - Deleting a client leaves the client_email lookup orphaned, blocking email re-use

- **Severity**: P2
- **Area**: api/dashboard/clients.js
- **File(s)**:
  - api/dashboard/clients.js (DELETE handler, lines 143-154)
- **Description**: The `DELETE /api/dashboard/clients` handler removes `client:{id}` and the `clients_index` ZSET membership but does not remove the `client_email:{normalizedEmail}` reverse lookup written by `client-portal.js` on portal enable. The orphaned key still points at the now-missing client ID. When the coach later tries to re-add the same person (returning client, accidental deletion, data cleanup), `clients.js` POST at line 70 reads the stale lookup and returns 400 "Email already in use by another client" - even though no client actually owns that email. There is no UI-exposed path to clear the stale key, so the coach has to either invent a new email or have someone touch Redis directly.
- **Reproduction**:
  1. Create a client with email `returning@example.com` and enable their portal (writes `client_email:returning@example.com = id`).
  2. `DELETE /api/dashboard/clients?id={id}` - returns 200.
  3. `POST /api/dashboard/clients` with `{ name: "Returning Client", email: "returning@example.com", tier: "growth" }`.
  4. API responds 400 "Email already in use by another client" despite there being no active client with that email. `GET /api/dashboard/clients` confirms no client with that email exists.
- **Expected**: Deleting a client removes their `client_email:{normalizedEmail}` lookup so the email is free for re-use.
- **Actual**: The orphaned lookup survives the delete and blocks the next create.
- **Suggested fix**: In the DELETE handler, fetch the client record before deleting it (`GET client:{id}`), and if `client.email` is present also run `DEL client_email:{normalizeEmail(client.email)}` in the same pipeline. Same hygiene would apply to the other per-client keys (`client_dailylog*`, `client_training_log*`, `client_545_*`, `client_nutrition_log*`, `client_supplement_*`, `client_activity_log*`, `client_push_sub`, `client_modules_watching`, `client_modules_completed`), though those are storage/noise issues rather than user-visible functional breakage.

## Notes

- `api/client/reset-password.js` has no rate limiting on the "request reset" path. An attacker can flood any client's inbox with reset emails (and burn Resend quota) by hitting `POST /api/client/reset-password {"email":"..."}` in a loop. Flagging for the Security rotation (day 3) rather than filing today - the focus rule is Functional only.
- `api/cron/engagement-check.js` re-sends the "we miss you" email every single day to a client that has been inactive 3+ days, with no cooldown. CLAUDE.md describes this as intentional daily behavior, so treating as design not a bug, but worth confirming with Thomas - a client inactive 30 days would receive 28 consecutive reminder emails.
- `api/client/module.js:50` initializes `minTier = 999` and falls back to `'lifestyle_plus'` when a module has an empty `tierAccess` array. A module saved with `tierAccess: []` is therefore inaccessible to every client and surfaces a misleading "requiredTier: lifestyle_plus" error. Not filed because the admin default in `modules.js:78` pre-populates `tierAccess` with all tiers - only an admin who manually clears the array can trigger it.
- Codebase has no test suite or lockfile. Future Functional runs will continue to rely on static code trace rather than `npm test` / `tsc --noEmit`.
