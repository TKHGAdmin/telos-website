Traced the critical paths in admin client management and the client portal auth stack (`api/dashboard/clients.js`, `api/dashboard/client-portal.js`, `api/lib/auth.js`, `api/lib/client-auth.js`, `api/client/login.js`, `api/client/reset-password.js`), plus the public form endpoints (`submit-quiz`, `submit-email`, `submit-chs-application`) and the logged-data endpoints (`daily-log`, `training-log`, `nutrition-log`, `five-four-five`, `activity-log`, `module-progress`). ## BUG-2026-04-16-001

- **Severity:** P1
- **Status:** pending-review
- **Area:** Admin / Client auth
- **File(s):** `api/dashboard/clients.js:60-96`, `api/dashboard/clients.js:100-140`, `api/dashboard/client-portal.js:89-136`
- **First seen:** 2026-04-16

### What's wrong

Email uniqueness across clients is enforced only through the `client_email:{normalizedEmail}` Redis key, which is created or refreshed exclusively when `portalEnabled` is true. Both the create (POST) and edit (PUT) paths in `clients.js` therefore accept a duplicate email whenever the colliding client has their portal disabled. When the admin later enables the portal for the duplicate, `client-portal.js:115-118` performs an unconditional `SET client_email:{email} = {newClientId}`, silently overwriting the lookup that still belongs to the original, active client. The original client can no longer log in: their email resolves to the wrong client record, `verifyPassword` returns false because the hashes differ, and the login endpoint returns a generic "Invalid email or password" with no indicator that an admin action just locked them out.

### How to reproduce

1. Create Client A with email `alice@example.com` via `POST /api/dashboard/clients`, then enable portal via `POST /api/dashboard/client-portal` with `{ clientId: A.id, enabled: true, password: "Secret123!" }`. Confirm Client A can log in at `/client-dashboard`.
2. Create Client B with the same email `alice@example.com` via `POST /api/dashboard/clients`. This succeeds because `clients.js:70` only checks the `client_email` lookup, and B's portal is not yet enabled so no lookup collision is detected during creation.
3. Enable portal for Client B via `POST /api/dashboard/client-portal` with `{ clientId: B.id, enabled: true, password: "Different456!" }`. This hits `client-portal.js:115-118` which runs `SET client_email:alice@example.com = B.id`, silently overwriting the lookup.
4. Client A attempts `POST /api/client/login` with their email and password. Response: 401 "Invalid email or password" because the email now resolves to B's record, which has a different password hash.

The PUT path in `clients.js` has the same failure mode: changing Client B's email to `alice@example.com` only runs its uniqueness check inside `if (... && client.portalEnabled)` at line 121, so a duplicate sails through whenever portal is disabled at the moment of the edit.

### Suggested fix

Enforce uniqueness on every write path that sets `client.email`, regardless of `portalEnabled`, and make the portal-enable handler refuse to overwrite a lookup that belongs to a different client.

```js
// api/dashboard/clients.js POST - check uniqueness unconditionally
var normalized = normalizeEmail(body.email);
var existingEmailOwner = await redis('GET', 'client_email:' + normalized);
if (existingEmailOwner) {
  return res.status(400).json({ error: 'Email already in use by another client' });
}

// api/dashboard/clients.js PUT - drop the `&& client.portalEnabled` guard on the uniqueness check
if (body.email !== undefined && body.email !== existingRecord.email) {
  var newNorm = normalizeEmail(body.email);
  var emailOwner = await redis('GET', 'client_email:' + newNorm);
  if (emailOwner && String(emailOwner) !== String(client.id)) {
    return res.status(400).json({ error: 'Email already in use by another client' });
  }
  // ... existing DEL old, SET new lookup logic, but gate on portalEnabled only for the actual SET
}

// api/dashboard/client-portal.js POST - pre-read before overwriting
if (client.portalEnabled && client.email) {
  var email = normalizeEmail(client.email);
  var currentOwner = await redis('GET', 'client_email:' + email);
  if (currentOwner && String(currentOwner) !== String(client.id)) {
    return res.status(400).json({ error: 'Email already in use by another client' });
  }
  await redis('SET', 'client_email:' + email, String(client.id));
}
```

### Evidence

- `api/dashboard/clients.js:70` - POST uniqueness check relies on `client_email` lookup which only exists for portal-enabled clients
- `api/dashboard/clients.js:121` - PUT uniqueness check is gated on `&& client.portalEnabled`
- `api/dashboard/client-portal.js:115-118` - unconditional `SET` of `client_email`, no check that current value belongs to the same client
- `api/client/login.js:24-48` - login resolves email → id → record → hash, so a stale lookup produces a silent auth failure rather than an error

## BUG-2026-04-16-002

- **Severity:** P2
- **Status:** pending-review
- **Area:** Admin / Client management
- **File(s):** `api/dashboard/clients.js:143-154`
- **First seen:** 2026-04-16

### What's wrong

`DELETE /api/dashboard/clients` removes the `client:{id}` record and the `clients_index` ZSET membership, but does not remove the `client_email:{normalizedEmail}` reverse lookup written by `client-portal.js` when the portal was first enabled. The orphaned key still points at the now-missing client ID. When the coach later tries to re-add the same person (returning client, accidental deletion, data cleanup), `clients.js:70` reads the stale lookup and rejects the create with 400 "Email already in use by another client" - even though no client actually owns that email. There is no UI path to clear the stale key, so the coach has to invent a new email or touch Redis directly.

### How to reproduce

1. `POST /api/dashboard/clients` with email `returning@example.com`, then enable the portal via `POST /api/dashboard/client-portal` with `{ clientId, enabled: true }`. This writes `client_email:returning@example.com = {id}`.
2. `DELETE /api/dashboard/clients?id={id}` - responds 200, record gone from the Clients table.
3. `POST /api/dashboard/clients` with `{ name: "Returning Client", email: "returning@example.com", tier: "growth" }`.
4. API responds 400 "Email already in use by another client". `GET /api/dashboard/clients` confirms no client with that email exists.

### Suggested fix

Fetch the client record inside the DELETE handler, capture the email, and tear down the reverse lookup alongside the primary record.

```js
// api/dashboard/clients.js DELETE
var existing = await redis('GET', 'client:' + id);
if (existing) {
  var client = JSON.parse(existing);
  if (client.email) {
    await redis('DEL', 'client_email:' + normalizeEmail(client.email));
  }
}
await redis('DEL', 'client:' + id);
await redis('ZREM', 'clients_index', String(id));
```

Same hygiene should eventually apply to the per-client data keys (`client_dailylog:*`, `client_training_log:*`, `client_545_*`, `client_nutrition_log:*`, `client_supplement_*`, `client_activity_log:*`, `client_push_sub:*`, `client_modules_watching:*`, `client_modules_completed:*`), but those are storage-noise issues rather than user-visible functional breakage, so this fix only has to cover `client_email:*` to resolve the reported behavior.

### Evidence

- `api/dashboard/clients.js:143-154` - DELETE body only touches `client:{id}` and `clients_index`
- `api/dashboard/client-portal.js:115-118` - the write that creates `client_email:{normalized}` in the first place
- `api/dashboard/clients.js:68-73` - the POST uniqueness gate that rejects the re-create using the now-orphaned lookup

## Notes

- `api/client/reset-password.js` has no rate limiting on the "request reset" path. An attacker can flood any client's inbox with reset emails (and burn Resend quota) by hitting `POST /api/client/reset-password {"email":"..."}` in a loop. Flagging for the Security rotation (day 3) rather than filing today - today's focus is Functional only.
- `api/cron/engagement-check.js` re-sends the "we miss you" email every single day to a client inactive 3+ days, with no cooldown. CLAUDE.md describes this as intentional daily behavior, so treating as design not a bug, but worth confirming with Thomas - a client inactive 30 days receives 28 consecutive reminder emails.
- `api/client/module.js:50` initializes `minTier = 999` and falls back to `'lifestyle_plus'` when a module has an empty `tierAccess` array. A module saved with `tierAccess: []` is therefore inaccessible to every client with a misleading error. Not filed because the admin default in `modules.js:78` pre-populates `tierAccess` with all tiers - only an admin who manually clears the array can trigger it.
- Codebase has no test suite or lockfile. Future Functional runs will continue to rely on static code trace rather than `npm test` / `tsc --noEmit`.
