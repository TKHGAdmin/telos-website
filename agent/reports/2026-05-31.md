## BUG-20260531-01 — Admin login has no rate limiting (brute-forceable)

- **Severity:** P1
- **Area:** API / auth
- **File:** `api/dashboard/login.js:1-17`, `api/lib/auth.js:52-59`
- **Description:** `POST /api/dashboard/login` accepts unlimited password attempts from any IP. `verifyPassword` uses `crypto.timingSafeEqual` so timing attacks are mitigated, but the dashboard password (`DASHBOARD_PASSWORD`) is a single shared secret with no lockout, no per-IP throttle, and no CAPTCHA. The `/thomas` dashboard exposes every client's PII, training logs, nutrition data, revenue, leads, and Charleston applications — a successful brute-force compromises the entire business. Vercel's function ceiling (60s `maxDuration` per request) does not slow attackers materially.
- **Reproduction:** `for i in {1..1000}; do curl -s -X POST https://www.telosathleticclub.com/api/dashboard/login -H 'Content-Type: application/json' -d '{"password":"guess'$i'"}'; done` returns 1000 × `401`. No `Retry-After`, no cooldown, no log of the burst.
- **Suggested fix:** Add an Upstash-backed rate limit on `/api/dashboard/login` keyed by client IP (use `x-real-ip` — see BUG-04). E.g. `INCR ratelimit:dashlogin:<ip>` with `EXPIRE 900`, reject after 5 attempts in 15 min. Optionally add a slow exponential backoff (`SETEX block:<ip> 60*2^attempts`) and surface the lockout in the response.

## BUG-20260531-02 — Client login has no rate limiting (per-IP and per-account)

- **Severity:** P1
- **Area:** API / auth
- **File:** `api/client/login.js:9-60`
- **Description:** `POST /api/client/login` has no throttle. PBKDF2 at 100k iterations slows each attempt server-side (~50–150ms of CPU) but does not stop a dictionary attack against a known client email — common-password lists run in minutes. Because the portal exposes coach-set training programs, nutrition plans, journal prompts, and the client's own health logs (weight, sleep, mood, stress), each compromised account is a real privacy breach. Anyone with one client's email (leakable from past client testimonials, social media, or by guessing) can run a credential-stuffing attack.
- **Reproduction:** Repeated POSTs to `/api/client/login` with the same email and different passwords always reach `verifyPassword` and return a 401 — no 429, no incremental delay.
- **Suggested fix:** Per-IP limit (e.g. 10 failures / 15 min) AND per-email limit (e.g. 5 failures / 15 min) keyed on `normalizeEmail(body.email)`. Reset the counter on successful login. Same Upstash `INCR` + `EXPIRE` pattern used in the public submit endpoints.

## BUG-20260531-03 — Password-reset endpoint has no rate limit (email bombing + Resend quota burn)

- **Severity:** P2
- **Area:** API / auth / abuse
- **File:** `api/client/reset-password.js:19-67`
- **Description:** The "request a reset email" branch (`body.email && !body.token`) calls the Resend API every time a valid client email is submitted, with no per-IP or per-target rate limit. An attacker who knows one client's email can trigger unlimited reset emails to that mailbox, harassing the client and burning the project's Resend quota / sending reputation. The "always return success to prevent enumeration" pattern (line 25) is good — but without a rate limit the response timing itself leaks valid emails (the valid-email branch awaits a Resend HTTP round-trip, the invalid branch returns immediately).
- **Reproduction:** `for i in {1..50}; do curl -s -X POST .../api/client/reset-password -H 'Content-Type: application/json' -d '{"email":"<known-client>@example.com"}'; done` — 50 reset emails delivered, no throttle.
- **Suggested fix:** Two layers: (a) per-IP cap (e.g. 5 reset requests/hour) keyed on `x-real-ip`; (b) per-email cap (e.g. 3 reset emails/hour) keyed on `normalizeEmail(body.email)` — applied AFTER the existence check so it doesn't help enumeration. Both can use the same `ratelimit:` Redis pattern already in use elsewhere.

## BUG-20260531-04 — Rate-limit key uses spoofable `x-forwarded-for` (Vercel-trusted header is `x-real-ip`)

- **Severity:** P2
- **Area:** API / abuse
- **File:** `api/submit-quiz.js:23`, `api/submit-email.js:28`, `api/submit-chs-application.js:40`
- **Description:** All three public submission endpoints derive the rate-limit key from `req.headers['x-forwarded-for'] || req.headers['x-real-ip']`. On Vercel, an inbound `X-Forwarded-For` header from the client is preserved in the chain — the final header an Edge Function sees can look like `"<client-spoofed>, <real-edge-source>"`. Because the rate-limit key uses the full string, the attacker just varies the spoofed prefix on each request to get a fresh bucket. Vercel's `x-real-ip` is the platform-trusted single client IP and is not modifiable by the caller. Same risk applies to any future endpoint that copies this pattern.
- **Reproduction:** `for i in {1..50}; do curl -s -X POST .../api/submit-quiz -H 'Content-Type: application/json' -H "X-Forwarded-For: 10.0.0.$i" -d '{"name":"x","email":"x@x.x"}'; done` — all 50 succeed with `{"ok":true}` (rate limit is 10/hr per IP). Also fills `quiz_submissions` ZSET with junk.
- **Suggested fix:** Prefer `req.headers['x-real-ip']` first, fall back to the **leftmost** token of `x-forwarded-for` validated as an IP. Cleanest:
  ```js
  const ip = req.headers['x-real-ip']
    || (req.headers['x-forwarded-for'] || '').split(',')[0].trim()
    || 'unknown';
  ```

## BUG-20260531-05 — Admin session signature compared with `!==` (non-timing-safe), inconsistent with client-auth

- **Severity:** P3
- **Area:** API / auth
- **File:** `api/lib/auth.js:23-27`
- **Description:** `verifyToken` in the admin auth lib does `if (signature !== expected) return false;`. The sister file `api/lib/client-auth.js:46-49` correctly uses `crypto.timingSafeEqual` on the HMAC. While timing attacks against HMAC validation over the network are largely theoretical, the fix is trivial and the inconsistency suggests a missed update. Worth fixing to keep the two libs aligned and to make the audit cleaner.
- **Suggested fix:**
  ```js
  const sigBuf = Buffer.from(signature, 'hex');
  const expBuf = Buffer.from(expected, 'hex');
  if (sigBuf.length !== expBuf.length) return false;
  if (!crypto.timingSafeEqual(sigBuf, expBuf)) return false;
  ```
  Same shape as `client-auth.js`.

## BUG-20260531-06 — `/thomas` admin dashboard has no frame-protection header (clickjacking surface)

- **Severity:** P3
- **Area:** Infra / headers
- **File:** `vercel.json:1-28`
- **Description:** `vercel.json` sets only CORS headers on `/api/submit-*`. No `X-Frame-Options`, no `Content-Security-Policy: frame-ancestors`, no `Strict-Transport-Security` are configured anywhere. The admin dashboard at `/thomas` can therefore be embedded in any attacker-controlled iframe and overlaid for clickjacking (e.g. tricking a logged-in admin into clicking "Delete client" or "Send to all"). Note: the **client dashboard** is intentionally embedded in Whop iframes (per CLAUDE.md), so a blanket policy won't work — this needs to be scoped to `/thomas` only.
- **Suggested fix:** Add a path-scoped header block in `vercel.json`:
  ```json
  {
    "source": "/thomas",
    "headers": [
      { "key": "X-Frame-Options", "value": "DENY" },
      { "key": "Content-Security-Policy", "value": "frame-ancestors 'none'" },
      { "key": "Referrer-Policy", "value": "no-referrer" }
    ]
  }
  ```
  Also worth adding `Strict-Transport-Security: max-age=63072000; includeSubDomains; preload` at the root once you're comfortable committing to HTTPS-only forever.
