## Summary

Bootstrap run. Reviewed auth, public form endpoints, cron jobs, and admin/client API routes. Two real bugs in client-lifecycle flows: the daily engagement-reminder cron has no suppression and will email lapsed clients every single day, and the dashboard client-delete path orphans the email-to-id lookup so the same email can never be reused.

## Findings

### [P1] Engagement-check cron emails inactive clients every day with no suppression

- **ID:** 2026-05-10-01
- **Area:** api/cron/engagement-check.js:61, vercel.json:25
- **Reproduction:** Cron `0 15 * * *` runs `engagement-check` daily. The handler loops active clients, looks up the last 5 days of `client_dailylog:{id}:{date}`, and sends a reminder email when `lastLogDaysAgo === -1 || lastLogDaysAgo >= 3`. Nothing is written to Redis to record that an email was sent — there is no `lastReminderAt`, no per-client cooldown, no per-day suppression. A client who stops logging on Monday will receive a reminder on Thursday, Friday, Saturday, Sunday, Monday... indefinitely until they either log a daily entry or the coach flips `emailEnabled = false` on the client record.
- **Impact:** Daily-spam churn risk for the most fragile segment (clients already disengaging). Resend deliverability reputation also takes a hit if recipients mark the repeated emails as spam. Active impact scales linearly with the active-but-inactive client count.
- **Suggested fix:** After a successful Resend call, set a key like `client_engagement_sent:{id}` with `EX 86400 * 6` (or write `lastEngagementEmailAt` onto the client record) and skip the send when it's present. Alternatively, only send when `lastLogDaysAgo` equals exactly 3 and 7 (two-shot cadence) instead of every day from day 3 onward.
- **Confidence:** high

### [P2] Deleting a client leaves the email-to-id lookup behind, blocking reuse of the email

- **ID:** 2026-05-10-02
- **Area:** api/dashboard/clients.js:143-154 (DELETE), interacts with api/dashboard/client-portal.js:114-118 (where `client_email:{normalized}` is created)
- **Reproduction:**
  1. POST `/api/dashboard/clients` to create client A with `email = foo@bar.com`.
  2. PUT/POST `/api/dashboard/client-portal` with `enabled: true` to register `client_email:foo@bar.com -> A.id`.
  3. DELETE `/api/dashboard/clients?id=A.id`. The handler runs `DEL client:{id}` and `ZREM clients_index`, but never touches `client_email:foo@bar.com`.
  4. POST `/api/dashboard/clients` again with the same email. The pre-check at clients.js:70-73 finds the orphaned lookup and returns `400 "Email already in use by another client"`, even though no client with that id exists.
- **Impact:** Coach-visible papercut. Any client deleted from the dashboard cannot be re-onboarded under the same email — the only workarounds are manual Redis surgery or a different email address. Also leaves orphan keys (`client_dailylog:*`, `client_training_log:*`, `client_545_*`, `client_supplement_*`, `client_nutrition_*`, etc.) but those are inert; the email lookup is the one that actively breaks a flow.
- **Suggested fix:** In the DELETE branch, GET the client first, parse `email`, and if it exists run `DEL client_email:{normalizeEmail(email)}` alongside the existing two cleanup commands. While in there, consider also removing `client_dailylogs_index:{id}`, `client_training_logs_index:{id}`, and the per-day data keys, but that's hygiene rather than a correctness bug.
- **Confidence:** high
