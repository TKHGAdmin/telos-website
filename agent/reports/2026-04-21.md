## Summary
Security pass across auth libraries, API endpoints, and public form handlers. Found five concrete issues spanning brute-force exposure, timing-attack inconsistency, rate-limit bypass, session lifecycle, and account enumeration. ## Findings

#### BUG-20260421-01 - Admin dashboard login has no rate limiting, enabling unbounded brute force of the shared password
- **Severity:** P1
- **Location:** `api/dashboard/login.js:3-17`
- **Category:** auth
- **Reproduce:**
  1. POST any JSON body with `{ "password": "guess" }` to `/api/dashboard/login`.
  2. Observe a 401 on mismatch, a 200 + `Set-Cookie` on match.
  3. Repeat from the same IP as fast as you can. No counter, no lockout, no delay. `api/lib/redis.js` is not imported and no `ratelimit:*` key is written.
- **Impact:** `DASHBOARD_PASSWORD` is a single shared secret that gates all 14 admin endpoints (clients, revenue, analytics, Charleston applications, pipeline, email blasts, video upload/delete). A moderate-speed online attack against a human-memorable password is viable. The three public submit endpoints (`submit-quiz`, `submit-email`, `submit-chs-application`) already use a `ratelimit:*:{ip}` pattern against Upstash Redis, so the infrastructure and idiom are already proven in-tree - only the admin and client logins omit it.
- **Suggested fix:** Add the same `INCR` + `EXPIRE` pattern used in `api/submit-quiz.js:23-29`, keyed on the real client IP (see BUG-20260421-03), with a tighter ceiling (for example 10 attempts per IP per hour). Consider a second counter keyed on a stable fingerprint so one attacker rotating XFF cannot grind it down. On limit breach return 429 before calling `verifyPassword` to also kill any timing side channel.
- **Confidence:** high

#### BUG-20260421-02 - Admin session token signature compared with string `!==`, not constant-time, diverging from the client-auth implementation
- **Severity:** P2
- **Location:** `api/lib/auth.js:18-31`
- **Category:** auth
- **Reproduce:**
  1. Open `api/lib/auth.js:27`: `if (signature !== expected) return false;`.
  2. Compare with `api/lib/client-auth.js:46-49`, which builds Buffers and calls `crypto.timingSafeEqual`.
  3. The admin path leaks a timing signal proportional to the common prefix length between the attacker-supplied signature and the real HMAC (V8 short-circuits string equality on first differing byte).
- **Impact:** In principle a remote timing attack could recover the 64-char hex HMAC digest byte-by-byte and forge an admin session cookie without knowing `SESSION_SECRET`. The practical bar is high (HTTP jitter dominates, the payload that would be signed is just the expiry timestamp, and an attacker would need to pick a future expiry), but the fix is a two-line change and the inconsistency with `client-auth.js` (which already does this correctly) is the clearest signal that this is an oversight rather than an intentional tradeoff.
- **Suggested fix:** Mirror the client-auth implementation: convert `signature` and `expected` to Buffers, length-check, then `crypto.timingSafeEqual`. Also convert `verifyPassword` in the same file to avoid the early length-based return at `auth.js:55` so password length is not leaked either.
- **Confidence:** high

#### BUG-20260421-03 - Rate-limit keys derived from raw `x-forwarded-for` are trivially bypassed by client-supplied header chaining
- **Severity:** P2
- **Location:** `api/submit-quiz.js:23`, `api/submit-email.js:28`, `api/submit-chs-application.js:40`
- **Category:** auth
- **Reproduce:**
  1. POST to `/api/submit-quiz` with header `X-Forwarded-For: 1.1.1.1`. Vercel appends the real client IP, producing `"1.1.1.1, <real-ip>"` in `req.headers['x-forwarded-for']`.
  2. The code concatenates the whole comma-joined string into `ratelimit:quiz:1.1.1.1, <real-ip>`.
  3. Replay with `X-Forwarded-For: 1.1.1.2`, `1.1.1.3`, ... Each request writes a brand-new Redis key with count 1 and sails past the `if (count > 10)` check.
- **Impact:** Three public endpoints become unlimited write sinks. `submit-quiz` and `submit-email` grow unbounded ZSETs (`quiz_submissions`, `email_protein_calculator`, `email_hyrox_predictor`), corrupting lead dashboards and consuming Upstash quota. `submit-chs-application` can flood `chs_applications_index` with garbage, burying real Charleston leads. Also inflates Vercel function invocations during abuse.
- **Suggested fix:** Take the last IP in the chain (which Vercel appends), or - cleaner - use `req.headers['x-real-ip']`, or Vercel's `x-vercel-forwarded-for` which is not client-injectable at the edge. Normalize to a single IP before composing the key. Apply in all three submit handlers plus anywhere the new admin/client login rate limits land.
- **Confidence:** high

#### BUG-20260421-04 - Client password reset does not invalidate existing sessions
- **Severity:** P2
- **Location:** `api/client/reset-password.js:71-96`
- **Category:** auth
- **Reproduce:**
  1. Log in as a client on device A; `telos_client_session` cookie now valid for 7 days (`client-auth.js:6`).
  2. From device B, request a reset via `/api/client/reset-password` with only `email`, follow the link, POST `{ token, password }` to set a new password.
  3. `reset-password.js` writes the new `passwordHash` / `passwordSalt` and deletes the reset token. It does NOT touch the session-invalidation path - there is none. Device A keeps full portal access until the token expiry stamped in its cookie.
- **Impact:** The usual threat that drives a password reset (someone else has the credentials or an active session) is not actually cut off. An attacker who already signed in keeps reading nutrition logs, training logs, 545 entries, and mindset content, and can still POST to every `/api/client/*` endpoint, for up to 7 days after the client resets.
- **Suggested fix:** Add a session-version field on the client record (for example `sessionEpoch`), bake it into the token payload in `createClientToken`, verify it in `verifyClientToken`, and bump it whenever `reset-password.js` or `client-portal.js` rewrites the password. Same treatment would protect the admin side if `DASHBOARD_PASSWORD` is ever rotated.
- **Confidence:** high

#### BUG-20260421-05 - Client login returns distinguishable error messages, enabling email enumeration
- **Severity:** P2
- **Location:** `api/client/login.js:25-47`
- **Category:** auth
- **Reproduce:**
  1. POST `{ "email": "not-a-client@example.com", "password": "x" }` - returns `"Invalid email or password"`.
  2. POST an email that exists but `portalEnabled` is false - returns `"Portal access not enabled"` (line 37).
  3. POST an email for a client whose `status !== 'active'` - returns `"Account is not active"` (line 40).
  4. POST an email for a client with no password hash yet - returns `"Password not set - contact your coach"` (line 43).
  5. Each case is a distinct 401 body, so an attacker can probe a list of emails (purchased leads, breach dumps, guessed coach contacts) and cleanly separate "is a Telos client" from "isn't".
- **Impact:** Discloses customer relationships. Telos's client list is implicitly confidential (tier, payment amount, high-touch coaching). Confirmed membership is also a high-value input for targeted phishing ("Hi, this is your Telos coach, click to reset your portal..."), especially given BUG-20260421-04 lets a phisher who captures credentials hold a session through a subsequent reset.
- **Suggested fix:** Collapse all failure paths to the same generic 401 body (`"Invalid email or password"`). Keep the granular states in server logs for debugging. The reset-password handler already models this correctly at `reset-password.js:24-36` - mirror that shape.
- **Confidence:** high

## Not-bugs considered
- `api/client/daily-log.js` and peers take `clientId` from the session cookie, not query string - no IDOR.
- `esc()` helper at `client-dashboard.html:1831` is a correct HTML-escape (uses `createTextNode`); spot checks of innerHTML sinks that render coach-supplied fields apply it consistently.
- `hashPassword` in `client-auth.js` uses PBKDF2-SHA512 100k iterations with 32-byte salt - adequate.
- Cron handlers `weekly-summary.js` and `engagement-check.js` require `Bearer ${CRON_SECRET}` and fail closed if the env var is missing.
- All 14 admin dashboard endpoints and all 19 authenticated client endpoints call `verifySession`/`verifyClientSession` at the top of the handler.
- Repo grep for common secret prefixes (`sk-ant`, `re_`, `sk_live`, `ghp_`, `xoxb-`, `AIA[A-Z0-9]{15,}`) returned nothing.
- `analytics.js:6` hardcodes a Vercel project ID - this is an identifier, not a secret; ignored.
- `upload-video.js` accepts a filename from the query string and passes it to `put('courses/' + filename, ...)`, but the endpoint is behind admin auth and the blob pathing is controlled by Vercel Blob, so path traversal does not escape the bucket.
- `javascript:` URLs survive `esc()` in `client-dashboard.html:4929` for coach-supplied resource links, but the field is admin-authored - low-value to flag until/unless a client-writable path appears.
