## Summary

5 findings — 2× P1, 2× P2, 1× P3. No P0s. No hardcoded secrets, no IDOR, no exposed admin endpoints, no SSRF, no obvious XSS sinks (all coach-authored unsanitized HTML is intentional and gated to admin input). ## Findings

### [P1] No rate limiting on `/api/dashboard/login`

- **File**: `api/dashboard/login.js:1-17`
- **Description**: The admin login endpoint accepts unlimited password attempts. There is no Redis counter, no IP throttle, no lockout, no captcha. Every other entry point on the site (`submit-quiz`, `submit-email`, `submit-chs-application`) does rate-limit by IP via `ratelimit:*` Redis keys, so this is a clear inconsistency rather than a deliberate exemption.
- **Reproduction**: `for i in $(seq 1 1000); do curl -s -X POST https://www.telosathleticclub.com/api/dashboard/login -H 'Content-Type: application/json' -d '{"password":"guess'$i'"}'; done` — every request gets evaluated, no 429 ever returned. `verifyPassword` uses timing-safe comparison so the password value itself doesn't leak, but the endpoint is happy to keep guessing.
- **Impact**: The entire `/thomas` dashboard sits behind a single `DASHBOARD_PASSWORD` env var. With unlimited attempts, an attacker who guesses correctly gets full CRUD on every client record, every quiz lead, every CHS application, plus the ability to set/reset any client password (`api/dashboard/client-portal.js`). The blast radius is the whole business backend.
- **Suggested fix**: Mirror the pattern in `submit-quiz.js:22-29` — `INCR ratelimit:dashlogin:{ip}` with `EXPIRE 3600` on first miss, reject at e.g. 10 failed attempts/hour. Optionally also throttle per-account (one counter, since there's only one admin password) so a botnet can't trivially scale around the IP throttle.

### [P1] No rate limiting on `/api/client/login` or `/api/client/reset-password`

- **Files**:
  - `api/client/login.js:9-60`
  - `api/client/reset-password.js:8-103`
- **Description**: Neither endpoint has any rate-limit. The reset-password endpoint also makes one outbound call to Resend per request (line 45-65), so it's both a brute-force vector for `login` and an email-bomb / cost-amplification vector for `reset-password`.
- **Reproduction**:
  - `login`: POST `{"email":"<known client email>","password":"..."}` in a loop; each request runs a full PBKDF2 (100k iterations) and returns `Invalid email or password` on miss. PBKDF2 slows the attacker but doesn't stop them and burns server CPU during the attack.
  - `reset-password`: POST `{"email":"<known client email>"}` in a loop; each request that hits a real, portal-enabled client triggers one Resend email. Constant-response (`{ ok: true }`) intentionally prevents enumeration, but it does not prevent the email send. Targets that don't exist get the same constant response with no email — so the attacker can't enumerate but can definitely flood any specific known address.
- **Impact**:
  - Brute force of client portal accounts. Client passwords are set by the coach via a `prompt()` (`thomas.html:2029,2049`) with only an 8-character minimum — common in practice to be weak. Successful login exposes that client's logs, nutrition, training, daily check-ins, and 545 content.
  - Mailbox flooding of any client whose email is known (and these get harvested from CHS applications, quiz submissions, etc., so they aren't hard to obtain). Will eventually get the `noreply@telosathleticclub.com` sender flagged by Resend / receiving providers, damaging deliverability for legitimate reset emails too.
- **Suggested fix**: Add IP rate limit on both, plus per-email rate limit on `reset-password` (`INCR ratelimit:reset:{normalizedEmail}` with `EXPIRE 3600`, cap at e.g. 3/hr per email). Apply the per-email cap *before* the Resend call but *after* the constant-response branch, so enumeration resistance is preserved.

### [P2] Non-constant-time HMAC comparison in admin session verification

- **File**: `api/lib/auth.js:18-31` (specifically `if (signature !== expected) return false;` on line 27)
- **Description**: Admin `verifyToken` compares the cookie's HMAC signature against the computed one with `!==` (JS string compare, short-circuits on first mismatch). The companion `api/lib/client-auth.js:46-49` does this correctly with `crypto.timingSafeEqual` — admin code is the outlier.
- **Reproduction**: Code-only finding; practical exploit is hard (network jitter dominates timing of a single HMAC compare and the secret never appears in user-controllable input — only the *signature* would leak, not `SESSION_SECRET`). Including it because (a) the fix is one line, (b) it's a code-consistency bug that the existing convention in `client-auth.js` already shows the right answer for.
- **Impact**: Theoretical. Practical impact is bounded — an attacker who could exfiltrate the signature for a specific (expiry) payload could mint a session for that expiry. They'd still need the expiry timestamp to predate cookie expiration, and they'd be racing against new tokens being minted, but it's the kind of thing a code reviewer flags.
- **Suggested fix**: Replace line 27 with the same `Buffer.from` + `timingSafeEqual` pattern used in `client-auth.js:46-49`. Add a length check before `timingSafeEqual` to avoid throwing.

### [P2] `x-forwarded-for` taken as-is for rate-limit keys

- **Files**:
  - `api/submit-quiz.js:23` (`req.headers['x-forwarded-for'] || req.headers['x-real-ip'] || 'unknown'`)
  - `api/submit-email.js:28`
  - `api/submit-chs-application.js:40`
- **Description**: Vercel appends the real client IP to whatever `x-forwarded-for` value the client supplies. The endpoints use the full header value as the rate-limit key, so an attacker who varies the `X-Forwarded-For` request header on each call gets a different key each time and bypasses the cap entirely.
- **Reproduction**:
  ```
  for i in $(seq 1 100); do
    curl -s -X POST https://www.telosathleticclub.com/api/submit-email \
      -H "Content-Type: application/json" \
      -H "X-Forwarded-For: spoof-$i" \
      -d '{"email":"a@b.co","source":"protein-calculator"}'
  done
  ```
  All 100 succeed; the documented cap of 10/hr/IP never engages because the Redis key includes the user-supplied prefix.
- **Impact**: Rate limiting on public submission endpoints is currently bypassable from a single IP. Allows a determined adversary to flood the `quiz_submissions` ZSET, `email_*` ZSETs, and `chs_applications_index` with junk and pollute the dashboard's leads view. Lower severity than the login findings because no auth boundary is at stake, but it makes the existing controls effectively decorative.
- **Suggested fix**: Use only the first comma-separated IP of `x-forwarded-for` (the real client IP that Vercel appends is the *last* on Vercel; double-check current Vercel docs before flipping). Or, more robustly, prefer `req.headers['x-real-ip']` when present (Vercel sets it to the real IP and does not concatenate caller input), falling back to a parsed-and-validated `x-forwarded-for[0]`.

### [P3] Path-traversal in admin video upload key

- **File**: `api/dashboard/upload-video.js:17-33`
- **Description**: `filename = req.query.filename` is concatenated into `put('courses/' + filename, ...)` with no path-segment validation. A `filename` value like `../foo.mp4` would write the blob outside the `courses/` prefix.
- **Reproduction**: `POST /api/dashboard/upload-video?filename=../../escape.mp4` (with valid admin session) writes to a blob path outside the intended namespace. Whether `@vercel/blob` itself rejects `..` is implementation-defined and not documented as a guarantee.
- **Impact**: Admin-only, so an external attacker can't reach this without first breaking admin auth (see P1 above — making this less hypothetical than it would otherwise be). Even with admin access, the blob token is read-write to a single namespace, so the practical worst case is collision/overwrite with another path. Reporting it for hygiene and because the fix is trivial.
- **Suggested fix**: Reject filenames containing `/`, `\`, or `..`, or extract just the basename: `var safe = filename.split(/[\/\\]/).pop().replace(/^\.+/, '');`.

## Negative results (things checked, found clean)

- No hardcoded API keys, tokens, or passwords in source or git history (`git log --all -p -S` for common secret prefixes returned nothing).
- No IDOR in `api/client/*` — every audited endpoint derives `clientId` from `verifyClientSession(req)`; the two endpoints that take a body `clientId` (`notify.js`, `send-email.js`) are admin-auth-gated.
- Cron endpoints (`weekly-summary`, `engagement-check`) check `CRON_SECRET` with constant-prefix `Bearer ` match and fail-closed when the secret is unset.
- DOM-injection sinks in `client-dashboard.html` route user-derived data through `esc()`. Two unescaped paths exist (`drawerCustomContent.innerHTML = item.content`, `href="' + esc(r.url) + '"` which would still allow `javascript:`) but both are coach-authored input, where the coach already has full admin authority.
- `vercel.json` CORS allow-list is scoped to `/api/submit-*` only; no global `*` ACAO.
