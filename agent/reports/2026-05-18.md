# Telos Bug Hunter — 2026-05-18

- **Focus**: Functional
- **Findings**: 4
- **Tools used**: Read, Bash (grep), git log

## Summary

First run. Walked the API surface — auth libs, public submission endpoints, cron jobs, and the client portal lifecycle (`login`, `reset-password`, `daily-log`, `training-log`, `clients`, `client-portal`). The cron jobs and the client-deletion path turned up four reproducible defects with concrete user impact: the weekly-summary email reports a streak of 0 for most clients regardless of how consistent they are; the engagement-check cron has no per-client dedup and will send an inactive client a reminder every single day until they log in; deleting a client orphans the `client_email:` lookup so the same address can't be reused; and the client login endpoint leaks account existence via distinct error messages, while the parallel reset-password endpoint correctly returns a uniform response. No P0/P1 bugs found.

## Findings

### 20260518-1 — Weekly summary email shows streak=0 for active clients

- **Severity**: P2
- **Area**: cron / email
- **Location**: `api/cron/weekly-summary.js:87-90`

**What's wrong**

The streak counter starts at `d=0`, which is "today" — the day the cron actually runs. The cron is scheduled for Monday 14:00 UTC (9am ET), and almost no client will have completed Monday's daily check-in by that hour, so `weekData[0][0].result` is null and the `else break` immediately exits the loop with `streak = 0`. Even a client who logged seven consecutive days Sunday-back-to-the-previous-Monday gets a "Current Streak: 0 days" email.

**How to reproduce**

```js
// api/cron/weekly-summary.js
for (var d = 0; d < 7; d++) {
  if (weekData[0][d] && weekData[0][d].result) streak++;
  else break;
}
```

`weekData[0][0]` is today's daily log. The cron runs before clients have a chance to fill it in. The loop breaks on the first miss, so any client who hasn't already logged Monday morning gets streak 0 regardless of prior consistency.

**Why it matters**

The weekly recap is the headline retention email. Showing "Streak: 0" to a client who logged six days in a row is demotivating and undercuts the email's own message ("Every day counts.").

**Suggested fix**

Start the streak walk at yesterday and continue back through the week — e.g. `for (var d = 1; d < 7; d++)`. Or, gate it: if today has a log, start at 0; otherwise start at 1.

### 20260518-2 — Engagement-check cron has no dedup, spams inactive clients daily

- **Severity**: P2
- **Area**: cron / email
- **Location**: `api/cron/engagement-check.js:36-94` (logic) and `vercel.json:23-26` (daily schedule)

**What's wrong**

The cron runs daily (`0 15 * * *`) and emails every active, portal-enabled client whose most-recent daily log is 3 or more days old. There is no per-client tracking of "last reminder sent." A client who goes inactive for ten days will receive a "We miss you at Telos" email on days 3, 4, 5, 6, 7, 8, 9, 10 — eight identical reminders.

**How to reproduce**

1. Have a client `C` with `status: 'active'`, `portalEnabled: true`, `emailEnabled !== false`, a valid `email`.
2. Ensure no `client_dailylog:{C.id}:{date}` keys exist for the last 5 days.
3. Trigger the cron with the right `Authorization: Bearer ${CRON_SECRET}` header.
4. Observe one email sent.
5. Wait 24 hours. Trigger again. Observe a second identical email — no record of the first send is consulted.

**Why it matters**

Repeated identical emails to disengaged clients drive unsubscribes and spam complaints, and damage the sending domain's reputation. The intent of the feature (gentle nudge) is undone by the cadence.

**Suggested fix**

Track `client_engagement_last_sent:{clientId}` with a TTL (e.g. 4 days) in Redis. Before sending, `GET` that key — skip if present. After sending, `SETEX` it with a 4-day TTL. Same pattern as the existing rate-limit counters.

### 20260518-3 — Deleting a client leaves an orphaned `client_email:` lookup

- **Severity**: P2
- **Area**: client lifecycle / Redis hygiene
- **Location**: `api/dashboard/clients.js:143-153`

**What's wrong**

The DELETE handler removes `client:{id}` and the `clients_index` ZSET entry, but it does not remove `client_email:{normalizedEmail}`. That mapping is created in `api/dashboard/client-portal.js:117` whenever portal access is enabled. After deletion the mapping still points at the now-missing client record. The duplicate-email check on creation (`clients.js:70-73`) then fires on the orphaned key and blocks creating any new client with the same email.

**How to reproduce**

1. Create client A with email `a@example.com` and enable portal via `client-portal.js` POST (this sets `client_email:a@example.com` → `<id_A>`).
2. Delete client A via `DELETE /api/dashboard/clients?id=<id_A>`.
3. `GET client:<id_A>` returns null — the record is gone.
4. `GET client_email:a@example.com` still returns `<id_A>`.
5. Try to create a new client with email `a@example.com` via `POST /api/dashboard/clients`. The handler hits the `existingEmailOwner` check at line 70 and returns 400 "Email already in use by another client." Same email can never be reused without manual Redis cleanup.

A related secondary issue: the DELETE also leaves every per-client data key behind (`client_dailylog:*`, `client_training_log:*`, `client_nutrition_*`, `client_545_*`, `client_supplement_*`, `client_activity_*`, `client_sidemenu:*`, plan keys, and any open `password_reset:*` token referencing this clientId). Worth raising for a follow-up — primary issue is the email lookup because it has a user-visible failure mode.

**Why it matters**

A coach who deletes a client (e.g. duplicate entry, off-boarded, then re-onboarding) cannot recreate the same email address from the admin UI. The error message blames the user ("Email already in use") with no indication the conflict is with a deleted record. Requires direct Redis access to recover.

**Suggested fix**

In the DELETE branch of `clients.js`, fetch the client object before deleting and `DEL client_email:{normalizeEmail(client.email)}` when `client.portalEnabled`. Pattern already used in `client-portal.js:121-124` and `clients.js:130` — same call, just missing here.

### 20260518-4 — Client login leaks account existence via distinct error messages

- **Severity**: P2
- **Area**: client portal auth
- **Location**: `api/client/login.js:30-47`

**What's wrong**

The login handler returns four different error strings depending on what failed:

```js
if (!clientId)                                    → "Invalid email or password"  // line 26
if (!raw)                                         → "Invalid email or password"  // line 31
if (!client.portalEnabled)                        → "Portal access not enabled"  // line 37
if (client.status !== 'active')                   → "Account is not active"     // line 40
if (!client.passwordHash || !client.passwordSalt) → "Password not set - contact your coach"  // line 43
if (!verifyPassword(...))                         → "Invalid email or password"  // line 47
```

An attacker who can submit any email to `/api/client/login` learns whether that email is a Telos client (and what state the account is in) by reading the error string. The parallel `reset-password.js` endpoint already gets this right — it returns `{ ok: true }` for every failure mode (lines 25-36) precisely to prevent enumeration. The login flow contradicts that decision.

**How to reproduce**

1. `POST /api/client/login` with `{ email: "not-a-client@example.com", password: "x" }` → 401 `"Invalid email or password"`.
2. `POST /api/client/login` with `{ email: "<real client with portal disabled>", password: "x" }` → 401 `"Portal access not enabled"`.
3. Different response body confirms the email is a known client. Repeat with a list to enumerate all clients.

**Why it matters**

Confirms which emails are paying Telos clients (sensitive — coaching/fitness is a personal category) and which accounts are mid-onboarding (no password set yet) versus fully active. Useful intel for phishing those specific people with a "your coach asked me to follow up" pretext.

**Suggested fix**

Collapse the three pre-password-check failure modes (`portalEnabled`, `status !== 'active'`, missing hash/salt) into the same 401 `"Invalid email or password"` returned for unknown emails and bad passwords. If Thomas wants to surface the friendly "contact your coach" message for the password-not-set case, gate it behind something the legitimate user already knows (e.g. surface only after a separate "send setup link" flow), not on the unauthenticated login response.
