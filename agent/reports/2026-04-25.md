## 2026-04-25-01 — Tool pages do not load `js/main.js`, and inline-duplicate the hamburger handler

**Severity:** P1
**Location:** `protein-calculator.html`, `hyrox-predictor.html`

### Repro

```
$ grep -c 'main.js' protein-calculator.html hyrox-predictor.html
hyrox-predictor.html:0
protein-calculator.html:0
```

Compare to `index.html:843`, `pricing.html:895`, `resources.html:611` which all load `js/main.js?v=2`.

Both tool pages also contain an inline copy of the hamburger handler:
- `protein-calculator.html:783-789`
- `hyrox-predictor.html:1085-1095`

The inline copies replicate `body.classList.toggle('nav-open')` etc. — exactly the same DOM mutation `js/main.js:30-34` performs.

### Why it matters

`CLAUDE.md` ("Important Conventions") explicitly states that every public marketing page must load `js/main.js` and that tool pages must NOT duplicate hamburger/nav handlers. Both rules are violated on the two calculator pages. Concrete consequences:

- Scroll-driven nav state, smooth-scroll for `#anchor` links, FAQ accordion, IntersectionObserver-driven `.animate-on-scroll` reveals, cursor glow, hero spotlight, tilt, and counter animations are all dead on those pages.
- The naive fix ("just add `<script src="js/main.js?v=2"></script>"`) will silently break the hamburger: each click would toggle `nav-open` twice (inline handler + main.js handler), making the menu open-then-close on a single tap.

### Suggested fix

Two-step on each file:

1. Add `<script src="js/main.js?v=2"></script>` immediately before `</body>`.
2. Delete the inline `hamburger?.addEventListener(...)` block (`protein-calculator.html:782-789`, `hyrox-predictor.html:1083-1095`).

Verify after: hamburger toggles once per click, hash links smooth-scroll, scroll header behavior matches index.html.

## 2026-04-25-02 — `todayStr()` returns UTC date, miscounts logs and streaks for non-UTC users

**Severity:** P1
**Location:** `client-dashboard.html:1840-1842` (definition) plus ~25 call sites

### Repro

```js
// client-dashboard.html:1840
function todayStr() {
  return new Date().toISOString().split('T')[0];
}
```

`toISOString()` formats in UTC. For a Charleston (ET) client checking in at 8:30 PM EDT, `new Date()` is `2026-04-25 20:30 EDT` = `2026-04-26 00:30 UTC`, so `todayStr()` returns `"2026-04-26"`. The check-in is written under tomorrow's date.

Same UTC pattern recurs at:
- `client-dashboard.html:1794, 1795, 1800, 2149, 2177, 2278-2294, 2513, 2589-2595, 2784, 2939, 3029-3050, 3156, 3502, 3556, 3583, 3682, 3743-3749, 3807, 3852, 3890, 4033`
- The `for (var i = 0; i < N; i++) { var dateStr = new Date(now - i * 86400000).toISOString().split('T')[0]; }` loops at `3556, 3583, 3807, 3890` use the same UTC slice, so the 7-day trend and 30-day heatmap are off-by-one for the same users.

`CLAUDE.md` "Known Limitations" lists this as still-unresolved from the April 2026 bug crawl.

### Why it matters

- Daily check-ins, nutrition logs, training logs, supplement logs, 545 daily entries — all keyed by `todayStr()` — get written to the wrong date when the user is east of UTC near midnight (or west of UTC near local evening).
- Execution Score on Home tab reads "today's" log via the same string and may show "not done" when the user just submitted, or "done" because yesterday's evening submission landed on what is now displayed as "today".
- Streak math compares consecutive dates from the index ZSET; a UTC-shifted entry creates phantom gaps.
- Coach view in `/thomas` shows the date column from the same key, so the admin sees a mismatched date next to the activity.

### Suggested fix

Replace the function and every `.toISOString().split('T')[0]` used as a "today" / "N days ago" string with a local-date helper. Server endpoints already accept `?date=YYYY-MM-DD` from the client, so making the client local-date authoritative is sufficient — no API changes needed.

```js
function ymd(d) {
  d = d || new Date();
  var y = d.getFullYear();
  var m = String(d.getMonth() + 1).padStart(2, '0');
  var dd = String(d.getDate()).padStart(2, '0');
  return y + '-' + m + '-' + dd;
}
function todayStr() { return ymd(); }
function daysAgoStr(n) { var d = new Date(); d.setDate(d.getDate() - n); return ymd(d); }
```

Then sweep every `new Date(now - i * 86400000).toISOString().split('T')[0]` to `daysAgoStr(i)`. Backfill the test plan: log in from a device with TZ=America/New_York at 9 PM, submit a check-in, confirm `client_dailylog:{id}:2026-04-25` exists (not `:2026-04-26`).

## 2026-04-25-03 — `engagement-check` cron sends a reminder email every day a client stays inactive

**Severity:** P1
**Location:** `api/cron/engagement-check.js:61-94`

### Repro

The cron runs daily (per `vercel.json`). The send condition is:

```js
// engagement-check.js:61
if (lastLogDaysAgo === -1 || lastLogDaysAgo >= 3) {
  // ... send email via Resend, no record kept of having sent
}
```

There is no Redis key, no field on the client record, and no in-flight check that tracks "we already nudged this client". `grep -nE 'last.*reminder|reminder.*sent|engagement.*timestamp' api/` returns no matches.

So the timeline for a client who logs in Day 0, then goes silent:
- Day 3: 3 days inactive → email sent.
- Day 4: 4 days inactive → email sent.
- Day 5: 5 days inactive → email sent.
- Day 6+: lookup window only spans 5 days, so `lastLogDaysAgo === -1` → email sent.

A client on a vacation week receives 5+ identical "We miss you at Telos" emails.

### Why it matters

- Resend sender reputation: repeat-spam triggers Gmail/Outlook spam classification.
- Real risk of clients marking the address as spam and losing transactional reliability for password-reset and weekly-summary emails sent from the same sender.
- Inverse of the intended user experience — a re-engagement nudge becomes a nag.

### Suggested fix

After a successful Resend POST, write:

```js
await redis('SET', 'client_engagement_reminder:' + client.id, '1', 'EX', 5 * 86400);
```

And before deciding to send, skip if that key exists (`GET` returns non-null). Alternatively, add `lastEngagementReminderAt` to the client record. Either gives a 5-day cooldown that pairs cleanly with the 3-day inactivity threshold.

## 2026-04-25-04 — 28 pages still link with `.html` extensions, defeating Vercel `cleanUrls`

**Severity:** P2
**Location:** All 23 files in `blog/`, plus `pricing.html`, `protein-calculator.html`, `hyrox-predictor.html`, `chs.html`, `resources.html`

### Repro

```
$ grep -rlE 'href="[^"]*\.html' blog/ pricing.html protein-calculator.html hyrox-predictor.html chs.html resources.html | wc -l
28
```

Sample lines: `pricing.html:509,511,512,522,534,871,874`, `chs.html:792,798,812,1043`, `blog/stress-cortisol-gains.html:271,274,275,285,298,299,309,512`, etc. (full list available via the grep above; sample shown in the hunt log).

`vercel.json` sets `cleanUrls: true`, which means every one of these links triggers a 308 redirect from `/index.html` to `/`.

`CLAUDE.md` "Important Conventions" lists this as a known unresolved P1-P3 backlog item ("`.html` extensions in internal links"). `index.html` itself is already clean — verified `grep -nE 'href="[^"]*\.html' index.html` returns nothing.

### Why it matters

- Each cross-page nav click pays an extra round trip.
- Hash fragments after a redirect can land before `main.js` finishes wiring smooth-scroll, causing the "click 'How It Works' from blog page" flow to land at `#how-it-works` without smooth animation, and occasionally not scroll at all.
- Search engines see two URLs for the same resource in internal anchors, weakening canonical signal.

### Suggested fix

Mechanical sed replacement:
- Root pages: `href="index.html` → `href="/`, `href="index.html#` → `href="/#`.
- Blog pages: `href="../index.html` → `href="/`, `href="../index.html#` → `href="/#`.

Verify by re-running the same grep and confirming zero matches outside `<link rel="canonical">` (which already uses absolute URLs).

## 2026-04-25-05 — `chs.html` loads `js/main.js` without the cache-busting query the rest of the site uses

**Severity:** P3
**Location:** `chs.html:1050`

### Repro

```
$ grep -nE 'main.js' chs.html index.html pricing.html resources.html
chs.html:1050:  <script src="js/main.js"></script>
index.html:843:  <script src="js/main.js?v=2"></script>
pricing.html:895:  <script src="js/main.js?v=2"></script>
resources.html:611:  <script src="js/main.js?v=2"></script>
```

### Why it matters

Next time `js/main.js` is updated, returning visitors who land on `/chs` first may serve a cached older copy until they hard-reload, while every other page picks up the new bundle. CSS uses `?v=16` consistently across all pages — the JS pin should match.

### Suggested fix

Edit `chs.html:1050` from `<script src="js/main.js"></script>` to `<script src="js/main.js?v=2"></script>`.
