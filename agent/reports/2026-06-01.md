## Bug 1 - Quiz retake attaches duplicate submit listeners, causing duplicate `/api/submit-quiz` POSTs

- **ID**: 2026-06-01-01
- **Severity**: P1
- **Area**: Lead capture, dashboard data quality
- **Files**: `js/quiz.js:221-276` (`setupLeadCapture`), `js/quiz.js:303-413` (`renderResults`)
- **Affects**: `index.html` and `pricing.html` (both load `js/quiz.js`)

### What happens

`setupLeadCapture()` is called every time the user reaches the lead-capture step. It does `submitBtn.addEventListener('click', ...)` against `.quiz-see-score-btn`, which is the same DOM node every time. There is no `removeEventListener` and no idempotency guard.

After one retake, the submit button has 2 click listeners. After N retakes, N+1. Each click then fires every attached handler, so the user's final submission produces:

- N+1 POSTs to `/api/submit-quiz` (each with `submittedAt` differing by ms, so Redis stores them as distinct ZSET members)
- N+1 entries in the `quiz_submissions` ZSET, all attributed to the same lead
- N+1 calls to `renderResults()`, which itself starts N+1 `setInterval` count-up timers updating the same `.quiz-score-number` element, animates the pillar bars N+1 times, and attaches yet another `.quiz-retake-btn` listener (compounding the issue for the next retake)

### Why it matters

- **Dashboard data pollution**: Thomas sees duplicate leads in `/thomas` -> Leads tab for any user who took the quiz, hit Retake, and submitted again. Lead counts and conversion rates are inflated.
- **Visible glitch on retake**: Pillar-bar animations and the score count-up visibly race / replay on the second submission, which looks broken to the user.
- **Rate-limit waste**: The IP rate limit (10/hr) is partially consumed by the duplicates.

### Repro

1. Open `https://www.telosathleticclub.com/` (or `/pricing`).
2. Scroll to the Execution Score Quiz, click Start, complete all 8 questions, enter name/email/phone, click "See Your Score".
3. On the results screen, click "Retake Assessment".
4. Complete the quiz again. On the lead-capture form, open DevTools Network tab, then click "See Your Score".
5. Observe **two** POST requests to `/api/submit-quiz`. After a second retake, three. Etc.

### Suggested fix

Make `setupLeadCapture` idempotent. Easiest option: replace the submit button before attaching, e.g.

```js
const oldBtn = quizLeadCapture.querySelector('.quiz-see-score-btn');
const submitBtn = oldBtn.cloneNode(true);
oldBtn.parentNode.replaceChild(submitBtn, oldBtn);
submitBtn.addEventListener('click', function (e) { /* ... */ });
```

Or track a `quiz.leadCaptureBound = true` flag and skip rebinding. Same treatment for the `retakeBtn` listener in `renderResults` (line 397).

### Confidence

High. The control flow is straightforward and visible in the source. Easy to verify in DevTools.

## Bug 2 - `/api/submit-quiz` does not validate email format; accepts any non-empty string

- **ID**: 2026-06-01-02
- **Severity**: P2
- **Area**: Lead data quality
- **Files**: `api/submit-quiz.js:18-20`

### What happens

The handler only checks truthiness:

```js
if (!email || !name) {
  return res.status(400).json({ error: 'Name and email required' });
}
```

Compare to the sibling endpoints:

- `api/submit-email.js:18` checks `email.includes('@')`
- `api/submit-chs-application.js:35` enforces `/^[^\s@]+@[^\s@]+\.[^\s@]+$/`

So `"asdf"`, `" "` (a space), `"x"`, or any other junk string passes `submit-quiz` and is persisted to the `quiz_submissions` ZSET as a real lead.

### Why it matters

- The client-side form input is `type="email"`, which gives some browser-level validation, but anyone POSTing directly (curl, malicious bot) gets through.
- Thomas's Leads tab fills with un-contactable rows that look real until clicked.
- Inconsistent with the same pattern enforced on sibling endpoints, which suggests it's an oversight, not a deliberate looser policy.

### Repro

```bash
curl -X POST https://www.telosathleticclub.com/api/submit-quiz \
  -H 'Content-Type: application/json' \
  -d '{"name":"Test","email":"not-an-email","scores":[1,1,1,1],"totalScore":4}'
```

Returns `{"ok": true}`. Lead appears in `/thomas` Leads tab.

### Suggested fix

Reuse the same regex from `submit-chs-application.js`:

```js
if (!email || !/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email)) {
  return res.status(400).json({ error: 'Valid email required' });
}
```

Optional: add a similar regex helper in `api/lib/` since three handlers now want it.

### Confidence

High. Direct file read; behavior trivially reproducible against prod.

## Items considered and not reported

For trust calibration, here are things I looked at and decided *not* to flag:

- **`api/lib/auth.js verifyPassword` empty-buffer edge case**: if `DASHBOARD_PASSWORD` env var were unset, `verifyPassword('')` could short-circuit. But the upstream `!password` check in `api/dashboard/login.js:10` blocks empty input before it reaches `verifyPassword`. Not exploitable.
- **`clients.js` POST does not write `client_email:` lookup**: traced full lifecycle; the lookup is correctly created when portal is enabled (`client-portal.js:117`) and correctly updated/deleted on email changes (`clients.js:121-132`). No orphan possible from the paths I followed.
- **`js/quiz.js` Calendly CTA href `1-1-growth-consultation-call-clone`**: per `CLAUDE.md` this slug is the Lifestyle-tier consultation URL, not the generic / Growth one. Could be misrouting quiz takers, or the slug naming in CLAUDE.md is misleading. Not confident enough to call it a bug without Thomas confirming intent.
- **No rate limit on `api/dashboard/login.js` and `api/client/login.js`**: real concern, but it's a security finding and today's focus is functional. Holding for the security rotation.
- **`redisPipeline` doesn't check `data.error`**: would throw at the caller's `.forEach`, hits the existing `try/catch`, returns 500. Defensive-coding nit, not user-facing.
