## BUG-2026-04-23-01 — Weekly summary email always reports streak of 0 on Mondays

- **Severity:** P1
- **Area:** `api/cron/weekly-summary.js`
- **Focus:** functional

### Where
`api/cron/weekly-summary.js:86-90`

```javascript
// Count streak (from today backwards)
for (var d = 0; d < 7; d++) {
  if (weekData[0][d] && weekData[0][d].result) streak++;
  else break;
}
```

`weekData[0][0]` is the daily-log lookup for *today's* date (line 46-47 builds the pipeline starting at `d = 0`, i.e. `new Date(now - 0 * 86400000).toISOString().split('T')[0]`).

### What happens
The cron schedule in `vercel.json:21` runs this handler at `0 14 * * 1` — 14:00 UTC on Mondays, which is 09:00 ET (EST) or 10:00 ET (EDT). At that hour virtually no client has already submitted a daily check-in for Monday.

Because `d = 0` is Monday and Monday's log is missing, the streak loop hits `else break;` on the very first iteration and `streak` stays at `0`. The rendered email then says:

```
Current Streak: 0 days
```

even for clients who logged every day from the prior Sunday all the way back through the week. The "Avg Execution Score", "Workouts Completed" and "Days Logged" numbers are unaffected — only the streak is wrong.

### Impact
Every active + portal-enabled client that receives the Monday weekly recap sees a streak of 0. This is the single motivational stat in the email and it silently undercuts the product message for the most engaged users. It has likely been miscounted every week since the cron was wired up.

### Suggested fix
Skip today in the streak count when today has no log yet, rather than breaking. Something like:

```javascript
var startIdx = (weekData[0][0] && weekData[0][0].result) ? 0 : 1;
for (var d = startIdx; d < 7; d++) {
  if (weekData[0][d] && weekData[0][d].result) streak++;
  else break;
}
```

(Or compute streak from `client_dailylogs_index:{id}` directly — it's already a sorted ZSET of dates.)

## BUG-2026-04-23-02 — Deleting a client orphans the email→id lookup and blocks re-adding them

- **Severity:** P2
- **Area:** `api/dashboard/clients.js`, `api/dashboard/client-portal.js`
- **Focus:** functional

### Where
`api/dashboard/clients.js:143-154` (DELETE handler):

```javascript
if (req.method === 'DELETE') {
  try {
    var id = req.query.id || (req.body && req.body.id);
    if (!id) return res.status(400).json({ error: 'ID required' });
    await redis('DEL', 'client:' + id);
    await redis('ZREM', 'clients_index', String(id));
    return res.status(200).json({ ok: true });
  }
  ...
```

### What happens
`client-portal.js` enables the portal by writing `client_email:{normalizedEmail}` → `clientId` (line 115-118). That lookup is what `api/client/login.js:24` uses to resolve email → client at sign-in.

The clients DELETE removes `client:{id}` and the `clients_index` ZSET entry but **never removes `client_email:{normalizedEmail}`**. The stale mapping then poisons two downstream flows:

1. **Re-creating a client with the same email fails.** `clients.js:70-73` checks `GET client_email:{normalized}` before creating, and bails with `"Email already in use by another client"` — even though the previous owner no longer exists.
2. **If a new client is eventually created with a *different* id but somehow the stale key remains, login becomes ambiguous.** `client/login.js` follows the stale lookup to a deleted `client:{oldId}` and returns `"Invalid email or password"`.

### How to reproduce
1. In `/thomas` → Clients, create a client with email `foo@bar.com`.
2. Open Client Portal, set password — this writes `client_email:foo@bar.com`.
3. Delete the client from the Clients tab.
4. Try to re-create a client with `foo@bar.com` → API responds 400 "Email already in use by another client".

### Impact
A legitimate admin workflow (remove-and-readd, typo cleanup, test client housekeeping) is silently blocked. There is no recovery path from the UI — it requires a manual Redis DEL.

Related orphaning on the same DELETE (worth fixing in the same pass, lower priority): `client_dailylogs_index:{id}`, `client_training_logs_index:{id}`, `client_nutrition_logs_index:{id}`, `client_activity_logs_index:{id}`, `client_supplement_logs_index:{id}`, `client_545_index:{id}`, `client_push_sub:{id}`, plus all the per-date entries under each. These aren't user-facing but accumulate as cruft.

### Suggested fix
Before the two DELs in the DELETE handler, GET the client, normalize its email, and DEL `client_email:{normalized}` too. Optionally DEL the `*_index:{id}` keys (the per-date child keys are harmless once the indexes are gone).

## BUG-2026-04-23-03 — Quiz lead-capture silently drops leads when fields are left empty

- **Severity:** P2
- **Area:** `js/quiz.js`, `index.html` lead-capture form
- **Focus:** functional

### Where
- `index.html:574-579` — the form's submit button:
  ```html
  <button type="button" class="btn btn-primary quiz-see-score-btn" ...>See Your Score</button>
  ```
- `js/quiz.js:225-275` — click handler: no validation before fetching and rendering results.

### What happens
Because the button is `type="button"` and not `type="submit"`, clicking it never triggers HTML5 form submission. The `required` attributes on the `name` and `email` inputs therefore never fire — the browser does nothing to block an empty-field click.

The JS click handler reads the (possibly empty) values, immediately POSTs to `/api/submit-quiz`, and synchronously calls `renderResults()` + `localStorage.setItem('telosQuizCompleted', 'true')`. The API rejects the empty body with `400 "Name and email required"`, but:

- The `.catch(function() {})` on line 269 swallows the failure silently.
- The result screen has already rendered.
- `telosQuizCompleted=true` is persisted, so the pricing page will no longer gate this user behind the quiz on return.

### How to reproduce
1. Open homepage, scroll to the Execution Score Quiz.
2. Click through all 8 questions (answering any option).
3. On the lead-capture card, leave Name and Email blank and click **See Your Score**.
4. Result ring animates and tier copy appears. No error is shown.
5. Check the admin → Leads → Quiz Submissions — no row was written.
6. Visit `/pricing` — the overlay is gone (quiz marked completed locally).

### Impact
The quiz is a primary lead-generation surface and its only capture point is this form. Users who skip the fields still get the payoff (their score) while the business loses the contact. Because the request is fired and silently fails, there's no log-surface signal either — the loss is invisible.

### Suggested fix
Either (a) change the button to `type="submit"` and attach the handler to the form's `submit` event so `required` validation runs, or (b) before the `fetch(...)` in `setupLeadCapture`, validate `quiz.userName` and `quiz.userEmail` (plus a basic email-shape regex) and early-return with an inline error message.  Also: only set `telosQuizCompleted=true` and call `renderResults()` after the submit succeeds, to keep the gate honest.
