## Bug 1: Homepage Instagram nav icon points to wrong account

- **ID:** `2026-04-30-01`
- **Severity:** P1
- **Area:** `index.html` nav (desktop)
- **Status:** new

### What's wrong
The Instagram icon in the homepage top-nav links to `instagram.com/telosathleticclub`. Every footer on the site - homepage, `pricing.html`, `resources.html`, and all 14+ blog posts - links to `instagram.com/telos_fitness`, and `CLAUDE.md` lists `@telos_fitness` as the company Instagram. Commit `f845b46` ("Remove @tomgetshybrid Instagram links from public site", 2026-04-14) deliberately kept `@telosathleticclub` in the nav while removing `@tomgetshybrid`, but did not update the footers or `CLAUDE.md` to match. Net effect: a visitor who clicks the homepage IG icon and a visitor who clicks any footer IG link end up at two different accounts. Either one of the two accounts is stale/wrong (in which case half the site is sending users to the wrong place), or both accounts are real and the docs/footer never got updated to reflect the canonical handle. Either way it's user-visible inconsistency on the most prominent social entry point.

### Where
- File: `index.html:34`

### Repro / evidence
```html
<!-- index.html:34 (homepage nav) -->
<a href="https://www.instagram.com/telosathleticclub" target="_blank" class="nav-ig-story" title="@telosathleticclub">
```

Compare to footer of the same file:
```html
<!-- index.html:814 (homepage footer) -->
<a href="https://instagram.com/telos_fitness" target="_blank" ...>
```
And every blog post + `pricing.html` + `resources.html` consistently uses `instagram.com/telos_fitness`. `CLAUDE.md` "External Services" section also lists `Instagram: @telos_fitness`.

### Suggested fix
Pick the canonical handle. Two reasonable options:
1. **`@telosathleticclub` is canonical** (matches the production domain `telosathleticclub.com`): update every footer (`index.html:814`, `pricing.html:880`, `resources.html:596`, all `blog/*.html`) and the `CLAUDE.md` "External Services" line to `instagram.com/telosathleticclub`.
2. **`@telos_fitness` is canonical**: revert the homepage nav `href` and `title` on `index.html:34` to `instagram.com/telos_fitness` / `@telos_fitness`.

Whichever you choose, end with one consistent IG handle across nav, footer, and `CLAUDE.md`.

## Bug 2: Deleting a client orphans the `client_email` lookup, blocks re-adding the same email

- **ID:** `2026-04-30-02`
- **Severity:** P2
- **Area:** `api/dashboard/clients.js` (DELETE handler)
- **Status:** new

### What's wrong
When the admin deletes a client via the `/thomas` Clients tab, the DELETE handler removes the `client:{id}` record and the `clients_index` entry, but it does not remove the `client_email:{normalizedEmail}` lookup that was created when the portal was enabled for that client. The lookup row persists indefinitely and points at a now-non-existent client id. Practical impact: if the coach later tries to re-create a client with the same email (e.g. the client returns), the POST handler's duplicate-email guard at line 70 hits the orphaned key and rejects creation with `"Email already in use by another client"` — even though no client with that email actually exists. There is no admin UI to clean up the orphan, so the email is effectively burned for that workspace until someone manually deletes the Redis key.

### Where
- File: `api/dashboard/clients.js:143-154` (DELETE handler — missing `client_email:*` cleanup)
- Related: `api/dashboard/clients.js:70-73` (the duplicate guard that gets tripped)
- Related: `api/dashboard/client-portal.js:115-118` (where the lookup is created on portal-enable)

### Repro / evidence
```js
// api/dashboard/clients.js:143-154 - DELETE handler
if (req.method === 'DELETE') {
  try {
    var id = req.query.id || (req.body && req.body.id);
    if (!id) return res.status(400).json({ error: 'ID required' });
    await redis('DEL', 'client:' + id);
    await redis('ZREM', 'clients_index', String(id));
    return res.status(200).json({ ok: true });
  } catch (err) { ... }
}
```
Note the absence of any `redis('DEL', 'client_email:' + normalizeEmail(client.email))` call. The PUT handler (lines 121-132) does clean up the lookup when a portal-enabled client's email changes, and `client-portal.js` POST cleans it up when the portal is disabled — but no code path cleans it up on full client deletion.

Reproduction:
1. Admin creates client `alice@example.com`, enables portal, sets a password.
2. Admin deletes that client from the Clients tab.
3. Admin tries to re-create a client with `alice@example.com`. POST returns 400 `"Email already in use by another client"` even though no client with that email exists.

### Suggested fix
In the DELETE handler, fetch the client record before deletion, normalize its email, and `DEL` the lookup as part of the same operation. Roughly:

```js
if (req.method === 'DELETE') {
  var id = req.query.id || (req.body && req.body.id);
  if (!id) return res.status(400).json({ error: 'ID required' });
  var raw = await redis('GET', 'client:' + id);
  if (raw) {
    var existing = JSON.parse(raw);
    if (existing.email) {
      var { normalizeEmail } = require('../lib/client-auth');
      await redis('DEL', 'client_email:' + normalizeEmail(existing.email));
    }
  }
  await redis('DEL', 'client:' + id);
  await redis('ZREM', 'clients_index', String(id));
  return res.status(200).json({ ok: true });
}
```
A separate cleanup pass should also consider the per-client log/index keys (`client_dailylog:*`, `client_training_log:*`, `client_545_*`, `client_modules_*`, etc.), but those are pure data orphans rather than a lockout — the email lookup is the one that produces a user-visible error.
