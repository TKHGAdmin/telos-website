## Bug 2026-05-13-01 — Quiz retake fires duplicate submissions and stacks listeners

- **Severity**: P2
- **Area**: Quiz frontend (homepage + pricing page)
- **File**: `js/quiz.js`
- **Lines**: 225-275 (`setupLeadCapture`), 396-397 (retake button binding in `renderResults`), 415-442 (`resetQuiz`)
- **Status**: new

### What's wrong
Each time the lead-capture step is reached, `setupLeadCapture()` calls `submitBtn.addEventListener('click', …)` without removing any prior listener. Same pattern for the retake button inside `renderResults()`. `resetQuiz()` does not clear listeners.

Result: after every retake, an extra listener is bound to both the submit and retake buttons. If a user retakes the quiz once, the next submit fires the form-submit handler **twice**, sending two `POST /api/submit-quiz` requests and writing two ZADD entries to `quiz_submissions` (each with the same `name`/`email`/`scores` but slightly different `submittedAt` and `ip`, so they are NOT deduped). Retake clicks similarly multiply: 1, 2, 3, 4… `resetQuiz` invocations per click.

### Reproduction
1. Load homepage, scroll to quiz, click "Start".
2. Answer all 8 questions, fill out lead-capture form, click "See My Score".
3. Click "Retake Quiz" on the results screen.
4. Re-answer all 8 questions, fill lead-capture, click "See My Score" again.
5. Open Network tab — observe **two** `POST /api/submit-quiz` calls fire.
6. (Confirm in Redis / admin Leads tab: two rows for the same person.)

### Suggested fix
Either bind listeners exactly once at DOMContentLoaded against fixed DOM nodes, or use `onclick =` assignment instead of `addEventListener` so a new binding replaces the old, or remove the previous listener inside `resetQuiz`.

## Bug 2026-05-13-02 — Deleting a client orphans the email lookup and all per-client data

- **Severity**: P2
- **Area**: Admin dashboard — Clients tab
- **File**: `api/dashboard/clients.js`
- **Lines**: 143-154 (DELETE handler)
- **Status**: new

### What's wrong
The DELETE handler only removes two keys:

```js
await redis('DEL', 'client:' + id);
await redis('ZREM', 'clients_index', String(id));
```

It does **not** clean up:
- `client_email:{normalizedEmail}` → email-to-id lookup (created by `client-portal.js` when portal is enabled and by `clients.js` PUT on email change)
- Per-client logs/plans: `client_dailylog:*`, `client_dailylogs_index:*`, `client_nutrition_plan:*`, `client_nutrition_log:*`, `client_545_*`, `client_training_program:*`, `client_training_log:*`, `client_sidemenu:*`, `client_supplement_plan:*`, `client_supplement_log:*`, `client_activity_log:*`, push subscriptions, etc.

User-visible impact: when an admin deletes a client whose portal was enabled, **the email becomes permanently unusable for new clients**. `clients.js` POST checks `redis('GET', 'client_email:' + normalized)` and rejects with `"Email already in use by another client"` (line 71-72), even though the underlying client no longer exists.

Storage-side impact (smaller): orphaned per-client keys persist indefinitely.

### Reproduction
1. In `/thomas` → Clients tab, create a client with email `test@example.com`.
2. Click "Enable Portal" / set a password (POST `/api/dashboard/client-portal`). This writes `client_email:test@example.com → {id}`.
3. Delete the client.
4. Try to create a new client with email `test@example.com` → request fails with `400 Email already in use by another client`.

### Suggested fix
In the DELETE handler, before `DEL client:{id}`:
1. Read the client, normalize the email, and `DEL client_email:{email}`.
2. Either also `DEL` the per-client keys (use SCAN to gather, or maintain a per-client key index), or accept the orphans but at minimum clean up the lookup so email reuse works.

## Bug 2026-05-13-03 — Engagement-check cron sends a reminder every day to inactive clients

- **Severity**: P2
- **Area**: Cron — client engagement email
- **File**: `api/cron/engagement-check.js`
- **Lines**: 36-94 (per-client loop, send block)
- **Status**: new

### What's wrong
The daily cron picks any active+portal-enabled client whose most recent `client_dailylog` within the last 5 days is 3 days old or older (or absent entirely) and sends a "We miss you" email via Resend. There is **no record of when (or whether) a reminder was last sent**, so:

- A client who is inactive Day 3 → reminder sent.
- Day 4 → reminder sent again.
- Day 5 → reminder sent again.
- After 5 days the check window is exhausted and `lastLogDaysAgo === -1`, so reminders keep firing **every day forever** until the client logs in.

Realistic user impact: a one-week vacation triggers ~5 identical reminder emails. Likely outcome is "Mark as spam" or unsubscribe — net negative on retention.

### Reproduction
Static reproduction: code-trace the loop. There is no `client.lastReminderSentAt` (or similar) read; nothing prevents the same client from being emailed on consecutive days. Run-time: deploy with a client who has not logged in for 4+ days, run the cron twice 24h apart, and observe two emails delivered.

### Suggested fix
After sending, write a guard key, e.g. `client_engagement_last_sent:{clientId}` with a 7-day TTL, and skip the client if it exists. Or store `lastReminderSentAt` on the client record and require `>= 7 days` between reminders.

## Bug 2026-05-13-04 — Weekly summary email shows streak = 0 on Monday mornings even after a perfect week

- **Severity**: P3
- **Area**: Cron — weekly summary email
- **File**: `api/cron/weekly-summary.js`
- **Lines**: 86-90 (streak loop)
- **Status**: new

### What's wrong
The streak is computed starting from `d = 0` (today) and breaks on the first missing log:

```js
for (var d = 0; d < 7; d++) {
  if (weekData[0][d] && weekData[0][d].result) streak++;
  else break;
}
```

The cron is scheduled for Monday 2pm UTC (9am ET) per `vercel.json` / CLAUDE.md. At 9am ET Monday, virtually no client has logged a Monday daily-log yet — so `weekData[0][0]` is empty, the loop breaks at `d=0`, and `streak = 0`.

Result: a client who logged Sun, Sat, Fri, Thu, Wed, Tue, Mon-prior (perfect 7-day streak) receives a recap email that says "Current Streak: 0 days". That contradicts what their in-app streak shows and is demoralizing.

### Reproduction
Static reproduction: trace the loop with the assumption "today's log not yet written" — `streak` stays at 0 regardless of prior 6 days. The weekly cadence makes this almost the default Monday-morning experience.

### Suggested fix
Either start the streak count from `d = 1` (yesterday), or treat a missing `d = 0` as "look at d = 1 and start counting from there." Mirror whatever logic the in-app streak uses to stay consistent.
