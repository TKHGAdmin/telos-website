## BUG-2026-05-06-01 - P1 - Quiz lead capture form bypasses HTML5 validation; empty leads accepted

**Where:**
- `js/quiz.js:225-275` (click handler)
- `index.html:578` (button markup)
- `pricing.html:593` (button markup)

**What:**
The "See Your Score" button on the quiz lead capture step is declared as `<button type="button">` and the JS attaches an `addEventListener('click', ...)` handler that calls `e.preventDefault()` then reads `nameInput.value`, `emailInput.value`, `phoneInput.value` directly with **no validation check**. Because the button is `type="button"` (not `type="submit"`), the surrounding `<form>` is never submitted, so the `required` attributes on the name and email inputs never trigger HTML5 validation.

The empty-string values are then:
1. POSTed to `/api/submit-quiz` (which correctly rejects with 400 - "Name and email required"), and the rejection is silently swallowed by `.catch(function(){})`.
2. Stored in `localStorage` under `telosQuizCompleted=true` and `telosQuizData={...empty fields...}`.
3. The results screen renders with the score regardless.

**Repro:**
1. Open `https://www.telosathleticclub.com/` (or `/pricing`)
2. Scroll to the Execution Score Quiz, answer all 10 questions.
3. On the lead-capture screen, leave Name and Email empty.
4. Click "See Your Score".
5. Observe: results screen renders with full score breakdown; no lead recorded server-side; localStorage marked complete (so the pricing page also unlocks).

**Impact:**
- Marketing leads from the homepage and pricing-gate quiz are silently dropped whenever a visitor doesn't bother to type their info. Lead capture is the primary acquisition funnel for the business.
- The pricing-page gate also unlocks for these visitors, so even the "give us your email to see prices" trade is broken.
- Server-side rate limiting and validation are correct - the bug is purely client-side.

**Suggested fix:**
Either:
- Change the button to `type="submit"` and listen on the `<form>`'s `submit` event (browser will enforce `required`), OR
- Keep the click handler but call `form.checkValidity() && form.reportValidity()` before reading values, returning early if invalid.

Confidence: 100% - verified against current source.

## BUG-2026-05-06-02 - P1 - Engagement-check cron has no de-duplication; spams inactive clients daily

**Where:**
- `api/cron/engagement-check.js:36-94`
- `vercel.json:23-26` (cron schedule `0 15 * * *` - daily)

**What:**
The engagement-check cron runs **every day at 15:00 UTC**. It iterates every active, portal-enabled client, looks at their last 5 days of `client_dailylog:{id}:{date}` keys, and sends a Resend reminder email to any client whose most recent log is 3+ days old (or who has no log in the last 5 days).

There is **no tracking of when the last reminder was sent** to a given client. As a result, a client who hasn't logged in for 7 days will receive a reminder email on day 3, day 4, day 5, day 6, and day 7 - 5 emails in 5 days, all with subject "We miss you at Telos" and identical body copy.

**Repro:**
1. Active client with `portalEnabled=true`, `emailEnabled !== false` stops logging.
2. Day 3 after their last log: cron sends reminder #1.
3. Day 4: cron sees same condition, sends reminder #2.
4. ...continues daily until they log in or a coach disables `emailEnabled`.

**Impact:**
- High-paying clients ($1,497-$6,997 tier) get spammed when they're already disengaged - the worst possible UX moment.
- Damages brand trust and the "we care about your consistency" message the email is trying to convey.
- Burns Resend API quota linearly with the inactive-day count.

**Suggested fix:**
Before sending, check (and on success, set) a key like `client_engagement_last_sent:{clientId}` with TTL of e.g. 5-7 days. Skip clients who have a recent send. A `SETNX`/`SET ... NX EX` pattern would also work atomically.

Confidence: 100% - verified by reading the entire cron file; no de-dup branch exists.

## BUG-2026-05-06-03 - P2 - Client password reset request has no rate limit; allows email-bomb of any known client

**Where:**
- `api/client/reset-password.js:20-67` (request-reset branch)

**What:**
The "request password reset" branch accepts an unlimited number of POSTs per IP per minute. For any email that resolves to an existing portal-enabled client, the endpoint generates a fresh reset token and triggers a real email send via Resend. There is no IP-based throttle (compare with `api/submit-email.js:27-34` and `api/submit-quiz.js:22-29`, both of which gate on a `ratelimit:*:{ip}` Redis counter).

The handler does correctly avoid email enumeration (always returns `{ok: true}` regardless of whether the email resolves), so this is not an enumeration issue - it's a flooding/abuse issue.

**Repro:**
1. Find a real client email (e.g. by guessing telosfitnessllc-style domains, or by being a former lead Tom contacted).
2. Loop POSTing `{"email":"<known-email>"}` to `/api/client/reset-password`.
3. Each request that resolves to an existing client generates a token and dispatches a Resend email.

**Impact:**
- Email harassment vector against any client whose address an attacker knows.
- Resend quota burn (and bill) at attacker-chosen rate.
- Reset tokens stack up in Redis (1hr TTL each) - low storage cost but noisy.
- Account compromise is *not* possible (32-byte token entropy is fine, and the redeem branch invalidates after first use).

**Suggested fix:**
Add the same rate-limit pattern used by the other public endpoints, e.g.:
```js
const ip = req.headers['x-forwarded-for'] || 'unknown';
const key = `ratelimit:reset:${ip}`;
const count = await redis('INCR', key);
if (count === 1) await redis('EXPIRE', key, 3600);
if (count > 5) return res.status(429).json({ ok: true }); // still avoid enumeration signal
```
Consider also a per-email cap (e.g. max 3 reset emails per address per hour) to limit the attack even when the attacker rotates IPs.

Confidence: 100% - verified by reading the full handler; no rate-limit branch exists.
