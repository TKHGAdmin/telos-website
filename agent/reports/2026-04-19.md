## Bug 1 - Deleting a client orphans `client_email:*` lookup, permanently blocking email reuse

- **Severity**: P1
- **Area**: api/dashboard/clients.js (DELETE handler)
- **Reproducible**: Yes

**Steps to reproduce**
1. In /thomas, create a client with email `alice@example.com` and enable their portal. This writes `client_email:alice@example.com -> {clientId}` in Redis (api/dashboard/client-portal.js:117).
2. Delete the client via the Clients tab (DELETE /api/dashboard/clients?id={clientId}).
3. Try to recreate a client with the same email `alice@example.com` (POST /api/dashboard/clients).

**Expected**: The new client is created successfully.
**Actual**: Request is rejected with `{"error":"Email already in use by another client"}` because the orphaned `client_email:alice@example.com` key still resolves to a now-nonexistent client ID. The POST check at api/dashboard/clients.js:70-73 does not verify that the referenced client record still exists.

**Root cause**
api/dashboard/clients.js:143-154 DELETE only removes:
- `client:{id}`
- `clients_index` ZSET entry

It does NOT remove:
- `client_email:{normalizedEmail}` (needed for reuse + login cleanup)
- Any of the client's portal data (`client_dailylog:*`, `client_nutrition_plan:*`, `client_training_program:*`, `client_545_*`, `client_sidemenu:*`, `client_supplement_*`, `client_training_log:*`, `client_nutrition_log:*`, `client_activity_log:*`, `client_push_sub:*`, and their `_index` ZSETs).

**Impact**
- Admin cannot recreate a deleted client if they use the same email address - hard blocker on account recovery.
- Orphaned PII (email address) remains in Redis indefinitely after deletion, a minor privacy concern.
- Significant Redis bloat over time as portal data for deleted clients never gets cleaned.

**Suggested fix**
In api/dashboard/clients.js DELETE:
1. GET the client record first, read `client.email`, and `DEL client_email:{normalizeEmail(client.email)}`.
2. Batch-delete or SCAN+DEL the per-client portal keys, or at minimum document that deletion is a soft-delete.
3. Alternatively, in the POST uniqueness check (line 70-73), verify the referenced client actually exists before rejecting.

## Bug 2 - Quiz submit handler stacks on retake, causing duplicate `/api/submit-quiz` POSTs

- **Severity**: P2
- **Area**: js/quiz.js
- **Reproducible**: Yes

**Steps to reproduce**
1. Open / (homepage) or /pricing.
2. Complete the Execution Score quiz and submit the lead-capture form (fill name/email/phone, click "See my score"). One POST to `/api/submit-quiz` occurs.
3. Click "Retake Quiz" on the results page.
4. Complete the quiz again with any answers and submit the lead-capture form.

**Expected**: One POST to `/api/submit-quiz` per submit click.
**Actual**: Two POSTs are fired on the second submit. Each subsequent retake + submit compounds (N+1 on the Nth submission). DevTools Network panel shows duplicate POSTs with identical bodies.

**Root cause**
js/quiz.js:221-275 `setupLeadCapture()` calls `submitBtn.addEventListener('click', ...)` on the same persistent DOM node (`.quiz-see-score-btn`). `setupLeadCapture()` is invoked every time the user finishes the 8-question flow (js/quiz.js:215), and `resetQuiz()` never removes listeners, so each retake adds another handler.

Secondary instances of the same pattern (less impactful):
- js/quiz.js:397 - `retakeBtn.addEventListener('click', resetQuiz)` inside `renderResults()`. Stacks, but handler is idempotent.
- js/quiz.js:403 - `viewPricingBtn` handler inside `renderResults()`. Stacks, runs scroll+reveal multiple times.

**Impact**
- Duplicate rows in `quiz_submissions` ZSET in Redis (the JSON payload is the ZADD member, so identical retakes within the same millisecond would collide on `score` but differ in `submittedAt` millisecond, producing real duplicates).
- Lead pipeline in /thomas gets duplicate entries for the same person.
- Rate-limit counter (`ratelimit:quiz:{ip}`, 10/hr) fills twice as fast per user.
- Each retake after the first quietly wastes a Vercel function invocation.

**Suggested fix**
Two clean options:
1. In `setupLeadCapture()`, clone-replace the submit button before attaching, or use `{ once: true }` on the listener and re-attach per completion.
2. Attach the submit listener once outside `setupLeadCapture()` (inside `DOMContentLoaded`), and have it read the current quiz state when fired. Same for `retakeBtn` and `viewPricingBtn` in `renderResults()`.
