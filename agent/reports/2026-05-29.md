## BUG-2026-05-29-001 - Deleting a client leaves orphaned `client_email` lookup, blocking re-onboarding

**Severity**: P2
**Component**: `api/dashboard/clients.js` (DELETE handler)
**Status**: New

### What's wrong
When admin deletes a client via `DELETE /api/dashboard/clients?id={id}`, only the primary record (`client:{id}`) and the index entry (`clients_index`) are removed. The secondary `client_email:{normalizedEmail}` lookup is left pointing to the now-deleted client ID.

The next time admin tries to add a client with the same email - the realistic case being a former client returning - the POST handler hits this orphaned key at `api/dashboard/clients.js:70` and rejects the request with "Email already in use by another client". There is no admin UI to clear the orphan.

Also leaked (lower-impact but worth noting in the same fix): `client_dailylog:{id}:*`, `client_dailylogs_index:{id}`, `client_545_*:{id}*`, `client_nutrition_*:{id}*`, `client_training_*:{id}*`, `client_supplement_*:{id}*`, `client_activity_log:{id}:*`, `client_supplement_plan:{id}`, `client_supplement_logs_index:{id}`, `client_sidemenu:{id}`, `client_mindset:{id}`, `client_resources:{id}`, `client_training_program:{id}`, `client_nutrition_plan:{id}`. These don't surface to the admin immediately but do bloat Redis.

### Reproduction
1. In `/thomas`, add a new client with email `test@example.com`.
2. Open the client in the Client Portal tab, enable portal access (this calls `POST /api/dashboard/client-portal` which writes `client_email:test@example.com -> {id}`).
3. Delete the client via the Clients tab (DELETE).
4. Try to add a new client with the same email `test@example.com`.
5. **Expected**: Client is created.
6. **Actual**: 400 response with `{ "error": "Email already in use by another client" }`. Only workaround is direct Redis access (`DEL client_email:test@example.com`).

### Code locations
- Bug site - `api/dashboard/clients.js:143-154` (DELETE handler does not call `DEL client_email:...`).
- Surfaces at - `api/dashboard/clients.js:70` (POST email-uniqueness check) and indirectly at `api/client/login.js:24` and `api/client/reset-password.js:22` (both `GET client_email:` then load the missing `client:{id}` record; they correctly fail with generic "Invalid email or password", but the lookup never gets cleaned by these calls).
- Confirms - `grep -n client_email api/`: cleanup only happens in `client-portal.js:123` (`enabled === false`) and `clients.js:130` (PUT email change with portal enabled).

### Suggested fix
In the DELETE branch, before removing the primary record:

```js
var existing = await redis('GET', 'client:' + id);
if (existing) {
  try {
    var c = JSON.parse(existing);
    if (c.email) {
      var { normalizeEmail } = require('../lib/client-auth');
      await redis('DEL', 'client_email:' + normalizeEmail(c.email));
    }
  } catch (e) {}
}
```

Optionally also `DEL`/`ZREM` the per-client secondary keys listed above; safest is to scan-pattern them or maintain a `client_keys:{id}` SET written at create time so cleanup is bounded.

### Why I'm confident
- Search of `api/` for `client_email` shows exactly three writers (POST in clients.js, PUT in clients.js, POST in client-portal.js) and zero callers in the DELETE flow.
- I verified the POST path that surfaces the error: `clients.js:70` does an unconditional `GET client_email:{normalized}` and a 400 short-circuit if anything is returned, regardless of whether the referenced client still exists.

## BUG-2026-05-29-002 - Engagement-check cron sends a "We miss you" email every day to inactive clients

**Severity**: P2
**Component**: `api/cron/engagement-check.js`
**Status**: New

### What's wrong
The engagement-check cron runs daily at 15:00 UTC. For each active, portal-enabled client whose most recent `client_dailylog:{id}:{date}` is 3+ days old (or absent in the last 5 days), it sends a "We miss you at Telos" email via Resend. There is no per-client last-sent timestamp and no throttle - if a client stays inactive, they receive the same email every single day.

Realistic scenario: a client on a 14-day vacation returns to a dozen "We miss you" emails. Beyond the user-experience cost, repeated identical "we miss you" subjects to the same recipient is a strong spam/deliverability signal that can damage `noreply@telosathleticclub.com`'s sender reputation across all clients.

### Reproduction
1. Have an active, portal-enabled client with `emailEnabled !== false` and no `client_dailylog:{id}:*` entries for the last 5 days.
2. Trigger the cron (or wait for the daily run): `POST /api/cron/engagement-check` with `Authorization: Bearer $CRON_SECRET`.
3. Note an email is sent and the client's record is unchanged.
4. Wait 24 hours, run the cron again. Another identical email is sent. Repeat indefinitely.

### Code locations
- `api/cron/engagement-check.js:61` - the gate is purely "are they inactive 3+ days?". No "did we already remind them?" check.
- `api/cron/engagement-check.js:76-90` - send call has no follow-up Redis write to record the send.
- `grep -rn 'lastEngagementEmail\|reminderSentAt' api/ client-dashboard.html thomas.html` returns nothing - no infrastructure exists.

### Suggested fix
Smallest patch: write a `client_engagement_sent:{clientId}` key with a TTL (e.g. 7 days) right after a successful Resend call, and bail out at the top of the per-client loop if that key exists:

```js
var alreadySent = await redis('GET', 'client_engagement_sent:' + client.id);
if (alreadySent) continue;
// ...send email...
await redis('SETEX', 'client_engagement_sent:' + client.id, 7 * 86400, '1');
```

Alternative: stamp `client.lastEngagementEmailAt` onto the `client:{id}` record and skip in-loop when `now - lastEngagementEmailAt < 7 * 86400000`. Lower Redis churn but means rewriting the client record from a cron, which is heavier.

Either approach makes the reminder weekly-at-most per client; tune the TTL/threshold to match Thomas's outreach cadence.

### Why I'm confident
- The logic is self-contained in one file - I read the entire handler and there's no state-tracking branch.
- No code elsewhere writes any `lastEngagement*` / `reminderSent*` fields onto client records (grep returns zero hits).
- This is verifiable purely by reading the code; no need to run the cron.
