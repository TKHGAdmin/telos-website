## Finding 1: Client login leaks email existence via differentiated errors and PBKDF2 timing

- **id**: 2026-05-27-1
- **severity**: P2
- **confidence**: high
- **file**: api/client/login.js:24-47
- **status**: new

### What

The client login endpoint returns five distinct error responses depending on the account state of the submitted email, allowing an unauthenticated caller to enumerate which emails belong to active Telos clients. There is also a strong timing oracle: nonexistent emails return at line 26 before any password hashing, while existing emails fall through to a 100k-iteration PBKDF2 at line 46 (`client-auth.js:16`).

### Reproduction / Evidence

- `api/client/login.js:24-26` → "Invalid email or password" if `client_email:{email}` is unset (email unknown to system).
- `api/client/login.js:36-37` → "Portal access not enabled" if email is known but `portalEnabled` is false.
- `api/client/login.js:39-40` → "Account is not active" if email known but `status !== 'active'`.
- `api/client/login.js:42-43` → "Password not set - contact your coach" if email known but no `passwordHash`/`passwordSalt`.
- `api/client/login.js:46-47` → "Invalid email or password" if password verification fails.
- PBKDF2 at `api/lib/client-auth.js:16` runs 100,000 SHA-512 iterations — the request-duration delta between an unknown email (~Redis GET only) and a known email (Redis GET + PBKDF2) is easily measurable from the outside, distinguishing the two "Invalid email or password" branches.

Because the client roster is small and members are publicly visible (testimonials, IG @telos_fitness, blog mentions), an attacker can enumerate the active membership and target known-existing accounts for credential stuffing.

### Suggested fix

Collapse all auth failures (unknown email, disabled portal, inactive status, missing password, wrong password) to a single generic response such as `401 { error: 'Invalid email or password' }`. To neutralize the timing oracle, always run a PBKDF2 against a fixed dummy hash/salt when the email lookup fails, so all paths spend the same ~100k iterations before returning.

## Finding 2: Admin session signature comparison is not timing-safe

- **id**: 2026-05-27-2
- **severity**: P3
- **confidence**: high
- **file**: api/lib/auth.js:27
- **status**: new

### What

`verifyToken()` in the admin auth module compares the request-provided HMAC signature against the expected signature with `signature !== expected` (plain string compare). The sibling `verifyClientToken()` correctly uses `crypto.timingSafeEqual` (`api/lib/client-auth.js:49`). The admin path is inconsistent with the codebase's own timing-safe pattern.

### Reproduction / Evidence

- `api/lib/auth.js:23-27`:
  ```js
  const expected = crypto.createHmac('sha256', SESSION_SECRET).update(payload).digest('hex');
  if (signature !== expected) return false;
  ```
- `api/lib/client-auth.js:46-49` uses the safe pattern for the same shape of comparison.
- Practical exploitability against `/api/dashboard/*` is low (Vercel function jitter is large vs hex-char timing deltas, and the secret space is 2^256), but the inconsistency violates the project's own established standard and is a one-line fix.

### Suggested fix

Replace `auth.js:27` with the buffer-pair `timingSafeEqual` pattern already used in `client-auth.js:46-49`.

## Finding 3: submit-quiz.js accepts any non-empty string as `email`

- **id**: 2026-05-27-3
- **severity**: P3
- **confidence**: high
- **file**: api/submit-quiz.js:16-20
- **status**: new

### What

The quiz submission endpoint only checks that `email` is truthy — it does not validate format. Sibling public endpoints validate (`submit-email.js:18` rejects without `@`; `submit-chs-application.js:35` runs an RFC-shaped regex). As a result the `quiz_submissions` ZSET in Redis accumulates entries with invalid `email` strings such as `"asdf"`, polluting the leads pipeline shown in `/thomas`.

### Reproduction / Evidence

- `api/submit-quiz.js:18-20`:
  ```js
  if (!email || !name) {
    return res.status(400).json({ error: 'Name and email required' });
  }
  ```
- No format check follows; the entry is written to Redis at line 44.
- Manual repro: `POST /api/submit-quiz` with `{ name: 'x', email: 'x', totalScore: 0, scores: [] }` returns `200 { ok: true }` and inserts the record.

### Suggested fix

Add a minimal validation matching the other endpoints, e.g. `if (!/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email)) return res.status(400).json({ error: 'Valid email required' });` between the existing required-field check and the rate limiter.
