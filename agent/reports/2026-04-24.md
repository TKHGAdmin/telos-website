## BUG-2026-04-24-01 — `thomas.jpeg` is a 1.85 MB source file rendered at max 350px height

- **Severity:** P1
- **Area:** Homepage + Charleston landing page (public, above-or-near the fold)
- **File(s):**
  - `images/thomas.jpeg` — 1,897,318 bytes, 1206x2136 px, baseline JPEG, quality ~90+
  - `index.html:250`
  - `chs.html:946`
  - `css/style.css:2284-2292` (`.about-photo { width: 100%; max-height: 350px; object-fit: cover; }`)

**What's wrong**

The founder photo served on both the homepage About section and the Charleston landing page About section is a near-uncompressed Lightroom-export-grade JPEG: 2136 px tall and 1.85 MB. CSS renders it at `max-height: 350px`, i.e. ~6× smaller than the source resolution. The image also has no `loading="lazy"` attribute, so it downloads on every initial pageview of `index.html` and `chs.html` regardless of whether the user scrolls to the About section.

On a 4G mobile connection (~1.5 Mbps sustained) this adds roughly 10 seconds of transfer time to a page that is otherwise type-only above the fold, and ~1.5 MB to the mobile data budget per first visit. The other photos in the same section (`thomas-speaking-wide.jpeg` 251 KB, `thomas-speaking-close.jpeg` 216 KB) are proportionally sized — `thomas.jpeg` is the clear outlier.

**Repro**

```bash
curl -sI https://www.telosathleticclub.com/images/thomas.jpeg | grep -i content-length
# Expect: ~1,897,318
file images/thomas.jpeg
# Expect: 1206x2136, baseline JPEG
```

In DevTools, load `/` on a throttled "Fast 3G" profile and watch the Network panel — `thomas.jpeg` dominates the waterfall.

**Fix (suggested)**

1. Re-export `thomas.jpeg` at ~800×1424 px (2x DPR for a 400px render target), JPEG quality 78-82. Expect ~120-200 KB.
2. Add `loading="lazy"` and `decoding="async"` to the `<img>` at `index.html:250` and `chs.html:946`.
3. Optional: serve a `.webp` companion and use `<picture>` for modern browsers — cuts another ~25%.

Estimated savings: **~1.5 MB per first pageview on two of the three highest-traffic public pages.**

## BUG-2026-04-24-02 — Nutrition tab fans out 7 serverless requests for weekly chart; missing `?range` on `/api/client/nutrition-log`

- **Severity:** P1
- **Area:** Client Dashboard (authenticated, PWA hot path)
- **File(s):**
  - `client-dashboard.html:4698-4721` (`renderNtWeeklyChart()`)
  - `api/client/nutrition-log.js:11-23` (GET handler, no range support)
  - Compare: `api/client/daily-log.js:22-44` (pipelined range mode — the existing pattern to follow)
  - `api/lib/redis.js:20-31` (`redisPipeline` helper already exists)

**What's wrong**

Every time a client opens the Nutrition tab, `loadNutritionTab()` fires 5 API calls, then `renderNtWeeklyChart()` immediately fires **7 more parallel requests** — one `GET /api/client/nutrition-log?date=YYYY-MM-DD` per day for the last 7 days — just to compute one bar chart's calorie totals:

```js
// client-dashboard.html:4713
Promise.all(dates.map(function(d) {
  return api('nutrition-log?date=' + d).catch(function() { return { log: null }; });
}))
```

That's 7 independent browser → Vercel serverless invocations, each of which opens its own Upstash `/` call. The `nutrition-log.js` GET handler accepts only a single `date` query param — it has no `range` or `mode=recent` mode like `daily-log.js` and `training-log.js` do.

This is a pure N+1 pattern. It's wasteful in three dimensions:

1. **Latency:** On a cold region, cold-start variance across 7 invocations widens the tail and the chart-load wall time is determined by the slowest of 7 cold starts. A single call is bounded by one cold start plus one Redis pipeline round-trip.
2. **Serverless cost:** 7x invocation count per Nutrition tab open per client. Every returning client who opens Nutrition triggers this.
3. **Upstash cost:** 7x round-trips vs one pipelined `MGET`-equivalent. Every one of these 7 is its own Upstash REST POST.

The pattern is recognized elsewhere in the codebase — `daily-log.js:31-44` pipelines N days into one `redisPipeline([['GET', key1], ...])` call. This bug is a gap where that pattern wasn't applied.

**Repro**

1. Log into `/client-dashboard` as any test client.
2. Open DevTools → Network.
3. Tap the "Nutrition" bottom tab.
4. Filter on `/api/client/nutrition-log` — observe 7 sequential entries in the waterfall (in addition to the 1 legitimate `?date=<today>` call from `loadNutritionTab`).

**Fix (suggested)**

1. Add a `range` mode to `api/client/nutrition-log.js` mirroring `daily-log.js:14-44`:
   - Parse `?range=7` (or `?mode=recent&limit=7`).
   - Build N keys via a loop, call `redisPipeline(cmds)` once, return `{ logs: [...] }`.
2. In `client-dashboard.html:4703-4721`, replace the `Promise.all(dates.map(...))` with a single `api('nutrition-log?range=7')` call. Map dates from the result array index instead of per-request.

Reduces the Nutrition tab's initial request count from **12 → 6** and drops 6 serverless + 6 Upstash round-trips per load. No behavioral change expected.

## BUG-2026-04-24-03 — No `loading="lazy"` on any public-page images; below-the-fold photos on `/chs` download eagerly

- **Severity:** P2
- **Area:** Public marketing pages
- **File(s):**
  - `chs.html:922, 925, 928, 946, 950, 953` — 6 images below the fold, none lazy
  - `index.html:250, 256, 259` — 3 About-section images, none lazy
  - `chs.html:824` is hero and correctly `loading="eager" fetchpriority="high"` — leave alone

**What's wrong**

A grep across all 6 public pages (`index`, `pricing`, `chs`, `resources`, `protein-calculator`, `hyrox-predictor`) finds **zero** occurrences of `loading="lazy"`. Every `<img>` tag below the fold downloads during initial parse. On `/chs` this is particularly expensive because there are 3 event photos (chs-events-01/02/03.jpg totalling ~236 KB) plus the 3 Thomas photos (1.85 MB + 251 KB + 216 KB ≈ 2.3 MB) stacked in the Events and About sections — both of which most visitors won't scroll to before bouncing.

Separately from bug #1 (which fixes the 1.85 MB outlier at the source), adding `loading="lazy"` on non-hero images is a zero-risk one-line-per-tag change that defers ~0.3-0.5 MB on mobile until scroll.

**Repro**

```bash
grep -c 'loading="lazy"' index.html pricing.html chs.html resources.html \
  hyrox-predictor.html protein-calculator.html
# All zero
```

**Fix (suggested)**

Add `loading="lazy" decoding="async"` to every `<img>` below the first viewport on these pages. Keep the existing `loading="eager" fetchpriority="high"` on the `/chs` hero at `chs.html:824` — that's the correct call for an LCP image.
