## BUG-2026-05-05-01 - Charleston page is missing the Client Login nav link

- **Severity**: P2
- **Category**: Visual / UX (navigation consistency)
- **Status**: New

### Where
- `chs.html` lines 793-815 (desktop `<ul class="nav-links">` and `<div class="mobile-menu">`)

### What
Every other public page (index, pricing, protein-calculator, hyrox-predictor, resources, all 23 blog pages) has a `client-dashboard` "Client Login" link placed before "Book a Call" in both desktop and mobile navs. `chs.html` has zero links to `client-dashboard` in either nav.

CLAUDE.md explicitly documents this convention:
> **Client Login nav link**: All 28 public pages (5 root + 23 blog) have a "Client Login" link before "Book a Call" in both desktop and mobile navs.

`chs.html` is the 6th public root page and was added after that convention; the convention was not extended to it.

### Why it matters
An existing client who lands on `/chs` (search, social, word of mouth) has no in-nav path to log into their portal. They have to back out to `/` first. Cheap, one-off polish but a clear regression on a documented site-wide standard.

### Reproduce
1. Open `chs.html` in a browser at desktop or mobile width.
2. Inspect the nav: there's no "Client Login" entry.
3. Compare to `index.html` line 51 / `pricing.html` line 519 - both have `<li><a href="client-dashboard" class="nav-client-login">Client Login</a></li>` before the Book-a-Call button.

### Suggested fix
Add `<li><a href="client-dashboard" class="nav-client-login">Client Login</a></li>` to both the desktop `nav-links` (before the `Apply` CTA at line 799) and the mobile menu list (before `Apply` at line 813).

## BUG-2026-05-05-02 - Quiz lead-capture form has no client-side validation; pricing gate is bypassable with empty fields

- **Severity**: P2
- **Category**: Functional (surfaced during Visual/UX review of forms)
- **Status**: New

### Where
- `js/quiz.js` lines 225-275 (`setupLeadCapture` click handler)
- `index.html` line 578 (homepage quiz lead-capture button)
- `pricing.html` line 593 (pricing-page quiz overlay button)

### What
The "See Your Score" button is `<button type="button">`, so clicking it never triggers a form submit. That means the HTML5 `required` attribute on the name/email/phone inputs never fires. The click handler reads `nameInput.value` etc. without checking they're non-empty, fires a fire-and-forget `fetch` (whose `.catch` only catches network errors, not 4xx), and unconditionally:

```js
localStorage.setItem('telosQuizCompleted', 'true');     // line 247
localStorage.setItem('telosQuizData', JSON.stringify({...}));
quizLeadCapture.style.display = 'none';
quizResults.style.display = 'block';
renderResults();
```

Net effect when a user clicks "See Your Score" with empty fields:
- `/api/submit-quiz` rejects with 400 ("Name and email required") - lead is **not** captured in Redis.
- Client still sets `telosQuizCompleted=true`.
- Quiz results render.
- `pricing.html`'s gate (which checks `localStorage.telosQuizCompleted`) unlocks - the user sees actual pricing without ever providing contact info.

### Why it matters
The lead-capture form is the only chance to attach contact info to a completed quiz before the user sees their score. A motivated visitor (or anyone fast-clicking) bypasses it and Thomas loses the lead while still letting the visitor through the pricing gate. Even casual users who tab past the inputs by accident hit this path.

### Reproduce
1. Load `https://telosathleticclub.com/` and scroll to the Execution Score Quiz.
2. Answer all 9 questions (any answers).
3. On the lead-capture screen, leave name / email / phone empty and click "See Your Score".
4. Results render. Open DevTools - `localStorage.getItem('telosQuizCompleted')` returns `'true'`.
5. Network tab shows `POST /api/submit-quiz` returning 400.
6. Navigate to `/pricing` - the quiz overlay does not appear; prices are revealed.

### Suggested fix
In `setupLeadCapture` (js/quiz.js), validate the three fields before the localStorage / fetch / render block:

```js
if (!nameInput.value.trim() || !emailInput.value.trim() || !phoneInput.value.trim()) {
  // show inline error; don't proceed
  return;
}
```

Optionally also gate the `localStorage.setItem('telosQuizCompleted', 'true')` line on a successful API response (read `res.ok` from the fetch). The current "fire-and-forget" pattern means a server-side rejection is invisible to the gate.

## BUG-2026-05-05-03 - Charleston application submit button is stuck on "Sending..." after a successful submission

- **Severity**: P3
- **Category**: Visual / UX (button state)
- **Status**: New

### Where
- `chs.html` lines 1106-1119 (`fetch('/api/submit-chs-application', ...)` success branch)

### What
The submit handler sets `submitBtn.textContent = 'Sending...'` before the fetch (line 1106). On success, the handler dims the form (`form.style.opacity = '0.35'; form.style.pointerEvents = 'none';`) and shows a thank-you message - but the button text is never reset. The user sees a faded form with a button that still reads "Sending..." even after the success message appears.

```js
.then(function (result) {
  if (result.ok && result.body && result.body.ok) {
    form.reset();
    form.style.opacity = '0.35';
    form.style.pointerEvents = 'none';
    showStatus('Thank you. We’ll be in touch within 24 hours.', false);
  } else {
    // failure branch correctly resets the button text
    submitBtn.disabled = false;
    submitBtn.textContent = 'Submit Application';
  }
})
```

### Why it matters
Mild but real confusion: the "Sending..." label suggests the request is still in-flight even though the success message already says "Thank you". Cheap one-line fix.

### Reproduce
1. Open `/chs`, fill the application with valid values, click Submit.
2. On a successful response, observe the form fades out but the button text is still "Sending...".

### Suggested fix
In the success branch, set `submitBtn.textContent = 'Application Sent'` (or similar) so the button visibly resolves to a final state.

## BUG-2026-05-05-04 - Public lead-capture inputs missing autocomplete + visible labels (mobile autofill broken)

- **Severity**: P2
- **Category**: Visual / UX (accessibility + mobile UX)
- **Status**: New

### Where
- `index.html` lines 575-577 (homepage quiz lead capture - name / email / phone)
- `pricing.html` lines 590-592 (pricing-page quiz overlay - name / email / phone)
- `protein-calculator.html` line 644 (`#gateEmail`)
- `hyrox-predictor.html` line 668 (`#gateEmail`)

### What
None of these public lead-capture inputs have `autocomplete` attributes, an associated `<label for="...">`, or `aria-label`. They rely entirely on `placeholder` text for context.

Concrete impact:
- Mobile browsers (iOS Safari, Android Chrome) cannot offer autofill suggestions for name / email / phone, so users type by hand. Higher friction, lower conversion.
- Screen readers announce the placeholder once on focus and lose it the moment the user starts typing - the field has no programmatic name.

The same site already does this correctly elsewhere - `chs.html` lines 994-1003 use `autocomplete="name"`, `autocomplete="email"`, `autocomplete="tel"` and proper `<label for="...">` associations. So this is inconsistent within the codebase, not a stylistic call.

### Why it matters
Lead-capture conversion on mobile is sensitive to friction; autofill is the single biggest UX improvement available here and is one HTML attribute. A11y impact is real for screen reader users on the most important conversion forms on the site.

### Reproduce
1. Load `index.html` on a mobile device, scroll to the quiz, complete it, reach the lead capture.
2. Tap the email field - no autofill chip appears above the keyboard.
3. Compare to `/chs` form, where Safari/Chrome offer the saved email immediately.

### Suggested fix
On each of the four input clusters, add `autocomplete="name"`, `autocomplete="email"`, `autocomplete="tel"` and either an associated `<label>` or `aria-label` matching the placeholder.
