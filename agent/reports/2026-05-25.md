### [P1] Product detail page renders in wrong columns (3 children in 2-col grid)

**ID:** bug-2026-05-25-01
**File:** product.html:105-108, product.html:765-779
**Type:** functional

**What:**
`.product-detail.loaded` is `display: grid; grid-template-columns: 1fr 1fr;`, but the markup contains three direct children: `#productImage`, `#productThumbs`, and `.product-detail-info`. CSS grid auto-flow places `#productThumbs` in cell (row 1, col 2) — exactly where `.product-detail-info` should live — and pushes `.product-detail-info` down to (row 2, col 1) at half-width.

**Why it matters:**
Every desktop visitor opening a Shopify product is affected. On a single-image product the thumbs container is empty but still occupies the right cell, so the title/price/Add-to-Cart column collapses to half-width on row 2. On a multi-image product the thumbnails replace the buy column entirely. Mobile escapes this because the `@media (max-width: 768px)` rule collapses to one column. The image-gallery feature was just shipped (commit `f5cb8c4`), so this is fresh.

**Repro:**
1. Visit `/product?p=<any-handle>` on a desktop viewport (>768px).
2. Observe the title, price, and "Add to Cart" button render in the bottom-left at half the page width, with thumbnails (or empty space) occupying the top-right.

Concretely the markup at product.html:765-779 is:
```html
<div class="product-detail" id="productDetail">
  <div class="product-detail-image" id="productImage"></div>
  <div class="product-detail-thumbs" id="productThumbs"></div>   <!-- ends up in col 2 -->
  <div class="product-detail-info"> ... </div>                   <!-- wraps to row 2 -->
</div>
```

**Suggested fix:**
Wrap image + thumbs in a single column container so the grid sees only two children:

```html
<div class="product-detail" id="productDetail">
  <div>
    <div class="product-detail-image" id="productImage"></div>
    <div class="product-detail-thumbs" id="productThumbs"></div>
  </div>
  <div class="product-detail-info"> ... </div>
</div>
```

Or set `grid-column: 1` explicitly on `.product-detail-image` and `.product-detail-thumbs` and `grid-column: 2; grid-row: 1 / span 2` on `.product-detail-info`.

### [P2] Quiz retake double-submits to /api/submit-quiz

**ID:** bug-2026-05-25-02
**File:** js/quiz.js:221-276
**Type:** functional

**What:**
`setupLeadCapture()` attaches a new click handler to `.quiz-see-score-btn` every time it is called. It is called once per quiz completion, but the submit button itself is a stable DOM node — the previous handler is never removed. On the second pass (user retakes the quiz), one click fires two handlers and sends two `POST /api/submit-quiz` requests with identical payloads.

**Why it matters:**
Every retake produces a duplicate row in the `quiz_submissions` ZSET. The dashboard's submissions tab will show the same lead twice with consecutive timestamps, which Thomas would have to manually dedupe. It also doubles the rate-limit counter for legitimate retakers (10/hr cap).

**Repro:**
1. Complete the quiz once (questions → lead capture → see score).
2. Click "Retake quiz" on the results screen.
3. Complete the 8 questions again, fill the lead capture, click submit.
4. Watch the Network tab — two identical `POST /api/submit-quiz` requests fire.

**Suggested fix:**
Move the handler attach out of `setupLeadCapture()` and into the DOMContentLoaded block (attach once, read current quiz state at click time). Or guard with a flag:

```js
function setupLeadCapture() {
  if (setupLeadCapture._wired) return;
  setupLeadCapture._wired = true;
  // ... existing handler attach ...
}
```

Same pattern exists for `.quiz-retake-btn` (line 397) and the pricing-page "View Pricing" button (line 403) — both should be deduped in the same pass.

### [P2] Weekly summary email almost always reports streak = 0

**ID:** bug-2026-05-25-03
**File:** api/cron/weekly-summary.js:86-90
**Type:** functional

**What:**
The streak loop counts consecutive logged days starting at `d=0` (today, UTC) and breaks on the first miss. The cron fires Monday 14:00 UTC = 09:00 ET, before nearly any client has done Monday's check-in. Result: even a client who logged 6 days in a row last week gets an email saying "Current Streak: 0 days".

**Why it matters:**
This is the headline metric in the weekly recap email. Showing "0 days" to a consistent client is demoralizing and undermines trust in the platform's tracking. It will land in their inbox every Monday morning.

**Repro:**
1. Seed a client's `client_dailylog:{id}:{YYYY-MM-DD}` for the 7 days ending Sunday (no Monday log).
2. Invoke `api/cron/weekly-summary` with the cron secret.
3. The email renders `Current Streak: 0 days` despite the 7-day run.

**Suggested fix:**
Start the streak count from yesterday (`d=1`) when today is unlogged, or count all consecutive non-zero days in the window without forcing today to be included:

```js
// Allow today to be empty without zeroing the streak
var start = (weekData[0][0] && weekData[0][0].result) ? 0 : 1;
for (var d = start; d < 7; d++) {
  if (weekData[0][d] && weekData[0][d].result) streak++;
  else break;
}
```

### [P2] Engagement-check resends the "we miss you" email every single day

**ID:** bug-2026-05-25-04
**File:** api/cron/engagement-check.js:61-94
**Type:** functional

**What:**
The cron runs daily and sends a reminder to any active client whose most recent daily log is 3+ days old (or absent in the last 5 days). It does not record that a reminder was sent. As long as the client stays inactive, they receive the same "It's been N+ days since your last check-in" email every 24 hours.

**Why it matters:**
A genuinely lapsed client gets 7+ duplicate emails in a week. That's spam-grade behavior from a premium portal — likely to push them to mark the sender as junk, harming deliverability of the legitimate Monday recap. The text also says "N+ days" with N being the lookback floor (3, 4, or 5), which doesn't escalate meaningfully across days.

**Repro:**
1. Set a client `portalEnabled: true, emailEnabled: true, status: 'active'` with no `client_dailylog:*` keys.
2. Hit `api/cron/engagement-check` with the cron secret on three consecutive days.
3. Resend logs / inbox shows three identical emails to the same address.

**Suggested fix:**
Store a `client_last_engagement_email:{clientId}` key with a TTL (e.g. 5 days) before sending. Skip clients where that key exists:

```js
var sentKey = 'client_last_engagement_email:' + client.id;
var alreadySent = await redis('GET', sentKey);
if (alreadySent) continue;
// ... send email ...
await redis('SET', sentKey, '1', 'EX', 5 * 24 * 3600);
```

Or move the schedule to weekly and tune the lookback. Either is fine — current behavior is the worst of both worlds.
