## BUG-2026-05-11-001 — Deleting a client leaks email lookup and all portal data

- **Severity:** P2
- **Focus:** functional
- **Status:** new
- **File:** `api/dashboard/clients.js:143-154`
- **Confidence:** high

### Summary
The admin client DELETE handler only removes `client:{id}` and the `clients_index` ZSET entry. It does not remove the `client_email:{normalizedEmail}` reverse lookup, nor any of the per-client portal/log keys documented in CLAUDE.md (`client_nutrition_plan:`, `client_training_program:`, `client_dailylog:*`, `client_545_*`, `client_supplement_*`, `client_sidemenu:`, etc.).

### Reproduction
1. Create a client with email `foo@bar.com` in the Clients tab.
2. Enable the portal for that client (this writes `client_email:foo@bar.com → id`).
3. Optionally have the client submit one daily log so a `client_dailylog:{id}:{date}` row exists.
4. Delete the client from the Clients tab (calls `DELETE /api/dashboard/clients`).
5. Try to re-add a client with email `foo@bar.com`. The POST handler at `clients.js:70-73` rejects with `"Email already in use by another client"` because the orphan `client_email:foo@bar.com` still points at the old, now non-existent id.
6. The orphan portal rows (`client_nutrition_plan:{oldId}`, daily logs, etc.) remain in Redis indefinitely. They will never be served (login at `api/client/login.js:24-32` returns 401 once `client:{oldId}` is missing), but they take up storage and represent leaked PII for a deleted client.

### Why this matters
- Coach cannot reuse the email of a deleted client without manually purging Redis — a footgun during real onboarding flows.
- Client PII (daily logs, weight, mood, notes) survives "delete" with no GDPR-style purge path.

### Suggested fix
In the DELETE branch of `clients.js`, before removing `client:{id}`:
1. Read the existing client to recover `client.email`.
2. `DEL client_email:{normalizedEmail}` (using `normalizeEmail` from `lib/client-auth`).
3. Pipeline-delete all per-client keys: `client_nutrition_plan:{id}`, `client_mindset:{id}`, `client_resources:{id}`, `client_training_program:{id}`, `client_sidemenu:{id}`, `client_supplement_plan:{id}`, `client_545_goals:{id}`, `client_545_routine:{id}`, plus all dated logs via `ZRANGE client_dailylogs_index:{id} 0 -1` etc.
4. Delete the indexes themselves (`client_dailylogs_index:{id}`, `client_training_logs_index:{id}`, `client_nutrition_logs_index:{id}`, `client_supplement_logs_index:{id}`, `client_545_index:{id}`, `client_activity_logs_index:{id}`).

## BUG-2026-05-11-002 — Quiz scoring is inflatable via fast multi-click on a single question

- **Severity:** P2
- **Focus:** functional
- **Status:** new
- **File:** `js/quiz.js:198-218`
- **Confidence:** high

### Summary
`handleAnswerSelect` adds points to `quiz.scores[pillar]` immediately on click, then waits 400ms before advancing to the next question. During that 400ms window, the option buttons remain clickable — there is no `disabled` set, no `once: true`, no debounce, no `if (advancing) return` guard. Any extra click on a different option (or even an accidental double-tap on mobile) adds another `option.points` to the same pillar.

### Reproduction
1. Open the quiz on `/` or `/pricing`.
2. On any question, rapidly click two different options within 400ms.
3. Both option point values are added to the pillar. With 1+5 options you can reach `pillar.score = 10` on a single question instead of the intended 5.

### Why this matters
- Mobile fast-taps (which the iOS double-tap-zoom heuristic can register as two clicks ~300ms apart) will silently inflate scores.
- Inflated scores push leads into a higher tier ("Executing" instead of "Almost Dialed"), which changes the messaging Thomas sees in the dashboard and may misroute follow-up.
- If total inflated score exceeds 40 (the max of the last `resultTier.maxScore`), `getResultTier` at `quiz.js:285` returns `undefined`, and `renderResults` will throw on line 325 (`tier.name`) — the user sees a broken results page after submitting their lead info, but the lead is already POSTed to `/api/submit-quiz`, so Thomas sees a totalScore > 40 with no tier classification client-side.

### Suggested fix
In `handleAnswerSelect`, before the points line, guard against re-entry:

```js
function handleAnswerSelect(button, option, pillar) {
  if (quiz.advancing) return;
  quiz.advancing = true;
  // disable all option buttons in this question
  optionsContainer.querySelectorAll('.quiz-option').forEach(b => b.disabled = true);
  // ... existing points + setTimeout logic
  // inside the setTimeout, reset quiz.advancing = false before renderQuestion()
}
```

Also harden `getResultTier` to return the highest tier when `score > 40` so the results page can't crash if any future bug produces an over-max score.

## BUG-2026-05-11-003 — HMAC signature comparison in admin session is not timing-safe

- **Severity:** P3
- **Focus:** functional (consistency / correctness)
- **Status:** new
- **File:** `api/lib/auth.js:27` (compare with `api/lib/client-auth.js:46-49`)
- **Confidence:** high

### Summary
Admin `verifyToken` compares the HMAC signature with a plain `!==`:

```js
if (signature !== expected) return false;
```

The client-side equivalent in `lib/client-auth.js` for the exact same construction (HMAC over a payload, hex-encoded, split out of the cookie) uses `crypto.timingSafeEqual`. The two files were clearly written with different conventions for the same primitive. The admin file is the inconsistent one.

### Why this matters
- Real exploitability of a string-compare timing leak on HMAC-SHA256-hex over the public internet is low (jitter ≫ ns-scale leak), so I'm rating this P3 not P0. But it's the same data flow as `client-auth.js` which Thomas already wrote correctly — that asymmetry is the actual bug here, and it's a one-line fix that brings the two files into agreement.
- A consistent codebase is easier to audit. Future readers will assume the admin path is also timing-safe.

### Suggested fix
Replace `auth.js:23-27` with the same pattern used in `client-auth.js`:

```js
const expected = crypto.createHmac('sha256', SESSION_SECRET).update(payload).digest('hex');
const a = Buffer.from(signature, 'hex');
const b = Buffer.from(expected, 'hex');
if (a.length !== b.length) return false;
if (!crypto.timingSafeEqual(a, b)) return false;
```

## Items considered and excluded

These were investigated and intentionally NOT reported, to keep first-run precision high. Listed here so Thomas can sanity-check the triage and so future runs can see the exclusion logic.

- **`.html` extensions in internal nav links** — already documented in `CLAUDE.md` "Known Limitations" as a P3 backlog item; `cleanUrls: true` makes them work (with a 308 redirect hop).
- **Admin `/api/dashboard/login` has no rate limit and `verifyPassword` short-circuits on length mismatch** — real concern, but the password leak surface is tiny (1 byte of entropy), and this fits the security-focus day better. Saved for next security rotation.
- **`api/dashboard/upload-video.js` accepts a filename query param and concatenates into the blob path without traversal protection** — Vercel Blob treats the path as an opaque key (not a filesystem path), so `..` does not escape. Admin-only endpoint anyway. Not a real bug.
- **`api/client/daily-log.js` clamps invalid integer ratings to 1 via `Math.max(1, Math.min(5, parseInt(x, 10) || 0))`** — quirky but the UI only sends 1-5, so unreachable in practice.
- **`api/cron/engagement-check.js` will re-email an inactive client every day until they log in** — could be spammy, but appears to be intentional design (the email subject is "We miss you", suggesting recurrence is the point). Not a bug without product input.
