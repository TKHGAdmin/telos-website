# Telos Bug Hunter Report — 2026-05-12

**Focus:** Functional
**Bugs found:** 2
**Approval requested:** Yes

## 1. Summary

First run of the bug hunter. Surveyed the codebase shape, then traced the core
functional paths: public submit endpoints (quiz, email, chs application),
admin-side CRUD (clients, pipeline, chs-applications, client-portal), client
session/auth, daily/training/nutrition/545 logging, the engagement and weekly
summary crons, and the protein/hyrox tool calculators.

Two functional bugs worth flagging. The most concrete is an orphaned Redis
lookup left behind when an admin deletes a client — it silently blocks
re-creating a client with the same email. The other is an input-validation
inconsistency in `/api/submit-quiz` that lets clearly invalid emails through
where the two sibling endpoints reject them.

Several documented backlog items (todayStr UTC bug, tool pages missing main.js,
incomplete VAPID push notifications) were observed but intentionally not
reported — they are already known per CLAUDE.md and prior bug crawls.

## 2. Bugs

#### BUG-2026-05-12-01 — `clients` DELETE leaves orphaned `client_email:` lookup

- **Severity:** P2
- **Area:** `api/dashboard/clients.js` (DELETE branch)
- **Status:** new

**What's wrong:**
When a coach deletes a client via the dashboard, the handler only removes
`client:{id}` and the `clients_index` membership. It does not remove the
`client_email:{normalizedEmail}` lookup that was written when portal access
was enabled. The orphaned lookup permanently blocks creating a new client
with the same email and breaks the "duplicate email check" contract documented
in CLAUDE.md ("CRUD: client roster + portal fields + duplicate email check").

**Where:**
- `api/dashboard/clients.js:143-154` (DELETE handler — no `client_email:` cleanup)
- Compare with `api/dashboard/client-portal.js:121-124` which does clean it up
  when the portal is disabled.
- Lookup write happens at `api/dashboard/client-portal.js:115-118`.

**Repro / evidence:**
1. Create client A with email `foo@example.com` via `POST /api/dashboard/clients`.
2. Enable portal access via `POST /api/dashboard/client-portal` with
   `{ clientId, enabled: true, password: '...' }` — this SETs
   `client_email:foo@example.com → A.id`.
3. Delete client A via `DELETE /api/dashboard/clients?id=A.id`.
   - `client:A.id` and `clients_index` entry are gone.
   - `client_email:foo@example.com` still points at the now-missing `A.id`.
4. Create new client B with email `foo@example.com` via `POST /api/dashboard/clients`.
   - `clients.js:70-73` reads `client_email:foo@example.com`, sees a value,
     returns 400 `Email already in use by another client`.

Secondary effect: if a coach also tries to log in as the deleted client (e.g.
during testing), the login endpoint will resolve the email lookup, fail to
GET `client:A.id`, and return `Invalid email or password` — confusing diagnostic
when the client was supposed to be fully gone.

**Suggested fix:**
In the DELETE branch, read the client record first, normalize its email, and
`DEL client_email:{normalized}` before deleting the main key. Mirror the
cleanup already done by the `client-portal` disable path. Optionally also
clean up the per-client portal data keys (`client_nutrition_plan:`,
`client_training_program:`, `client_545_*`, `client_sidemenu:`,
`client_supplement_plan:`, daily log ZSETs) if a hard delete is intended,
otherwise document that delete is metadata-only.

#### BUG-2026-05-12-02 — `submit-quiz` accepts emails with no `@` (validation inconsistency)

- **Severity:** P3
- **Area:** `api/submit-quiz.js`
- **Status:** new

**What's wrong:**
`/api/submit-quiz` only checks `if (!email || !name)`. It accepts any
non-empty `email` string, including obvious garbage like `"foo"` or
`"123"`. The other two public submit endpoints both validate format:

- `api/submit-email.js:18` — `if (!email || !email.includes('@'))`
- `api/submit-chs-application.js:35` — `/^[^\s@]+@[^\s@]+\.[^\s@]+$/`

Browser-side `type=email` + `required` catches most cases, but the API is
the source of truth and is the boundary that should enforce the contract.
Any client (including the homepage quiz on a misbehaving older browser,
or a scripted submission) can drop unparseable strings into Redis, where
they'll surface in the `/thomas` dashboard's Leads tab and any future
email-driven outreach.

**Where:**
- `api/submit-quiz.js:18` — `if (!email || !name) { return 400 }`
- Compare: `api/submit-email.js:18`, `api/submit-chs-application.js:35`.

**Repro / evidence:**
```
curl -X POST https://www.telosathleticclub.com/api/submit-quiz \
  -H "Content-Type: application/json" \
  -d '{"name":"x","email":"not-an-email","scores":[1,1,1,1],"totalScore":4}'
# → 200 OK, stored in `quiz_submissions` ZSET
```

**Suggested fix:**
Add an email-format check matching `submit-email.js` (minimum) or the
stricter `submit-chs-application.js` regex (preferred for consistency).
Single line, e.g.:
```js
if (!email || !/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email)) {
  return res.status(400).json({ error: 'Valid email required' });
}
```

## 3. Notes / Observations

- `api/cron/weekly-summary.js:92` divides the 7-day execution score sum by
  `execScores.length` (always 7). Days with no logs/tasks contribute 0, which
  drags the displayed "Avg Execution Score" down — a client who logged 2 of 7
  days at 50% each shows 14% in their weekly recap email. Could be intentional
  (encourages full-week logging) so not flagged as a bug, but worth a product
  call. Either drop empty days from the denominator, or rename the metric to
  "7-Day Score" to make the semantics explicit.
- `api/lib/auth.js:27` compares HMAC signatures with `!==` instead of
  `crypto.timingSafeEqual`. The sibling `client-auth.js:47-49` uses a
  timing-safe compare. Not exploitable in practice for HMAC verification of
  cookies (forging requires the secret), but worth aligning. Holding for the
  Security focus day rotation.
- Admin push notifications (`api/client/notify.js`) POST directly to push
  endpoints without VAPID-signed JWT auth or RFC 8291 payload encryption.
  Will return non-2xx from any compliant push service (FCM/Mozilla/Apple).
  This is already self-flagged in code comments as incomplete, so not a new
  finding.
