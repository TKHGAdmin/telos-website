## Bugs

### [P1] BUG-2026-05-20-01 — No rate limiting on client login endpoint

- **Where:** `api/client/login.js:9-60`
- **What:** The client portal login handler accepts unlimited password attempts per IP. The codebase already implements an `INCR ratelimit:{ns}:{ip}` + `EXPIRE 3600` pattern in `api/submit-quiz.js:22-29`, `api/submit-email.js:27-34`, and `api/submit-chs-application.js`, but `login.js` does not use it.
- **Repro:** `curl -X POST https://www.telosathleticclub.com/api/client/login -H 'content-type: application/json' -d '{"email":"victim@example.com","password":"X"}'` in a loop — every request is processed, no 429 ever returned.
- **Why it matters:** Credential stuffing and online password guessing against real client accounts are possible. PBKDF2 (100k iterations) slows each attempt but does not stop sustained automated guessing.
- **Suggested fix:** Mirror the rate-limit block from `submit-email.js`, keyed on `ratelimit:client-login:{ip}` with a tighter ceiling (e.g. 10/hr), and ideally a second counter keyed on the submitted email to slow targeted attacks.

### [P1] BUG-2026-05-20-02 — No rate limiting on admin dashboard login

- **Where:** `api/dashboard/login.js:3-17`
- **What:** The `/thomas` admin login handler validates the password and sets a session cookie with no rate limit, no lockout, and no logging on failures. A single env-var password protects the entire client roster, revenue data, and Charleston applications.
- **Repro:** Same as above against `/api/dashboard/login`. Each request runs `verifyPassword` (timing-safe but trivially fast) and returns 200/401.
- **Why it matters:** This is the highest-value credential in the system. Sustained online guessing is the realistic threat — there's no second factor and no monitoring.
- **Suggested fix:** Add IP-keyed rate limiting (e.g. 5 attempts per 15 minutes) using the same Redis counter pattern; consider a short-term IP lockout after N consecutive failures.

### [P2] BUG-2026-05-20-03 — Client login error messages enable email enumeration

- **Where:** `api/client/login.js:37`, `api/client/login.js:40`, `api/client/login.js:43`
- **What:** When the email exists but other gates fail, the response body differs from the generic "Invalid email or password" used elsewhere in the same handler — specifically `"Portal access not enabled"` (line 37), `"Account is not active"` (line 40), and `"Password not set - contact your coach"` (line 43). Compare with the generic message used at lines 26, 31, and 47.
- **Repro:** Submit login with a real client email and any password → response body reveals account state (`portal not enabled` vs `invalid email or password`).
- **Why it matters:** An attacker can enumerate which emails belong to Telos clients without ever guessing a password, then target those addresses with phishing or credential stuffing.
- **Suggested fix:** Collapse lines 36-44 to the same generic 401 returned at line 26. Surface the specific reason only after a valid password is presented, or only in admin-facing logs.

### [P2] BUG-2026-05-20-04 — Email-to-client lookup can be silently overwritten

- **Where:** `api/dashboard/clients.js:59-92`, `api/dashboard/client-portal.js:115-118`
- **What:** `clients.js` POST checks `client_email:{normalized}` for duplicates (line 70) but never writes that key — the lookup is only created when the coach enables a client's portal (`client-portal.js:117`). Two distinct clients can therefore be created with the same email (both portal-disabled), both pass the duplicate gate, and when the coach later enables portal access for the second client, line 117 does an unconditional `SET` that overwrites the first client's email→id mapping.
- **Repro:**
  1. `POST /api/dashboard/clients` with `{email:"foo@bar.com", ...}` → client A created, no email key.
  2. `POST /api/dashboard/clients` with `{email:"foo@bar.com", ...}` → client B created (duplicate check passes because no key exists).
  3. Enable portal for A via `POST /api/dashboard/client-portal` → `client_email:foo@bar.com = A.id`.
  4. Enable portal for B via same endpoint → `client_email:foo@bar.com = B.id`. A can no longer log in.
- **Why it matters:** Quietly breaks login for the older client with no error surfaced to the coach. Data integrity invariant ("email uniquely identifies a portal user") is not enforced.
- **Suggested fix:** In `client-portal.js`, before the SET on line 117, `GET` the existing value and reject if it points to a different client id. Also have `clients.js` POST refuse duplicate emails regardless of portal state.

### [P2] BUG-2026-05-20-05 — No rate limiting on password reset email request

- **Where:** `api/client/reset-password.js:20-68`
- **What:** The reset-request branch accepts any email and, if it matches an enabled portal client, sends a Resend email. There is no per-IP or per-email throttle.
- **Repro:** Loop `POST /api/client/reset-password` with `{email:"client@example.com"}` — each call queues a real email send.
- **Why it matters:** Two concrete impacts: (a) any third party who knows a client's email can spam their inbox with reset messages; (b) every send hits Resend's metered API, so an attacker can drive up the monthly bill cheaply.
- **Suggested fix:** Add the standard IP rate-limit block (e.g. 5/hr per IP) plus a per-email cap (e.g. 3/hr per normalized email) using a second Redis counter. Continue returning 200 in both cases to preserve the existing "don't leak whether the email exists" behavior.

### [P3] BUG-2026-05-20-06 — Admin session signature uses non-timing-safe comparison

- **Where:** `api/lib/auth.js:27`
- **What:** `verifyToken` compares the cookie HMAC against the expected value with `if (signature !== expected) return false`. The parallel implementation for client sessions in `api/lib/client-auth.js:46-49` correctly converts both sides to Buffers and uses `crypto.timingSafeEqual` — and `verifyPassword` in the same file (lines 56-58) also uses `timingSafeEqual`.
- **Repro:** Code citation — line 27 of `auth.js` vs lines 46-49 of `client-auth.js`.
- **Why it matters:** Low real-world risk (remote timing attacks against an HMAC-SHA256 output over the public internet are nearly infeasible), but the unsafe pattern is inconsistent with the rest of the codebase and is the kind of finding any future security audit will flag.
- **Suggested fix:** Replace the `!==` check with the same `Buffer.from(...)` + `timingSafeEqual` block already used in `client-auth.js:46-49`.
