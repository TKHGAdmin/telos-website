## TELOS-2026-05-01-001 (P1, Functional)

**Title:** Quiz allows rapid double-tap to inflate scores and skip questions; can crash results screen

**Where:** `js/quiz.js:198-218` (`handleAnswerSelect`)

**Reproduction:**
1. Open the homepage Execution Score Quiz (or `/pricing` overlay).
2. On any question, tap two different option buttons within 400 ms (easy on mobile).
3. Both options' points are added to the same pillar; both auto-advance timeouts fire, so the next question is also skipped past.
4. With enough rapid taps, the total score can exceed 40 (the max valid score). When `getResultTier(totalScore)` runs at `quiz.js:284-286` it returns `undefined`; the next line `tier.name` (line 325) throws `TypeError: Cannot read properties of undefined (reading 'name')` and the results screen never renders.

**Why it's a bug:**
- `handleAnswerSelect` does not disable the buttons or set an "answered" flag. Each click both mutates `quiz.scores[pillar] += option.points` and schedules an independent 400 ms `setTimeout` that increments `quiz.currentQuestion`.
- Rapid double-taps are common on mobile (tap latency causes users to retap).
- The same defect skips a question because every queued timeout advances `currentQuestion`.

**Suggested fix:**
- Add a guard variable (`var advancing = false;`) that is set on first tap and gates further work in `handleAnswerSelect`. Reset it inside the auto-advance timeout. Optionally also disable all `.quiz-option` buttons after a click (`button.disabled = true` on siblings).

**Confidence:** High. Code inspection confirms no debounce; behavior is deterministic.

## TELOS-2026-05-01-002 (P1, Functional/Auth)

**Title:** Disabling a client's portal in `/thomas` does not revoke active client sessions

**Where:** All endpoints under `api/client/*.js` that call `verifyClientSession` (e.g. `daily-log.js:5-8`, `training-log.js:5-8`, `nutrition-log.js`, `five-four-five.js`, `activity-log.js`, `me.js`, etc.)

**Reproduction:**
1. As admin, enable portal for a client and have them log in (cookie issued, 7-day expiry).
2. As admin, in `/thomas` Client Portal tab, toggle `enabled` off (`api/dashboard/client-portal.js` POST sets `client.portalEnabled = false` and removes `client_email:` lookup).
3. The client's existing `telos_client_session` cookie remains valid because token verification is purely cryptographic (`api/lib/client-auth.js:34-52`). The client can keep hitting `GET /api/client/daily-log`, `POST /api/client/training-log`, etc. for up to 7 days.

**Why it's a bug:**
- `login.js:36-44` correctly blocks login when `portalEnabled === false`. But every other client endpoint only calls `verifyClientSession`, which returns `clientId` from a signed cookie without consulting Redis. There is no per-request enforcement of `portalEnabled`/`status`.
- A deleted client (admin DELETE in `clients.js:143-154`) has the same problem: their cookie still verifies and they can still POST to `client_dailylog:{deletedId}:{date}`, leaving orphan data in Redis.

**Suggested fix:**
- In `verifyClientSession` (or in a small wrapper used by every client handler), after recovering `clientId`, fetch `client:{clientId}` and reject if missing, `portalEnabled === false`, or `status !== 'active'`. Cache or accept the extra Redis GET per request - most client endpoints already do at least one Redis call.

**Confidence:** High. Verified by reading all 21 client handlers - none recheck portal/status.

## TELOS-2026-05-01-003 (P2, Functional)

**Title:** Daily engagement-check email spams inactive clients indefinitely (no cooldown / no last-sent tracking)

**Where:** `api/cron/engagement-check.js:36-94`

**Reproduction:**
1. A portal-enabled active client stops checking in for 3+ days.
2. The daily cron at 3pm UTC sends a "We miss you at Telos" email.
3. The next day (4 days inactive) it sends again. Day 5: again. Days 6, 7, ... 30: still sends every day, because the function only looks at the last 5 daily logs and `lastLogDaysAgo === -1` always triggers `daysSince = 5` and a send. There is no `last_engagement_email_sent:{clientId}` key or check against it.

**Why it's a bug:**
- A client who lapses for a week receives 5+ identical emails titled "We miss you at Telos." This pattern is classic ESP-flagging behavior and trains clients to mark Telos email as spam. CLAUDE.md describes the cron's intent as "send a reminder email to clients inactive 3+ days," not "send daily forever."
- Easy to fix: write `client_engagement_last_sent:{clientId}` with a TTL (e.g. 7 days) and skip if present.

**Suggested fix:**
- After successful send, `await redis('SETEX', 'client_engagement_last_sent:' + client.id, 7 * 86400, '1');` and check this key before sending. Or only send when `lastLogDaysAgo === 3` (boundary day) instead of `>= 3`.

**Confidence:** Medium-high. Behavior is clear from the code; severity depends on how many lapsed clients exist (operationally important when client count grows).

## TELOS-2026-05-01-004 (P2, Functional)

**Title:** Duplicate email check on client creation is bypassable for non-portal clients

**Where:** `api/dashboard/clients.js:68-73` (POST) and `client-portal.js:114-118` (where `client_email:` is actually written)

**Reproduction:**
1. As admin, create client A with email `foo@bar.com`. Do NOT enable their portal.
2. The handler runs `existingEmailOwner = await redis('GET', 'client_email:foo@bar.com')` and finds nothing - that lookup key is only ever written when the portal is enabled in `client-portal.js`.
3. Create client B with the same email `foo@bar.com`. The check still returns nothing → POST succeeds. Two distinct client records now share the same email.
4. Later, enable portal on client A → `client_email:foo@bar.com → A`. Then enable portal on client B → the same key is overwritten to point to B. Client A's portal access (and any password reset emails sent to that address) silently maps to client B.

**Why it's a bug:**
- The "duplicate email check" advertised in CLAUDE.md (`# CRUD: client roster ... + duplicate email check`) is only enforced when at least one of the two existing clients has portal access. It silently fails for the common "create now, enable portal later" workflow.
- The PUT handler at `clients.js:121-132` repeats the same conditional logic - the email lookup is only updated if `client.portalEnabled` is true.

**Suggested fix:**
- Maintain a separate index `client_email_all:{normalizedEmail} → clientId` that is set on every create/email-change regardless of portal state, and check that index in POST/PUT. Keep the existing `client_email:` for portal login resolution if you want portal-only semantics there, or unify them.

**Confidence:** High. Code paths confirm the lookup is only written when `portalEnabled === true`.
