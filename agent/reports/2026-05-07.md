# Telos Bug Hunt - 2026-05-07

**Focus**: Security
**Bugs found**: 5
**Summary**: First run of the agent. Both admin and client login endpoints accept unlimited password attempts, and the password-reset email trigger is unauthenticated and unrate-limited. The admin session HMAC compare is not constant-time, and the public submission endpoints' rate limiter is bypassable by spoofing `X-Forwarded-For`.

---

## Bugs

### [P1] Client login has no rate limiting - online brute-force is unrestricted

**ID**: `2026-05-07-01`
**File(s)**: `api/client/login.js:9-60`
**Category**: Auth

**What's wrong**
The client login handler accepts unlimited POST attempts per email. Client passwords have an 8-character minimum (no complexity rule beyond that), so an attacker who knows or guesses an active client email can hammer common passwords without ever being slowed down. The handler also doesn't normalize timing between "no such email" and "wrong password," so user enumeration may be possible on top of the brute-force surface.

**Reproduction**
1. Read `api/client/login.js` - no `ratelimit:` key is touched.
2. Compare to `api/submit-quiz.js:24-29`, which keys an Upstash `INCR`/`EXPIRE` per IP. The login handler has no equivalent.
3. POST any number of `{ email, password }` payloads in a loop; only Upstash request limits would slow you down.

**Suggested fix**
Add the same per-IP Redis rate limit pattern used in `submit-quiz.js`, e.g. `ratelimit:client_login:${ip}` with a low ceiling (5 failures / 15 min). Optionally add a per-email counter so distributed-IP brute force on a single account is also bounded. Use `req.headers['x-real-ip']` rather than the raw `x-forwarded-for` header for the key (see bug 2026-05-07-05).

### [P1] Admin login has no rate limiting - single password protects entire dashboard

**ID**: `2026-05-07-02`
**File(s)**: `api/dashboard/login.js:3-17`
**Category**: Auth

**What's wrong**
`/api/dashboard/login` is a single shared password for the entire admin dashboard (no username, no 2FA). The handler does a `verifyPassword` and immediately returns - with no rate limiting, an attacker can run an unlimited online dictionary against `DASHBOARD_PASSWORD`. Compromise of this single secret grants full read/write on every Redis key, including all client PII, payment-tier data, and the ability to set arbitrary HTML in side-menu panels rendered to clients (see bug 2026-05-07-04).

**Reproduction**
1. Read `api/dashboard/login.js` - no rate-limit logic.
2. Send `POST /api/dashboard/login` with `{ password: "guess" }` in a loop. Only Upstash and Vercel function-invocation limits apply.

**Suggested fix**
Add a per-IP Redis rate limit (e.g. 5 failures / 15 min, then 1 hour lockout) before calling `verifyPassword`. If feasible, also add an absolute per-IP cap (e.g. 50/day) so distributed attempts are slowed too. Long-term, layer in a TOTP second factor since this is the only thing standing between an attacker and the entire client database.

### [P2] Password-reset email endpoint is an unauthenticated email-spam / enumeration trigger

**ID**: `2026-05-07-03`
**File(s)**: `api/client/reset-password.js:20-67`
**Category**: Auth

**What's wrong**
The "request reset email" branch (`body.email && !body.token`) has no rate limit. Anyone can POST `{ email: "<known-client-email>" }` repeatedly and Resend will fire one email per call, creating an email-bomb vector against any known client address and burning through the Resend quota. The handler correctly returns `{ ok: true }` for unknown emails to prevent enumeration, but a side-channel still exists: known emails take longer (extra Redis GET + Resend round-trip) than unknown ones.

**Reproduction**
1. Read `api/client/reset-password.js:20-67`.
2. POST `/api/client/reset-password` with `{ "email": "victim@example.com" }` in a loop. Each call sends another reset email if the address is registered; quota is uncapped.

**Suggested fix**
Add a per-IP rate limit (3 reset requests / hour) and a per-email rate limit (1 reset email per 5 minutes), both keyed in Redis with `INCR`/`EXPIRE`. Always perform the same number of Redis ops regardless of whether the email exists, to flatten the timing signal.

### [P3] Admin session HMAC verification is not constant-time

**ID**: `2026-05-07-04`
**File(s)**: `api/lib/auth.js:18-31`
**Category**: Auth

**What's wrong**
`verifyToken` compares the signature with `if (signature !== expected) return false;` (line 27). String `!==` short-circuits on first byte mismatch, leaking byte-level timing info about the expected HMAC. The client-auth equivalent in `api/lib/client-auth.js:46-49` correctly uses `crypto.timingSafeEqual`, so this is an inconsistency, not a paradigm. Practical exploitability over the public internet is very low (network jitter swamps per-byte timing), but the fix is one line and brings admin auth to parity with client auth.

The `verifyPassword` helper (lines 52-59) also returns early on length mismatch, which leaks the exact length of `DASHBOARD_PASSWORD`. Same low-impact, easy-fix shape.

**Reproduction**
1. Open `api/lib/auth.js`, read line 27 - bare `!==`.
2. Open `api/lib/client-auth.js`, read lines 46-49 - this code path uses `Buffer.from(...,'hex')` + `crypto.timingSafeEqual`. That's the pattern admin auth should match.

**Suggested fix**
Replace lines 23-30 with:
```js
const expected = crypto.createHmac('sha256', SESSION_SECRET).update(payload).digest('hex');
const a = Buffer.from(signature, 'hex');
const b = Buffer.from(expected, 'hex');
if (a.length !== b.length) return false;
if (!crypto.timingSafeEqual(a, b)) return false;
```
For `verifyPassword`, pad both buffers to a fixed length (e.g. 64 bytes) before `timingSafeEqual` so the early length check can be removed.

### [P2] Rate-limit keys use raw `x-forwarded-for`, which is client-controllable on Vercel

**ID**: `2026-05-07-05`
**File(s)**: `api/submit-quiz.js:23`, `api/submit-email.js:28`, `api/submit-chs-application.js:40`
**Category**: Rate Limit

**What's wrong**
All three public endpoints build the rate-limit Redis key with `req.headers['x-forwarded-for'] || req.headers['x-real-ip'] || 'unknown'`. On Vercel, when a client sends their own `X-Forwarded-For` header, it is appended to the chain set by Vercel's edge - so `req.headers['x-forwarded-for']` returns something like `attacker-supplied-ip, real-client-ip`. Because this entire string is used as the rate-limit key, an attacker can cycle the leading value (`1.1.1.1`, `1.1.1.2`, ...) and reset their counter on every request, defeating the 10/hr (quiz, email) and 5/hr (chs-applications) caps. Vercel exposes the actual client IP via `x-real-ip` (a single value), which the code only reaches as a fallback.

**Reproduction**
1. Read `api/submit-quiz.js:23` - the OR chain prefers `x-forwarded-for`.
2. From any client, send 11+ requests to `/api/submit-quiz` while rotating `X-Forwarded-For: 1.1.1.${i}`. Each request gets a new Redis key and bypasses the limit.
3. Same trick on `/api/submit-email` and `/api/submit-chs-application`.

**Suggested fix**
Prefer `x-real-ip` (single, set by Vercel) for the rate-limit key, falling back to the *last* token of `x-forwarded-for` (the closest hop, which on Vercel is the trustworthy one). Concretely:
```js
const xff = req.headers['x-forwarded-for'] || '';
const ip = req.headers['x-real-ip']
  || xff.split(',').map(s => s.trim()).filter(Boolean).pop()
  || 'unknown';
```
Apply identically to all three submit endpoints (and any future rate-limited endpoint - good candidate for a shared helper in `api/lib/`).

## Notes

This is the first run, so there is no prior decisions history to calibrate against. I leaned conservative: every finding above is reproducible from the code as written and would actually affect a real attacker scenario. I deliberately did not include several lower-confidence items I considered (admin-set HTML rendered via `innerHTML` in client side-menu custom panels, custom email template accepting raw HTML body, `upload-video.js` accepting unsanitized filename in blob path) because they all require an already-compromised admin account, which is itself the bigger story (bug 2026-05-07-02). If P1 admin login hardening lands, those become more interesting to revisit.

No committed secrets found in `git log -p` scan. Cron endpoints correctly check `Bearer ${CRON_SECRET}` and fail closed when unset.
