## BUG-2026-05-02-01 — P1 — Quiz lead capture bypassable; pricing gate defeated

**Where**
- `js/quiz.js` lines 220-276 (`setupLeadCapture`)
- `index.html` line 578 + `pricing.html` line 593 (`<button type="button" class="quiz-see-score-btn">`)

**What**
The "See Your Score" button is `<button type="button">`, so the form's HTML5 `required` attributes on the name/email inputs do nothing. The click handler also performs no validation — it reads the input values, fires `fetch('/api/submit-quiz')` with whatever (including empty strings), then unconditionally:

1. Sets `localStorage.telosQuizCompleted = 'true'`
2. Hides the lead-capture step and renders results

**Reproduction**
1. Open https://www.telosathleticclub.com/pricing
2. Take the gating quiz (any answers — auto-advances)
3. On the "Your Score Is Ready" step, leave name + email + phone blank
4. Click **See Your Score**
5. Observed: results render, `localStorage.telosQuizCompleted === 'true'`, then the **View Pricing** button reveals the full pricing page. No data was captured.

The API also rejects the submission (`400 Name and email required`), but the `fetch(...).catch(function(){})` swallows the error and the UI advances regardless.

**Impact**
This defeats the entire purpose of the pricing gate (`pricing.html`) — visitors can unlock prices and skip lead capture. On the homepage it silently drops a real lead each time someone clicks through. Given the quiz is the primary top-of-funnel conversion mechanism, the lost-lead rate is potentially significant.

**Suggested fix**
In `js/quiz.js` `setupLeadCapture()` add a guard before the fetch / localStorage / render block:

```js
if (!quiz.userName.trim() || !quiz.userEmail.includes('@')) {
  // surface a visible error and return early
  return;
}
```

Optionally also flip the buttons to `type="submit"` and bind to `form.addEventListener('submit', ...)` so HTML5 validation triggers natively (matches the pattern used by `protein-calculator.html` line 643 / `chs.html` line 1100).

**Confidence:** High — verified by reading both HTML and JS; no env-dependent variables.

## BUG-2026-05-02-02 — P1 — Engagement-check cron sends duplicate "we miss you" emails every day

**Where**
- `api/cron/engagement-check.js` lines 60-94
- Cron schedule in `vercel.json` line 25 (`0 15 * * *` — daily)

**What**
The cron checks each active client's last 5 days of `client_dailylog:*` entries. If they haven't logged for 3+ days, it sends a "We miss you at Telos" email via Resend. **There is no record of the last reminder send and no rate limit between sends.** The cron runs daily, so an inactive client receives this email every single day until they log.

```js
if (lastLogDaysAgo === -1 || lastLogDaysAgo >= 3) {
  // ...build email...
  await fetch('https://api.resend.com/emails', { ... });   // sends EVERY day
}
```

**Reproduction**
1. Pick any active client with `portalEnabled: true` and `emailEnabled !== false`.
2. Ensure no `client_dailylog:{id}:{date}` entries exist for the last 3+ days.
3. Trigger the cron manually with the configured `CRON_SECRET` (or wait 24h).
4. Observed: Resend dispatch on day 3, day 4, day 5, day 6… with the day-count rendered identically (capped at "5+ days" — see `daysSince = lastLogDaysAgo === -1 ? 5 : lastLogDaysAgo`).

**Impact**
Paying clients receive a daily "we miss you" guilt-trip email while inactive. High likelihood of:
- Spam complaints / Resend deliverability hit on the `noreply@telosathleticclub.com` sending domain
- Client unsubscribes or churn requests
- Damaged trust ("Telos is spamming me")

This is a behavior bug, not just polish — the intent is clearly "send a reminder when they go inactive," not "harangue them every 24h."

**Suggested fix**
Track a `lastEngagementEmailAt` timestamp on the client record and skip if it was sent within the last N days (e.g., 4-7 days). Sketch:

```js
var lastSent = client.lastEngagementEmailAt ? new Date(client.lastEngagementEmailAt).getTime() : 0;
var FOUR_DAYS = 4 * 86400000;
if (now - lastSent < FOUR_DAYS) continue;
// ...send...
client.lastEngagementEmailAt = new Date().toISOString();
await redis('SET', 'client:' + client.id, JSON.stringify(client));
```

**Confidence:** High — control flow is unambiguous; cron schedule confirmed in `vercel.json`.

## BUG-2026-05-02-03 — P2 — Deleting a client orphans the email lookup, blocks re-onboarding with same email

**Where**
- `api/dashboard/clients.js` DELETE handler, lines 143-154
- Email lookup write happens in `api/dashboard/client-portal.js` line 117

**What**
When a coach deletes a client via the admin dashboard, the DELETE handler removes only:

```js
await redis('DEL', 'client:' + id);
await redis('ZREM', 'clients_index', String(id));
```

It does **not** clean up the `client_email:{normalizedEmail}` lookup that was created when the portal was enabled. The orphaned lookup still points to the now-deleted client ID.

The POST handler (line 70-73) then rejects any future creation attempt for the same email:

```js
var existingEmailOwner = await redis('GET', 'client_email:' + normalized);
if (existingEmailOwner) {
  return res.status(400).json({ error: 'Email already in use by another client' });
}
```

So the email is permanently "burned" from the coach's perspective, with no recovery path through the dashboard.

**Reproduction**
1. In `/thomas` → Clients tab, create client A with email `test@example.com` and tier `growth`.
2. In Client Portal sub-tab, enable portal for client A (this writes `client_email:test@example.com → A`).
3. Delete client A from the Clients tab.
4. Try to re-create a client with email `test@example.com`.
5. Observed: API returns `400 Email already in use by another client`. There is no UI affordance to clear the orphaned key.

**Impact**
Coach-facing edge case but breaks legitimate workflows (re-onboarding a returning client, fixing a typo by delete-and-recreate, cleaning up a test client). Recovery requires direct Redis access — Thomas would need to manually `DEL client_email:...`.

There are also other orphaned keys after delete (daily logs, training program, supplement plan, side menu, 545 data, push subscriptions, etc.) — those are pure data accumulation, not user-facing functional breaks. Calling those out as P3 tech debt rather than separate bugs to keep this report focused.

**Suggested fix**
In the DELETE handler, fetch the client first and clean up the email lookup if portal was enabled:

```js
var existing = await redis('GET', 'client:' + id);
if (existing) {
  try {
    var c = JSON.parse(existing);
    if (c.email) {
      var { normalizeEmail } = require('../lib/client-auth');
      await redis('DEL', 'client_email:' + normalizeEmail(c.email));
    }
  } catch (e) {}
}
await redis('DEL', 'client:' + id);
await redis('ZREM', 'clients_index', String(id));
```

(Optionally extend to wipe all `client_*:{id}*` keys; that's bigger scope and worth a separate decision.)

**Confidence:** High — verified both code paths and the duplicate-check gate.
