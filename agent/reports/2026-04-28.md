## BUG-2026-04-28-01 — Push notifications return 200 OK but never deliver

- Severity: **P1**
- Area: API / Push notifications
- File: `api/client/notify.js:62-76`

**What's wrong**

`api/client/notify.js` POSTs a plaintext JSON payload directly to `subscription.endpoint` (the browser's Web Push service: FCM for Chrome, Mozilla Autopush for Firefox, Apple/Edge endpoints, etc.) with only `Content-Type` and `TTL` headers:

```js
var pushRes = await fetch(subscription.endpoint, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json', 'TTL': '86400' },
  body: payload
});
```

This violates the Web Push protocol (RFC 8030 + RFC 8291):

1. The push service requires a **VAPID JWT** in the `Authorization` header signed with the VAPID private key (or the legacy `Crypto-Key` header). Without it, FCM and Mozilla Autopush return 401.
2. When delivering a payload, the body must be **encrypted** per RFC 8291 (ECDH key agreement with `subscription.keys.p256dh`, HKDF, AES-128-GCM, with `Content-Encoding: aes128gcm`). A plaintext JSON body is rejected (400) and never decrypts client-side even if accepted.

The handler then masks the failure: at line 71-76 it treats any non-OK response as success and returns `200 ok: true` with a `warning` field that the dashboard caller doesn't display. The earlier branch at line 41-45 also returns `200 ok: true` when VAPID keys aren't configured. So from the admin's perspective, "Send notification" always succeeds — but the client never gets a notification.

The CLAUDE.md "Known Limitations" section does not list push notifications as broken, and the PWA service worker (`sw.js`) does register a `push` event handler that expects to fire, suggesting this is intended to work.

**Repro**

1. Have a client register a push subscription via the PWA.
2. From `/thomas`, trigger a push send to that client (POST `/api/client/notify`).
3. Observe: API returns 200 OK; nothing arrives on the client device.
4. If you log `pushRes.status` from line 71, expect 401 (VAPID missing) or 400 (encryption missing).

**Fix sketch**

Either:
- Add the `web-push` npm package (the comment at line 56 notes this is the right path) and use `webpush.sendNotification(subscription, payload)` with VAPID credentials. Update the handler to surface real failures (4xx/5xx → `res.status(502)`), and remove the "always 200 OK" fallback so admins see when delivery fails.
- Or, until VAPID + encryption are wired up, return a clear `503 not_implemented` instead of `200 ok: true` so the admin UI doesn't lie about delivery.

## BUG-2026-04-28-02 — Deleting a client orphans the email lookup, blocking re-use of that email

- Severity: **P2**
- Area: API / Admin / Client lifecycle
- File: `api/dashboard/clients.js:143-153` (DELETE handler)

**What's wrong**

When a client with portal access is created, `api/dashboard/client-portal.js:115-118` writes a unique-email lookup:

```js
await redis('SET', 'client_email:' + email, String(client.id));
```

That key is the source of truth for the "email already in use" check at `clients.js:70-73` and for the login email lookup at `api/client/login.js:24`.

The DELETE handler in `clients.js`:

```js
await redis('DEL', 'client:' + id);
await redis('ZREM', 'clients_index', String(id));
```

…only removes the client record and index entry. It leaves behind:

- `client_email:{normalizedEmail}` (orphaned uniqueness record)
- All per-client log keys: `client_dailylog:{id}:{date}`, `client_nutrition_log:*`, `client_training_log:*`, `client_545_*`, `client_supplement_*`, `client_activity_*`, plus their `*_index:{id}` ZSETs.
- `client_nutrition_plan:{id}`, `client_mindset:{id}`, `client_resources:{id}`, `client_training_program:{id}`, `client_sidemenu:{id}`, `client_supplement_plan:{id}`, `client_push_sub:{id}`, etc.

User-facing impact (email lookup, the worst part): if Thomas deletes a former client and later tries to onboard them again (or onboard someone else with the same address), POST `/api/dashboard/clients` returns `400 "Email already in use by another client"` and there is no in-app way to clear it — Thomas has to drop into Upstash and `DEL` the key manually.

The remaining stale keys are mostly silent storage bloat, but a coincident `id` collision (via `INCR id:clients` after a Redis flush or restore) could surface a deleted client's old logs to a brand-new client.

**Repro**

1. POST `/api/dashboard/clients` with `{name:'A', email:'a@x.com', tier:'rebuild'}` → returns id N.
2. POST `/api/dashboard/client-portal` with `{clientId:N, enabled:true, password:'12345678'}` → email lookup is set.
3. DELETE `/api/dashboard/clients?id=N`.
4. POST `/api/dashboard/clients` with `{name:'A2', email:'a@x.com', tier:'rebuild'}` → expect 200, observe **400 "Email already in use by another client"**.

**Fix sketch**

In the DELETE handler, before removing the client record, GET it, parse out `email`, and if `portalEnabled || email`, also issue:

```js
await redis('DEL', 'client_email:' + normalizeEmail(client.email));
```

A separate sweep for the per-client log keys is nice-to-have but optional (they'll never be read once the id is gone). The email lookup is the only cleanup that has a user-facing failure mode.

## BUG-2026-04-28-03 — Weekly recap email reports streak as 0 days for almost every client

- Severity: **P2**
- Area: Cron / Outbound email
- File: `api/cron/weekly-summary.js:86-90`

**What's wrong**

The streak loop:

```js
for (var d = 0; d < 7; d++) {
  if (weekData[0][d] && weekData[0][d].result) streak++;
  else break;
}
```

`weekData[0][d]` is the daily log for `d` days ago. `d=0` is **today**. The cron runs Monday at 14:00 UTC (~9am ET). At that time virtually no client has logged "today" (Monday) yet — they get the email *before* their morning check-in. The loop sees today missing, breaks, and reports `streak = 0`, regardless of how many consecutive days the client logged Sun-Sat-Fri-…

Same shape applies in `engagement-check.js`, but its logic uses `lastLogDaysAgo` and is correct — only the weekly recap is affected.

User-facing impact: every Monday morning, every active portal client sees "Current Streak: 0 days" in their recap, even those with multi-week streaks. It actively contradicts what they see in the dashboard, undermining trust in the email.

**Repro**

1. Pick any active portal client with a daily log on Sunday but not yet on Monday.
2. Manually invoke `/api/cron/weekly-summary` with the cron auth header.
3. Inspect outbound email or `console.log` the computed `streak` — it will be 0.
4. Re-run after that client posts Monday's log → streak jumps to the correct value.

**Fix sketch**

Skip today (start at `d = 1`) when computing streak, or treat "no log yet today" as a non-break and start counting from yesterday:

```js
var startDay = weekData[0][0] && weekData[0][0].result ? 0 : 1;
for (var d = startDay; d < 7; d++) {
  if (weekData[0][d] && weekData[0][d].result) streak++;
  else break;
}
```

Bonus: the streak is capped at 7 days because only 7 days of logs are fetched. If Thomas wants the email to reflect true streaks (e.g. "23 days"), pull from `client_dailylogs_index:{id}` ZSET instead of the 7-day window.
