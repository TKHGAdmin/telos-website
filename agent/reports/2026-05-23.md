## BUG-20260523-01 | P2 | Deleting a client leaves orphan email lookup that blocks re-onboarding

**Severity:** P2
**Area:** admin dashboard - client management
**File(s):** api/dashboard/clients.js:142-154

### Symptom
After a coach deletes a client whose portal was enabled, attempting to create a new client with the same email address fails with "Email already in use by another client" - even though no client with that email exists. The only way to recover is to manually purge the orphaned `client_email:{normalized}` key from Redis.

### Reproduction
1. In the admin dashboard, create a client at jane@example.com.
2. Open Client Portal sub-tab, enable portal access (this writes `client_email:jane@example.com -> {id}`).
3. Delete that client from the Clients tab.
4. Try to re-create a client at jane@example.com.
5. POST `/api/dashboard/clients` returns 400 "Email already in use by another client".

### Root cause
`clients.js:147` only deletes `client:{id}` and removes the ID from `clients_index`. The email lookup key `client_email:{normalizedEmail}`, which was written by `client-portal.js:117` when the portal was enabled, is never cleaned up. The POST handler at `clients.js:70-73` then sees that key and refuses creation. Related orphans never cleaned by DELETE: `client_dailylog:*`, `client_training_log:*`, `client_545_*`, `client_nutrition_*`, `client_sidemenu:*`, `client_supplement_*`, `client_activity_log:*`, plus their `_index` ZSETs.

### Suggested fix
In the DELETE branch of `clients.js`, fetch the client first, then DEL `client_email:{normalizeEmail(client.email)}` alongside `client:{id}` and `ZREM clients_index`. For full hygiene, also DEL the per-client log/plan/index keys (those that exist for that client).

### Confidence
high - traced the read path (`POST /api/dashboard/clients` line 70) directly against the write path (`POST /api/dashboard/client-portal` line 117) and the delete path (line 147). No code touches `client_email:*` during deletion.

## BUG-20260523-02 | P2 | Inactive-client engagement email re-sends every single day with no de-duplication

**Severity:** P2
**Area:** cron jobs - email notifications
**File(s):** api/cron/engagement-check.js:36-95

### Symptom
A client who has not checked in for 3+ days receives an identical "We miss you at Telos" email *every single day* until they log in again. There is no cooldown, no per-client "last reminder sent" tracking, and no escalation backoff. A client on vacation for two weeks would receive ~12 identical reminders.

### Reproduction
1. Active client with `portalEnabled=true`, `emailEnabled !== false`, last check-in 4 days ago.
2. Vercel cron triggers `/api/cron/engagement-check` at 15:00 UTC.
3. Loop at `engagement-check.js:36` finds them, condition at line 61 (`lastLogDaysAgo >= 3`) matches, email sent.
4. Tomorrow at 15:00 UTC: same client is still inactive (5 days now), same condition matches, email sent again.
5. Repeat indefinitely. `grep` across the codebase for `lastReminder`, `reminderSent`, `emailSent`, `lastEngagement` returns zero hits, confirming no state is tracked.

### Root cause
The cron only inspects the daily-log presence. After sending, it never records that an email was sent for this client. The CLAUDE.md doc says "Daily: reminder email to clients inactive 3+ days" - but the intent surely was "check daily, send when newly inactive," not "spam daily forever."

### Suggested fix
Before sending, check a per-client cooldown key (e.g. `client_engagement_last:{id}` with a 7-day TTL). After a successful send, `SETEX client_engagement_last:{id} 604800 {iso-date}`. Skip the client when the key exists. Optionally escalate (3 days, then nothing for a week; 14 days, second reminder; etc.).

### Confidence
high - read the full cron handler and confirmed by grep that no last-sent timestamp is stored or checked anywhere in the repo.

## BUG-20260523-03 | P3 | Quiz submission endpoint accepts any string as the email address

**Severity:** P3
**Area:** public API - lead capture
**File(s):** api/submit-quiz.js:16-20

### Symptom
A quiz submission with `email: "not-an-email"` (or `email: "asdf"`, or `email: " "`) is accepted and stored as a real lead. This pollutes the lead pipeline in the admin dashboard with un-followable contacts and inflates the lead-count metric.

### Reproduction
```
curl -X POST https://www.telosathleticclub.com/api/submit-quiz \
  -H 'Content-Type: application/json' \
  -d '{"name":"Test","email":"not-an-email","scores":[5,5,5,5],"totalScore":20}'
```
Returns `{"ok":true}` and the entry shows up in `/thomas` → Leads tab with email "not-an-email".

### Root cause
`submit-quiz.js:18` only checks truthiness: `if (!email || !name)`. Compare with `submit-chs-application.js:35` which correctly applies `/^[^\s@]+@[^\s@]+\.[^\s@]+$/`. The quiz endpoint was either an early implementation or copy-pasted before the regex check existed.

### Suggested fix
Add the same regex sanity check used by the CHS application: `if (!/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email)) return res.status(400)...`. Trim the email first for consistency with the CHS endpoint. Same fix is worth applying to `submit-email.js:18`, which only checks `email.includes('@')` (would accept `"@"` or `"a@b"`).

### Confidence
high - reproducible, the format check exists elsewhere in the same project so this is a clear miss rather than a deliberate design choice.
