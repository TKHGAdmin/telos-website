## Bugs

### BUG-20260422-01 | P2 | Deleting a client orphans the `client_email` lookup, blocking re-creation with the same email

- **Severity:** P2
- **Area:** admin
- **File(s):** `api/dashboard/clients.js:143`
- **Status:** new

**Description**

The DELETE branch of `api/dashboard/clients.js` removes only `client:{id}` and the `clients_index` ZSET entry. It never deletes the `client_email:{normalizedEmail}` lookup that `api/dashboard/client-portal.js:117` creates when the portal is enabled. That lookup is the uniqueness index consulted by the create path at `api/dashboard/clients.js:70`, so after deletion the email stays reserved. Admin recreation of a client with the same email fails with "Email already in use by another client," pointing at a client that no longer exists. Also leaks stale pointers visible to client login (`api/client/login.js:24`) and password reset (`api/client/reset-password.js:22`) — both fail safely, but the admin-side block is confusing and has no in-product workaround.

**Reproduction**

1. Create a client in `/thomas` -> Clients with email `foo@example.com`.
2. Open the Client Portal sub-tab for that client, enable portal, set a password. (This calls `POST /api/dashboard/client-portal` which SETs `client_email:foo@example.com`.)
3. Delete that client from the Clients table (`DELETE /api/dashboard/clients`).
4. Try to create a new client with the same email `foo@example.com`.
5. **Observed:** 400 response `{"error": "Email already in use by another client"}`. **Expected:** The new client should be created successfully because no active client holds that email.

**Suggested fix**

In the DELETE branch of `api/dashboard/clients.js`, read the client record before deletion and, if it has `portalEnabled` and `email`, also `redis('DEL', 'client_email:' + normalizeEmail(client.email))`. Consider whether you also want to cascade-cleanup the per-client keys documented in `CLAUDE.md` (daily logs, training logs, 545 state, supplement plan/logs, activity logs, push subscription, module progress, watching/completed indexes) - right now all of these orphan in Redis forever and count against storage. If a soft-delete (status: 'archived') is preferable to hard delete, at minimum still drop the `client_email` entry so the email can be reused.
