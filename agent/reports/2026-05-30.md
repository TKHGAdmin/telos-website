# Bug Report - 2026-05-30

**Focus area today:** Functional

## Summary

First run of the Telos Bug Hunter. Audited the public-marketing flows (quiz, CHS application, shop checkout) and the API/cron layer. Two reproducible logic bugs found - both ride existing, deployed code paths and would affect real users.

## Bugs

### Bug 1 - P2 - Quiz lead-capture submits twice (and counting) after a retake

**ID:** 2026-05-30-1
**Severity:** P2
**Area:** Quiz (homepage + pricing page gate)
**File:** `js/quiz.js:221-276` (`setupLeadCapture`), called from `js/quiz.js:215`

**What's wrong**

`setupLeadCapture()` runs every time the user finishes question 8 and adds a click listener to the same `.quiz-see-score-btn` button. The retake flow (`resetQuiz` at line 416) clears state but does not remove or rebuild the button, so a new listener is appended on each retake. Same accumulation happens for the retake button in `renderResults` at line 397.

Concrete effects when a visitor retakes the quiz once and then taps "See Your Score":
- `/api/submit-quiz` is POSTed twice with identical data - shows up as a duplicate lead in the Thomas dashboard.
- `localStorage.telosQuizData` is written twice.
- `renderResults()` runs twice, starting a second `setInterval` count-up animation (line 319) that competes with the first - the displayed score can flicker.
- The retake-button listener is appended again, so the next cycle doubles again (2, then 4, then 8 submissions).

**How to reproduce**

1. Open `https://www.telosathleticclub.com/` and scroll to the quiz section.
2. Click "Start Assessment", answer all 8 questions, fill in name/email/phone, click "See Your Score". (Submission 1.)
3. On the results screen click "Retake Assessment".
4. Answer all 8 questions again, fill the form, click "See Your Score".
5. Open DevTools Network tab and watch `/api/submit-quiz` fire twice in step 4. Open the Thomas dashboard - two identical submissions appear.
6. Retake a third time: now four POSTs on submit. After 3 retakes (5 submissions in a row) the IP hits the 10/hr rate limit at `api/submit-quiz.js:27` and quiz submissions silently fail.

**Why it matters**

- Pollutes the lead pipeline with duplicate quiz submissions, making the dashboard's Leads count and conversion math wrong.
- After enough retakes a single visitor can poison their own IP's rate limit and then legitimate submissions silently fail (the fetch is `.catch(function(){})` per the quiz code, so the user sees no error - their final submission is just lost).
- Same listener accumulation runs on `pricing.html` where the quiz is the price gate - same risk that visitors who play with the quiz before checking pricing will produce duplicates.

**Suggested fix**

Replace the `addEventListener` setup with a guarded one-time bind, or rebuild the button node before binding. Cleanest fix:

```js
// js/quiz.js:222 - inside setupLeadCapture
const submitBtn = quizLeadCapture.querySelector('.quiz-see-score-btn');
const fresh = submitBtn.cloneNode(true);
submitBtn.parentNode.replaceChild(fresh, submitBtn);
fresh.addEventListener('click', function (e) { ... });
```

Same pattern for the retake button inside `renderResults` (line 396).

### Bug 2 - P2 - Weekly summary email reports streak = 0 for almost everyone

**ID:** 2026-05-30-2
**Severity:** P2
**Area:** Weekly client recap email (cron)
**File:** `api/cron/weekly-summary.js:87-90`

**What's wrong**

The streak loop walks d = 0..6 backwards and breaks on the first day with no daily log. d = 0 is "today". The cron runs Monday 14:00 UTC (`vercel.json:21`) - i.e. 9:00 AM Eastern - which is before most clients have submitted their Monday check-in. The Monday slot is therefore empty, the loop breaks immediately at d = 0, and `streak` is reported as `0` even for clients who logged Tuesday through Sunday in an unbroken run.

```js
for (var d = 0; d < 7; d++) {
  if (weekData[0][d] && weekData[0][d].result) streak++;
  else break;
}
```

**How to reproduce**

1. Pick any active client whose `client_dailylog:{id}:{YYYY-MM-DD}` exists for the previous Sunday, Saturday, Friday, etc. (i.e. a real recent streak).
2. Trigger the cron manually before that client has logged today (or just inspect what the cron computed in production logs from any past Monday morning send):
   ```
   curl -X POST https://www.telosathleticclub.com/api/cron/weekly-summary \
     -H "Authorization: Bearer $CRON_SECRET"
   ```
3. Inspect the rendered email body (`buildWeeklySummary`, line 124-141). The "Current Streak" line will show `0 days` even though the client visibly has 6 consecutive logged days.

**Why it matters**

- Every weekly recap email - the headline retention touchpoint for clients - shows an incorrect 0 streak for any client who hasn't already logged Monday at 9am ET (i.e. virtually everyone). It signals to a paying client "you're at zero" the moment they wake up, which is the opposite of the email's purpose.
- The email is sent to every active client with `portalEnabled` and `emailEnabled !== false`. So scope = the entire active roster.

**Suggested fix**

Start the streak loop from yesterday (or from the most recent logged day) rather than today. Simplest:

```js
// api/cron/weekly-summary.js:86 - skip today's empty slot
var startD = (weekData[0][0] && weekData[0][0].result) ? 0 : 1;
for (var d = startD; d < 7; d++) {
  if (weekData[0][d] && weekData[0][d].result) streak++;
  else break;
}
```

This preserves "today counts if logged" while not breaking the chain when the cron fires before today's check-in.
