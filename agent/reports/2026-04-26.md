# Telos Bug Hunter — 2026-04-26

**Focus:** Functional
**Bugs found:** 3

## Summary

First run of the agent. Hunted across the public lead-capture forms, the
admin client CRUD, and the client portal auth surface. Three reproducible
functional bugs: the pricing-page lead gate can be bypassed without entering
a name or email, deleting a portal-enabled client orphans the email lookup
and permanently locks that address, and the quiz lead-capture submit handler
stacks a new listener on every retake (causing duplicated POSTs and animation
glitches). All three reproduce from a clean codebase read; none rely on race
conditions or environment quirks.

## Bugs

### BUG-20260426-01 — Pricing-page lead gate can be bypassed with empty form

- **Severity:** P1
- **Area:** functional
- **Location:** `js/quiz.js:225-275`, `pricing.html:972-978`

**What's wrong**
The "See Your Score" button in the quiz lead-capture step is
`<button type="button">` (`index.html:578`), so the browser never enforces the
`required` attributes on the name/email inputs. The JS click handler at
`js/quiz.js:225` reads the inputs without validating them, then unconditionally
runs `localStorage.setItem('telosQuizCompleted', 'true')` (line 247) and
shows the results pane (lines 272-274) before the network call. On
`pricing.html` that flag is the *only* gate guarding the actual prices
(`pricing.html:972`), so a visitor can reveal the full pricing table with
empty inputs.

**Reproduction**
1. Open `https://www.telosathleticclub.com/pricing` in a fresh browser
   (no `telosQuizCompleted` in `localStorage`).
2. Click through the eight quiz questions (any answer).
3. On the lead-capture screen, leave name, email, and phone blank.
4. Click "See Your Score".
5. Results render. The API request returns 400, but localStorage is now
   `telosQuizCompleted=true` and the results screen exposes the
   "View Pricing" button, which reveals the gated `#pricingContent`.
6. Refresh: the gate stays open because of the localStorage flag.

**Impact**
Defeats the lead-capture funnel for the pricing page. Any visitor who is
mildly curious can inspect tier prices ($1,497 / $1,997 / $2,497 / $6,997)
without ever giving Thomas an email address. Also corrupts analytics:
empty-form submissions hit `/api/submit-quiz`, which rejects them with 400
but still increments the per-IP rate-limit counter.

**Suggested fix**
In `js/quiz.js` `setupLeadCapture`, validate `nameInput.value.trim()` and
`emailInput.value.trim()` (regex match) before doing anything else; bail out
with an inline error if either is missing. Move the `localStorage.setItem`
and the UI transition into the `.then()` of the fetch so the gate only opens
on a 200 response.

### BUG-20260426-02 — Deleting a portal-enabled client permanently locks their email

- **Severity:** P1
- **Area:** functional
- **Location:** `api/dashboard/clients.js:142-154`

**What's wrong**
The DELETE handler removes `client:{id}` and the `clients_index` membership
but never deletes `client_email:{normalizedEmail}`. That key is created by
`api/dashboard/client-portal.js:117` whenever the coach enables the portal,
and it is the same key the POST handler checks at `clients.js:70` to enforce
"email already in use". After a portal-enabled client is deleted, the lookup
points at a non-existent client ID and blocks the same email forever.

**Reproduction**
1. POST `/api/dashboard/clients` to create a client with
   `email: "alice@example.com"`.
2. POST `/api/dashboard/client-portal` with `enabled: true` for that client
   (creates `client_email:alice@example.com -> {id}`).
3. DELETE `/api/dashboard/clients?id={id}` — succeeds.
4. POST `/api/dashboard/clients` again with `email: "alice@example.com"` —
   responds `400 {error: "Email already in use by another client"}`.
5. Verify in Redis: `GET client_email:alice@example.com` still returns the
   deleted ID.

**Impact**
A coach who removes a client (cancellation, test record cleanup, mistaken
add) cannot re-add the same person without manual Redis surgery. Also leaves
an orphaned login lookup that could route a future ID collision incorrectly,
though `INCR id:clients` makes that unlikely in practice.

**Suggested fix**
In the DELETE branch, fetch the client record before deleting, then if
`client.email` exists also `DEL client_email:{normalizeEmail(client.email)}`.
While here, consider also cleaning the per-client portal keys
(`client_dailylog:*`, `client_545_*`, etc.) or at least documenting that
they are intentionally preserved.

### BUG-20260426-03 — Quiz lead-capture stacks a new submit listener on every retake

- **Severity:** P2
- **Area:** functional
- **Location:** `js/quiz.js:215-225`, `js/quiz.js:415-442`

**What's wrong**
`setupLeadCapture()` is invoked from `handleAnswerSelect` every time the
user finishes question 8 (`js/quiz.js:215`). It calls
`submitBtn.addEventListener('click', ...)` without first removing prior
handlers, and `resetQuiz()` does nothing to clean them up. Each retake adds
another listener; after N retakes a single click fires N+1 handlers, which
means N+1 POSTs to `/api/submit-quiz`, N+1 concurrent count-up `setInterval`
animations on the score number, and N+1 listeners attached to the retake
button by the duplicated `renderResults()` call.

**Reproduction**
1. Open `index.html`, complete the quiz, submit lead capture.
2. Click "Retake quiz".
3. Open DevTools → Network. Complete the quiz again and submit.
4. Observe two POSTs to `/api/submit-quiz` and a flickering score count-up
   in the results ring.
5. Each subsequent retake adds one more duplicate POST.

**Impact**
After ~10 retakes (a tester or a curious visitor), the per-IP rate limit
on `/api/submit-quiz` (10/hr) trips and legitimate submissions start
failing. Duplicate submissions also pollute the `quiz_submissions` ZSET
since each entry uses the unique `Date.now()` member — no dedupe.

**Suggested fix**
Attach the click handler with `{ once: true }`, or hoist the handler into
`DOMContentLoaded` scope and only call `addEventListener` once. Same change
needed for the retake button registration in `renderResults()` at
`js/quiz.js:397`.
