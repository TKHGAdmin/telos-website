## P2 Stale `client_email` lookup is left behind on client delete

- **ID:** 2026-05-03-01
- **Severity:** P2
- **Area:** api / dashboard
- **File(s):** `api/dashboard/clients.js:142-154`
- **Type:** data

### What's wrong
The DELETE handler removes `client:{id}` and the `clients_index` ZSET entry, but never removes the `client_email:{normalizedEmail}` lookup. That lookup is created on portal enable / email change in `api/dashboard/client-portal.js:117` and `api/dashboard/clients.js:131`, and is the entry point for client login (`api/client/login.js:24`).

```js
// api/dashboard/clients.js
if (req.method === 'DELETE') {
  var id = req.query.id || (req.body && req.body.id);
  if (!id) return res.status(400).json({ error: 'ID required' });
  await redis('DEL', 'client:' + id);
  await redis('ZREM', 'clients_index', String(id));
  return res.status(200).json({ ok: true });
}
```

### Repro
1. In `/thomas` Clients tab, create a client with email `test@example.com`.
2. In Client Portal tab, enable portal access (sets `client_email:test@example.com -> id`).
3. Back in Clients tab, delete the client.
4. Try to create a new client with the same email — POST to `/api/dashboard/clients` returns `400 Email already in use by another client` (clients.js:71-73), even though no client exists.

### Why it matters
Coach-side workflow breaks: a deleted email address is permanently unusable until the orphan key is purged manually in Redis. Also leaves a dangling pointer that, in `client/login.js:24-32`, walks to a non-existent `client:{id}` and emits a generic "Invalid email or password" — debuggable but confusing for any account admin.

### Suggested fix
In the DELETE branch, fetch the client first, normalize its email, and `DEL client_email:<normalized>` before removing the record.

## P2 Weekly summary email reports streak=0 every Monday morning

- **ID:** 2026-05-03-02
- **Severity:** P2
- **Area:** api / cron
- **File(s):** `api/cron/weekly-summary.js:86-90`
- **Type:** logic

### What's wrong
The streak loop starts at `d=0`, which is **today** (the cron's run-day):

```js
// api/cron/weekly-summary.js
for (var d = 0; d < 7; d++) {
  if (weekData[0][d] && weekData[0][d].result) streak++;
  else break;
}
```

`vercel.json` schedules this cron at `0 14 * * 1` — Monday 14:00 UTC, which is 9-10am ET. The vast majority of clients have not logged Monday's daily check-in by that time. So `weekData[0][0]` is empty, the loop breaks immediately, and the email reports `Current Streak: 0 days` to clients who actually have a multi-day active streak.

### Repro
Static analysis. Reproduce by running the cron handler against a fixture where:
- `client_dailylog:{id}:{today}` is missing
- `client_dailylog:{id}:{today-1}` ... `{today-6}` exist

Expected streak: 6. Actual: 0.

### Why it matters
Every Monday's recap email (the highest-touch retention signal in the product) underreports streak for any client who hasn't checked in before the cron fires. Demoralizing and inaccurate. Engagement-check.js (a separate cron) does not have this bug — it explicitly tolerates `lastLogDaysAgo === 0`, so the inconsistency makes the bug easy to miss in review.

### Suggested fix
Start the streak loop at `d=1` (skip "today" since it's still in progress at cron time), or iterate from `d=0` and tolerate a single missing-today before breaking:

```js
for (var d = 0; d < 7; d++) {
  if (weekData[0][d] && weekData[0][d].result) streak++;
  else if (d === 0) continue; // today not logged yet
  else break;
}
```

## P2 Quiz retake stacks duplicate submit/retake click handlers

- **ID:** 2026-05-03-03
- **Severity:** P2
- **Area:** index.html / pricing.html (shared `js/quiz.js`)
- **File(s):** `js/quiz.js:225`
- **File(s):** `js/quiz.js:397`
- **Type:** logic

### What's wrong
`setupLeadCapture()` runs every time the user reaches question 8 (`js/quiz.js:215`). It attaches a fresh `click` listener to `.quiz-see-score-btn` without removing prior listeners (`:225`). Likewise `renderResults()` adds a `click` listener to `.quiz-retake-btn` (`:397`) on every render. After one retake, the user has two submit handlers; after two retakes, three; etc.

### Repro
1. Open `/` (or `/pricing`), start the quiz, answer all 8 questions.
2. Submit the lead-capture form (1× POST to `/api/submit-quiz`).
3. Click "Retake Quiz", answer all 8 questions again.
4. Submit the lead-capture form a second time → DevTools shows **2× POSTs** to `/api/submit-quiz`, the localStorage `telosQuizData` is written twice, and the score count-up animation runs twice with conflicting `setInterval`s on the same DOM node.

### Why it matters
- Duplicates lead submissions in the admin pipeline (already counts against the 10/hour rate limit in `api/submit-quiz.js:23-29`).
- Score ring on retake animates erratically (overlapping intervals on the same element).
- Pollutes lead analytics — a single user appears as N entries.

### Suggested fix
Move the listener attachment outside `setupLeadCapture()` / `renderResults()` and attach once on DOMContentLoaded, OR use `submitBtn.onclick = handler` (replaces) instead of `addEventListener` for these two buttons.
