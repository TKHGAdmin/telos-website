## P1-001 — Quiz lead capture bypasses validation, allowing pricing gate bypass and lead loss

**Severity:** P1
**Area:** Functional / lead capture
**Files:** `js/quiz.js:225-275`, `pricing.html:589-595`, `index.html:574-580`

### Summary
The "See Your Score" button on the lead-capture step of the Execution Score Quiz performs no client-side validation before persisting completion state and revealing gated content. A user can click through with empty fields and:

1. Get the gated quiz results revealed.
2. Have `localStorage.telosQuizCompleted = 'true'` set, which bypasses the pricing-page overlay on every subsequent visit (`pricing.html:972-978`).
3. Trigger a `POST /api/submit-quiz` that the API rejects (400 — name/email required), so Thomas captures **no lead** for that visitor.

The visible `<input ... required>` attributes in the form are non-functional because the trigger button is `<button type="button">`, not `type="submit"`, so the browser's native form validation never fires. The click handler in `quiz.js` (`submitBtn.addEventListener('click', ...)`) immediately writes to `localStorage` and renders results without checking that name/email are non-empty or syntactically valid.

### Reproduction
1. Open `/pricing` (or scroll to the quiz on `/`) in a fresh browser/incognito.
2. Step through all 8 quiz questions until the "Your Score Is Ready" lead form appears.
3. Click **See Your Score** without filling any field.
4. **Observed:** Results render. On `/pricing`, the overlay closes and prices are visible. `localStorage.telosQuizCompleted === 'true'` is set, so future visits skip the gate entirely.
5. **Expected:** Form should require non-empty name + valid email before storing completion or revealing results.

### Why this matters
- **Lead capture is the entire purpose of the gate.** Every bypass is a lost lead Thomas could have contacted.
- **The pricing page is meant to be quiz-gated** (per `CLAUDE.md`: "Pricing page gating: pricing.html shows an Execution Score Quiz overlay before revealing prices"). The gate is currently a soft barrier any visitor can punch through with three button clicks.
- The bug exists on **both** `index.html` (homepage inline quiz) and `pricing.html` (overlay quiz), since both share `js/quiz.js`.

### Suggested fix
In `js/quiz.js` `setupLeadCapture()`, validate before `localStorage.setItem` / `renderResults`:

```js
if (!quiz.userName.trim() || !/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(quiz.userEmail)) {
  // surface inline error and return early
  return;
}
```

Alternative: change `<button type="button">` to `<button type="submit">` and bind the handler to `form.addEventListener('submit', ...)`. The native `required` and `type="email"` validators will then fire automatically. This is the pattern already used by `protein-calculator.html` (`gateForm.addEventListener('submit', ...)` at line 810), so adopting it would also harmonize the codebase.

## P2-002 — Charleston landing page (`/chs`) missing "Client Login" nav link

**Severity:** P2
**Area:** Functional / navigation consistency
**Files:** `chs.html:794-800` (desktop nav), `chs.html:808-814` (mobile nav)

### Summary
`CLAUDE.md` documents the convention: *"All 28 public pages (5 root + 23 blog) have a 'Client Login' link before 'Book a Call' in both desktop and mobile navs."* The newly added Charleston landing page (`chs.html`, commit `3d87367`) does not include this link in either nav, while every other root page (`index.html`, `pricing.html`, `resources.html`, `protein-calculator.html`, `hyrox-predictor.html`) does.

### Reproduction
1. Visit `/chs` on desktop and mobile.
2. **Observed:** Nav contains: Services, Events, The Coach, Online Coaching, Apply. No Client Login.
3. **Expected:** A Client Login link present, matching the other 5 root pages.

Verified by grep — every other root page has two `<a href="client-dashboard" class="nav-client-login">` entries (one desktop, one mobile). `chs.html` has zero.

### Why this matters
- Existing clients who land on `/chs` (e.g., from social, ads, or word-of-mouth promoting Charleston in-person services) have no in-page path to log in.
- Breaks the documented site-wide nav contract; if anyone ever adds a sixth root page by copying chs.html's nav, the regression compounds.

### Suggested fix
Add to both nav blocks in `chs.html`, in the same position used elsewhere (immediately before the CTA):

```html
<li><a href="client-dashboard" class="nav-client-login">Client Login</a></li>
```

Note: The `chs-page` body class scopes custom styles, but the nav uses standard markup, so the existing `nav-client-login` styles from `css/style.css` should apply unchanged. Visual review on Charleston's hotel-aesthetic theme is recommended after the change.
