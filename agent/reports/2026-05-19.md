### BUG-20260519-01 — Deleting a client orphans `client_email:{email}` and blocks re-creating with the same address

- **Severity:** P1
- **Area:** api/dashboard/clients
- **File(s):** `api/dashboard/clients.js:143-153`, interacts with `api/dashboard/client-portal.js:115-124`
- **Status:** new

**What's wrong**
The admin DELETE for a client removes `client:{id}` and the `clients_index` ZSET entry, but never deletes the `client_email:{normalizedEmail}` lookup that was created when the portal was enabled. The lookup now points to a non-existent client id.

**Reproduction**
1. Create client A with email `foo@example.com`, enable portal (`POST /api/dashboard/client-portal { enabled: true }`) — this writes `client_email:foo@example.com -> A.id`.
2. `DELETE /api/dashboard/clients?id=A.id`. The handler only runs `DEL client:A.id` and `ZREM clients_index A.id` (lines 147-148).
3. `POST /api/dashboard/clients { name, email: "foo@example.com", tier }` now fails with `400 "Email already in use by another client"` (clients.js:70-73) because the orphaned lookup still resolves.

**Impact**
Any client that ever had portal access can never have their email reused — including the legitimate case where the same person re-signs up after churning, or where a record was created in error and re-created cleanly. Operator-visible. Hits Thomas the next time he tries to re-add a churned client.

**Suggested fix**
In `api/dashboard/clients.js:143-154`, fetch the client before deleting; if `client.email` exists, also run `DEL client_email:{normalizeEmail(client.email)}`. While there, also drop the per-client portal-content keys (`client_nutrition_plan`, `client_545_*`, `client_training_program`, etc.) — they currently leak forever too.

### BUG-20260519-02 — Execution Score ignores the documented "tasks max 50%" cap (client dashboard + weekly email)

- **Severity:** P1
- **Area:** client-dashboard.html, api/cron/weekly-summary
- **File(s):** `client-dashboard.html:2773-2788`, `api/cron/weekly-summary.js:72-83`
- **Status:** new

**What's wrong**
CLAUDE.md specifies the Execution Score as `routine 25% + tasks 10% each (max 50%) + daily log 25%`. Both implementations add `+10` per completed task with no per-section cap, then `Math.min(100, score)` on the total. A coach who configures 6+ daily tasks lets the tasks bucket alone reach or exceed 60%, which displaces routine and log contributions in a 100-point cap and overstates the score.

**Reproduction**
Inspect `client-dashboard.html:2779-2787`:
```js
var tasks = daily.tasks || [];
tasks.forEach(function(t) { if (t.completed) score += 10; });
// ... no cap here ...
score = Math.min(100, score);
```
Same shape at `api/cron/weekly-summary.js:77-82`. A client with 7 completed tasks, routine done, no daily log: 25 (routine) + 70 (tasks) = 95% — should be 25 + 50 = 75% per spec.

**Impact**
The hero ring on the Home tab and the weekly summary email both report inflated execution scores whenever a client has more than 5 daily tasks. The two implementations drift from CLAUDE.md but drift identically, so the dashboard and email agree with each other — meaning Thomas would only notice by reading the spec.

**Suggested fix**
Replace the inner loop with a clamped subtotal in both files:
```js
var taskPoints = Math.min(50, tasks.filter(function(t){ return t.completed; }).length * 10);
score += taskPoints;
```
Apply at `client-dashboard.html:2779-2780` and `api/cron/weekly-summary.js:78`. Confirm the same pattern isn't repeated elsewhere — `grep -n "score += 10" client-dashboard.html` (lines 2780 and 4497 candidates).

### BUG-20260519-03 — Admin `verifyPassword` throws `RangeError` on multibyte input → 500 instead of 401

- **Severity:** P2
- **Area:** api/lib/auth, api/dashboard/login
- **File(s):** `api/lib/auth.js:52-59`, `api/dashboard/login.js:10-12`
- **Status:** new

**What's wrong**
`verifyPassword` compares `inputStr.length` (UTF-16 code-unit count) before feeding `Buffer.from(str,'utf-8')` to `crypto.timingSafeEqual`. For ASCII the two counts agree; for multibyte input they don't, and `timingSafeEqual` throws `RangeError: Input buffers must have the same byte length`. The dashboard login handler has no try/catch, so the request returns a 500 rather than a 401.

**Reproduction**
Assuming `DASHBOARD_PASSWORD = "abc"` (3 chars, 3 bytes):
```bash
curl -s -X POST https://www.telosathleticclub.com/api/dashboard/login \
  -H 'content-type: application/json' \
  -d '{"password":"ñax"}'   # 3 chars, 4 bytes
```
The lengths match in JS (`"ñax".length === 3`) so the length guard passes, then the buffer compare throws. Vercel returns 500.

**Impact**
Two issues: (1) a stack trace lands in function logs on every Unicode attempt (noise), and (2) the 500-vs-401 difference is a side-channel that distinguishes "your input has the same character count as the real password" from "wrong length" — small but real password-length leak.

**Suggested fix**
Compare byte length instead of char length at `api/lib/auth.js:55`:
```js
const inputBuf = Buffer.from(inputStr, 'utf-8');
const expectedBuf = Buffer.from(expected, 'utf-8');
if (inputBuf.length !== expectedBuf.length) return false;
return crypto.timingSafeEqual(inputBuf, expectedBuf);
```

### BUG-20260519-04 — `/api/client/daily-log` and `/api/client/activity-log` accept an unbounded `range` and build a Redis pipeline of that size

- **Severity:** P2
- **Area:** api/client/daily-log, api/client/activity-log
- **File(s):** `api/client/daily-log.js:13-34`, `api/client/activity-log.js:13-34`
- **Status:** new

**What's wrong**
Both endpoints compute `var days = range || 7;` from `parseInt(req.query.range)` with no upper bound, then push `days` commands into `redisPipeline`. An authenticated client can request `?range=100000` and the function will issue a 100k-key pipeline — slow, expensive against the Upstash plan, and likely to bump the 60s Vercel function timeout.

**Reproduction**
With a valid `telos_client_session` cookie:
```bash
curl -s 'https://www.telosathleticclub.com/api/client/daily-log?range=50000' \
  -b 'telos_client_session=...'
```

**Impact**
Authenticated DoS / cost amplification. Each portal client is a trusted user, but the client app only ever asks for 7/30/90, so there's no legitimate reason the endpoint should honor 100000. Doesn't affect anonymous traffic.

**Suggested fix**
Clamp at the top of each GET handler:
```js
var days = Math.min(Math.max(parseInt(req.query.range, 10) || 7, 1), 365);
```
Apply at `api/client/daily-log.js:23` and `api/client/activity-log.js:23`. The same shape appears in `api/dashboard/client-portal.js:55` (hard-coded 30) which is fine.

### BUG-20260519-05 — `/api/submit-email` 500s when `body.email` is not a string

- **Severity:** P2
- **Area:** api/submit-email (also affects api/submit-chs-application)
- **File(s):** `api/submit-email.js:16-18`, similar shape at `api/submit-chs-application.js:21-25`
- **Status:** new

**What's wrong**
Validation reads `email.includes('@')` directly. If a caller posts `{"email": 1}` or `{"email": ["a@b.com"]}` or any non-string value, `!email` is false but `.includes` either doesn't exist on the value or behaves on the wrong type — for numbers, it throws `TypeError: email.includes is not a function`. The handler catches nothing in the validation block, so the catch at line 49 swallows it and returns 500. The user sees `Internal error` instead of `Valid email required`.

**Reproduction**
```bash
curl -s -X POST https://www.telosathleticclub.com/api/submit-email \
  -H 'content-type: application/json' \
  -d '{"email": 1, "source": "protein-calculator"}'
```
Expected: `400 {"error":"Valid email required"}`. Actual: `500 {"error":"Internal error"}` (and a stack-trace in function logs on every garbage POST).

**Impact**
Low user impact — the lead-capture forms only ever send strings. But the public endpoints are CORS-open (`vercel.json: Access-Control-Allow-Origin: *`) and a scanner hitting them with malformed JSON turns into a steady stream of 500s in monitoring. The same one-liner crashes `/api/submit-chs-application` (line 21: `(body.email || '').trim()` throws when `email` is a number).

**Suggested fix**
Coerce to string before validating:
```js
const email = typeof req.body.email === 'string' ? req.body.email.trim() : '';
if (!email || !/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email)) {
  return res.status(400).json({ error: 'Valid email required' });
}
```
While there, replace the `.includes('@')` check with the regex already used in `submit-chs-application.js:35` so `"@"`, `"foo@"`, `"@bar"` stop landing in the lead list.
