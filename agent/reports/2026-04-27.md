# Telos Bug Hunter — 2026-04-27

**Focus:** security
**Findings:** 3

## Summary

First run. Bootstrapped agent memory and ran a security pass on the serverless API surface (auth, IDOR, XSS, secrets, CSRF, cookies, cron). Foundations are solid: every protected endpoint correctly gates on `verifySession` / `verifyClientSession`, `clientId` is always derived from the session (no IDOR), passwords use PBKDF2 with 100k iterations, comparisons are timing-safe, cookies are `HttpOnly; Secure` with the documented `SameSite` choices, cron endpoints check `CRON_SECRET`, and no secrets are committed. The single real gap is rate limiting: public form endpoints (`submit-quiz`, `submit-email`, `submit-chs-application`) all have Redis-backed limiters, but the three auth-adjacent endpoints below do not. That makes admin/client login brute-force-friendly and exposes the password-reset flow to email-bombing of any client.

## Findings

### [P1] 20260427-01 — Admin login has no rate limiting

- **File:** `api/dashboard/login.js:1-17`
- **Severity:** P1
- **Status:** new

**What's wrong**
The `/thomas` admin dashboard is gated by a single shared `DASHBOARD_PASSWORD`. The login handler verifies the password and issues a session cookie with no per-IP throttling, lockout, or backoff. An attacker can submit unlimited POSTs against `/api/dashboard/login` to brute-force the admin password, which would give full access to every client record, lead pipeline, revenue numbers, and the entire `/thomas` dashboard.

The contrast is sharp: every public *submission* endpoint already uses Redis-backed limiting (`ratelimit:quiz:${ip}` in `api/submit-quiz.js:24`, `ratelimit:email:${ip}` in `api/submit-email.js:29`, `ratelimit:chs:${ip}` in `api/submit-chs-application.js:41`), but the highest-value endpoint in the system has none.

**Repro**
```bash
for i in {1..10000}; do
  curl -s -X POST https://www.telosathleticclub.com/api/dashboard/login \
    -H 'Content-Type: application/json' \
    -d "{\"password\":\"guess$i\"}"
done
```
No 429s, no slowdown, nothing in Redis to track attempts.

**Suggested fix**
Apply the same `ratelimit:` Redis pattern (e.g. 10 attempts/hour per IP) before calling `verifyPassword`, and consider a tighter global counter since this protects only one shared password.

### [P1] 20260427-02 — Client login has no rate limiting

- **File:** `api/client/login.js:1-60`
- **Severity:** P1
- **Status:** new

**What's wrong**
`POST /api/client/login` accepts `{ email, password }` and runs the full PBKDF2 verification with no per-IP or per-email throttling. The endpoint is reachable from any origin (it sets `Set-Cookie` for `telos_client_session` directly). An attacker can perform credential-stuffing across the client list — the email field doubles as enumeration once the response time diverges between the early `client_email:*` miss path (lines 23-27) and the full PBKDF2 path (line 46), though the bigger issue is straightforward brute force.

The same `ratelimit:` infrastructure is already used in three sibling endpoints, so the fix is mechanical.

**Repro**
```bash
for i in {1..1000}; do
  curl -s -X POST https://www.telosathleticclub.com/api/client/login \
    -H 'Content-Type: application/json' \
    -d "{\"email\":\"target@example.com\",\"password\":\"pw$i\"}"
done
```
All requests are accepted; no limiter touches Redis.

**Suggested fix**
Add a Redis counter keyed on both IP and normalized email (e.g. `ratelimit:client_login:ip:${ip}` and `ratelimit:client_login:email:${email}`, both with 1h TTL, ~10 attempts each) before the password check.

### [P1] 20260427-03 — Password-reset endpoint allows email bombing of any client

- **File:** `api/client/reset-password.js:8-68`
- **Severity:** P1
- **Status:** new

**What's wrong**
The reset endpoint deliberately returns `200 { ok: true }` for unknown emails (good — prevents enumeration), but it does so with no rate limiting on either path:

1. **Email flood** (lines 19-67): each POST that matches a real, portal-enabled client triggers a Resend API call sending a real email to the client's inbox. An attacker who knows or guesses a client email can send unbounded reset emails — both an inbox-flood DoS against the user and a Resend quota/cost attack on the business.
2. **Token redemption** (lines 70-95): the second branch validates `body.token` against `password_reset:*` keys with no throttling. Tokens are 32 random bytes so brute force is statistically infeasible, but the lack of limiting is still worth flagging because the same handler should be protected uniformly.

**Repro**
```bash
# Email flood — every iteration triggers a real email if the address belongs to a portal-enabled client.
for i in {1..1000}; do
  curl -s -X POST https://www.telosathleticclub.com/api/client/reset-password \
    -H 'Content-Type: application/json' \
    -d '{"email":"known-client@example.com"}'
done
```
The user receives 1,000 password-reset emails; Resend is billed for each.

**Suggested fix**
Apply two limiters at the top of the handler:
- Per-IP: `ratelimit:reset:ip:${ip}` — e.g. 5/hour.
- Per-email (after normalization): `ratelimit:reset:email:${email}` — e.g. 3/hour, applied even on the "unknown email" branch so timing stays uniform.

## Checked but clean

- **Auth gating** — every handler under `api/dashboard/*` calls `verifySession`; every protected handler under `api/client/*` calls `verifyClientSession`. No missing checks.
- **IDOR** — no client endpoint reads `clientId` from the request body or query; it is always derived from the verified session token.
- **XSS sinks** — no `innerHTML` / `outerHTML` / `document.write` interpolating Redis or user-controlled strings on the public marketing pages or in the API responses.
- **Secrets in repo** — `git log -p` and a tree scan show no committed API keys, tokens, or passwords; everything reads from `process.env`.
- **Timing-safe compare** — admin and client password checks both go through `crypto.timingSafeEqual` (admin via `lib/auth.js`, client via `lib/client-auth.js`).
- **Cookie hardening** — both session cookies are `HttpOnly; Secure`. Admin is `SameSite=Strict`; client is `SameSite=None; Partitioned` (intentional for Whop iframe per CLAUDE.md, not a bug).
- **Cron endpoints** — `api/cron/weekly-summary.js` and `api/cron/engagement-check.js` both verify `Authorization: Bearer ${CRON_SECRET}` and fail closed when the env var is absent.
- **CSRF** — admin endpoints rely on `SameSite=Strict` cookies, which is sufficient given there is no public, non-iframe state-changing surface for the admin role.
- **Client food-search proxy** — `api/client/food-search.js` only forwards to a hardcoded Open Food Facts URL with the user's query string parameter, no SSRF risk.
