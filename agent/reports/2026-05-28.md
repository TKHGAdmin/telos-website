## Bug 20260528-01: Quiz retake duplicates `/api/submit-quiz` POSTs and re-runs results animation

- **Severity:** P2
- **Area:** Quiz flow (homepage + pricing overlay)
- **Files:** js/quiz.js:225, js/quiz.js:397
- **Status:** new

### What's wrong
`setupLeadCapture()` calls `submitBtn.addEventListener('click', ...)` every time the user reaches question 9 (the lead capture step). After the user clicks "Retake Assessment", `resetQuiz()` clears state and resets the form but does NOT remove the previously attached click listener. The same `submitBtn` DOM node is reused, so the next time the user finishes the quiz a *second* click handler is attached. One retake → two POSTs to `/api/submit-quiz`; two retakes → three POSTs; etc. The same pattern occurs for the retake button itself in `renderResults()` (`retakeBtn.addEventListener('click', resetQuiz)` on js/quiz.js:397), so subsequent retake clicks fire `resetQuiz` N+1 times.

### Repro / Evidence
1. Open the homepage, scroll to the Execution Score Quiz, complete it (Q1-Q8 + lead capture form), submit.
2. On the results screen click "Retake Assessment".
3. Complete the quiz again. Click "See Your Score".
4. Open DevTools → Network panel and observe **two** identical POSTs to `/api/submit-quiz` for the single click. The Redis ZSET `quiz_submissions` will contain two near-identical entries (timestamps differ by milliseconds because each handler builds its own `new Date().toISOString()` on js/quiz.js:38). Repeat retakes scale linearly.

Relevant code:
```js
// js/quiz.js:221-276 — setupLeadCapture is invoked every time the user lands on the lead capture step
function setupLeadCapture() {
  const submitBtn = quizLeadCapture.querySelector('.quiz-see-score-btn');
  submitBtn.addEventListener('click', function(e) { ... fetch('/api/submit-quiz', ...) ... });
}
```

```js
// js/quiz.js:396-397 — same pattern on the retake button
const retakeBtn = quizResults.querySelector('.quiz-retake-btn');
retakeBtn.addEventListener('click', resetQuiz);
```

`resetQuiz()` (js/quiz.js:416) resets state and `form.reset()` but never removes any listener.

### Suggested fix
Two options, pick one:
- Cleanest: replace `addEventListener` with a single `submitBtn.onclick = handler` assignment in both `setupLeadCapture` and `renderResults` so re-assignment overwrites the previous handler. Same change for the retake button.
- Alternative: hoist `setupLeadCapture` and the retake-button binding into the one-shot `DOMContentLoaded` body so they only run once, then have `handleAnswerSelect` just toggle visibility instead of re-binding.

### Why this matters
Each retake silently doubles the data Thomas sees in the Pipeline / Leads tab, inflating lead counts and creating duplicates that have to be deduped. On the results screen the count-up `setInterval` is started twice on a single submit, so the score number flickers/over-counts. Low-frequency path, but the data quality impact is real.

## Bug 20260528-02: `shop.js` cache-bust version is out of sync between shop.html and product.html

- **Severity:** P3
- **Area:** Shop pages
- **Files:** shop.html:841, product.html:822

### What's wrong
`shop.html` loads `js/shop.js?v=1`, while `product.html` loads `js/shop.js?v=2`. The project convention per CLAUDE.md is to bump the version on *all* pages that reference an asset when the asset changes. Whoever bumped `product.html` to `?v=2` (commits 21cc926 / f5cb8c4 added the product detail logic and the image-gallery thumbnails) forgot to update `shop.html`.

### Repro / Evidence
```
shop.html:841:  <script src="js/shop.js?v=1"></script>
product.html:822:  <script src="js/shop.js?v=2"></script>
```

The same file is served either way, so behavior is identical right now. But: browsers cache `?v=1` and `?v=2` as separate resources, doubling cache footprint, and the convention is silently broken so the next person bumping the version will likely repeat the mismatch.

### Suggested fix
Update shop.html:841 to `<script src="js/shop.js?v=2"></script>`. Going forward, treat `js/shop.js?v=N` like the `?v=16` CSS stamp - grep both files when bumping.

### Why this matters
Pure consistency. Low impact today, but cache-bust drift is the kind of thing that compounds — once shop.html ships v=3 while product.html stays at v=2, users on one page may get the new code and not the other. Easy to fix now while it's a one-line diff.
