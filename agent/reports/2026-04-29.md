## BUG-2026-04-29-01 — Quiz lead-capture form bypasses required-field validation; empty leads silently dropped

- **Severity:** P1
- **Component:** `js/quiz.js`, `index.html`, `pricing.html`
- **Status:** New

### Description
The "See Your Score" button on the Execution Score Quiz lead-capture step is wired with a `click` handler that calls `e.preventDefault()` before doing anything else. The button is declared as `<button type="button">`, so no form `submit` event ever fires — meaning the `required` attributes on the `name` and `email` inputs are **never enforced** by the browser.

A user who completes all 8 questions, leaves name and email blank, and clicks "See Your Score" will:

1. Have `quiz.userName = ""` and `quiz.userEmail = ""` written into the payload.
2. Trigger a `POST /api/submit-quiz` that the server rejects with **400 "Name and email required"** (`api/submit-quiz.js:18-20`).
3. See that 400 swallowed by `fetch(...).catch(function(){})` (`js/quiz.js:269`).
4. Still be shown the results screen.
5. Still have `localStorage.telosQuizCompleted = "true"` set (`js/quiz.js:247`), which on `pricing.html` permanently unlocks the price reveal.

The coach loses the lead entirely while the user gets full access. This affects the primary monetization funnel.

### Location
- `js/quiz.js:225-275` — handler attached to button click instead of form submit; no inline validation.
- `index.html:578` — `<button type="button" class="btn btn-primary quiz-see-score-btn">`
- `pricing.html:593` — same button markup.

### Reproduction
1. Open `pricing.html` in incognito.
2. Click through the 8-question quiz with any answers.
3. On the lead-capture screen, leave Name and Email blank.
4. Click "See Your Score".
5. Observe: results render, `/api/submit-quiz` returned 400 (DevTools Network), no submission appears in the `/thomas` Leads tab.
6. Reload `pricing.html` — the quiz overlay no longer shows; pricing is unlocked permanently.

### Suggested fix
Either:
- Change the button to `type="submit"` and attach the handler to the form's `submit` event (so `required` triggers natively), **or**
- Add explicit validation inside the click handler before calling `fetch`, e.g.

```js
if (!quiz.userName.trim() || !quiz.userEmail.includes('@')) {
  // surface inline error, do not advance
  return;
}
```

Also: only set `localStorage.telosQuizCompleted` after the fetch resolves with `ok: true` to prevent gate-bypass on submission failure.

## BUG-2026-04-29-02 — Weekly summary email reports `Current Streak: 0 days` for any client who hasn't logged Monday yet

- **Severity:** P2
- **Component:** `api/cron/weekly-summary.js`
- **Status:** New

### Description
The weekly recap cron is scheduled for Monday 14:00 UTC (≈ 9am ET). The streak counter iterates from today (`d=0`) backwards and breaks on the first missing day:

```js
// Count streak (from today backwards)
for (var d = 0; d < 7; d++) {
  if (weekData[0][d] && weekData[0][d].result) streak++;
  else break;
}
```

At 9am ET on a Monday, virtually no client has completed their Monday daily check-in yet — most haven't opened the app today. The loop therefore breaks on `d=0` and reports `streak = 0` even for a client who logged perfectly Tue–Sun the prior week.

The email is the company's main retention touchpoint; it shipping "Current Streak: 0 days" to highly-engaged clients is misleading and demoralizing.

### Location
`api/cron/weekly-summary.js:86-90` (streak loop), used in HTML at `:131`.

### Reproduction
1. Seed a test client with `client_dailylog:{id}:{date}` entries for the last 6 days but NOT today's UTC date.
2. Hit `/api/cron/weekly-summary` with the `Authorization: Bearer ${CRON_SECRET}` header.
3. Inspect the rendered email — `Current Streak: 0 days`.

### Suggested fix
Either start the streak loop from `d=1` (yesterday), or treat a missing `d=0` as "skip, don't break":

```js
var startedStreak = false;
for (var d = 0; d < 7; d++) {
  var has = weekData[0][d] && weekData[0][d].result;
  if (has) { streak++; startedStreak = true; }
  else if (startedStreak) break;
  // else: d=0 with no log yet — keep scanning into the past
}
```

The intent is clearly "consecutive recent logged days," not "consecutive logged days strictly anchored to today UTC."

## BUG-2026-04-29-03 — Admin dashboard login has no rate limiting / brute-force protection

- **Severity:** P2
- **Component:** `api/dashboard/login.js`, `api/lib/auth.js`
- **Status:** New

### Description
`POST /api/dashboard/login` accepts unlimited password attempts. The handler calls `verifyPassword` (timing-safe) and either sets a session cookie or returns 401 — there is no IP-based rate limit, no exponential backoff, no lockout.

By contrast, every other write endpoint in the codebase rate-limits by IP:
- `api/submit-quiz.js:24-29` — 10/hr
- `api/submit-email.js:29-34` — 10/hr
- `api/submit-chs-application.js:41-46` — 5/hr

The admin dashboard exposes the entire client roster, revenue, content calendar, lead pipeline, and Charleston applications. A single password guards all of it. Without a brute-force barrier, an attacker can run an unbounded online dictionary/password attack — limited only by Vercel's per-function concurrency.

This is defense-in-depth; the system is not currently known to be exploited. But the risk surface is unjustifiably wide given the existing rate-limit pattern in the same codebase.

### Location
`api/dashboard/login.js:1-17` — entire handler, no rate-limit calls.

### Reproduction
```bash
for i in {1..200}; do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST https://www.telosathleticclub.com/api/dashboard/login \
    -H 'Content-Type: application/json' \
    -d '{"password":"guess'$i'"}'
done
```
All 200 requests will hit the handler; none are throttled.

### Suggested fix
Mirror the pattern from `submit-chs-application.js`:

```js
const ip = req.headers['x-forwarded-for'] || req.headers['x-real-ip'] || 'unknown';
const rateLimitKey = `ratelimit:dash_login:${ip}`;
const count = await redis('INCR', rateLimitKey);
if (count === 1) await redis('EXPIRE', rateLimitKey, 900); // 15 min window
if (count > 10) {
  return res.status(429).json({ error: 'Too many attempts' });
}
```

Consider also a longer-lived counter (e.g. 50/day) to slow distributed attempts, and logging failed attempts so brute-force activity is visible.

## BUG-2026-04-29-04 — `submit-quiz.js` accepts emails that fail any reasonable format check

- **Severity:** P3
- **Component:** `api/submit-quiz.js`
- **Status:** New

### Description
Quiz submissions only validate `!email` — i.e., presence — with no format check at all:

```js
if (!email || !name) {
  return res.status(400).json({ error: 'Name and email required' });
}
```

`submit-email.js:18` at least checks `email.includes('@')`. `submit-chs-application.js:35` runs the regex `^[^\s@]+@[^\s@]+\.[^\s@]+$`. Quiz — the highest-volume lead source — has neither.

Result: payloads like `{"name":"x","email":"a"}` or `{"name":"x","email":"@"}` are accepted with 200 OK, persisting unusable leads to the `quiz_submissions` ZSET. The coach then attempts to follow up with garbage addresses, and lead-quality KPIs are inflated.

### Location
`api/submit-quiz.js:16-20`.

### Reproduction
```bash
curl -X POST https://www.telosathleticclub.com/api/submit-quiz \
  -H 'Content-Type: application/json' \
  -d '{"name":"Test","email":"x","scores":[1,1,1,1],"totalScore":4}'
# → {"ok":true}
```
Then check the `/thomas` Leads tab — the entry is present.

### Suggested fix
Adopt the same regex used by `submit-chs-application.js`:

```js
if (!email || !/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email)) {
  return res.status(400).json({ error: 'Valid email required' });
}
```
