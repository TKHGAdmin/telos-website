## BUG-20260509-1: Deleting a client orphans the email lookup, blocking re-creation with the same email

- **Severity:** P1
- **File(s):** api/dashboard/clients.js:142-154
- **Status:** new

### What's wrong
The DELETE handler removes `client:{id}` and the `clients_index` entry, but it
never removes the reverse-lookup `client_email:{normalizedEmail}` that was
written when the portal was enabled (api/dashboard/client-portal.js:115-118).
After deletion that key still points to a non-existent client ID. All
per-client data keys (`client_dailylog:`, `client_nutrition_*`,
`client_training_*`, `client_545_*`, `client_supplement_*`,
`client_activity_*`, plus their `*_index` ZSETs) are also left orphaned.

### How to reproduce
1. POST `/api/dashboard/clients` to create a client with `email: "demo@x.com"`.
2. POST `/api/dashboard/client-portal` with `{ clientId, enabled: true, password: "..." }` —
   this writes `client_email:demo@x.com → id` (client-portal.js:115-118).
3. DELETE `/api/dashboard/clients?id={id}` — succeeds; only `client:{id}` and
   `clients_index` are touched (clients.js:147-148).
4. Try to POST a new client with the same email. clients.js:70-73 sees the
   stale `client_email:` entry and returns
   `400 "Email already in use by another client"`.
5. (Side effect) Issue a login with `demo@x.com` and any password — login.js:24
   resolves the email to the deleted ID, then line 30 returns
   `401 "Invalid email or password"`. No data leak, but the system thinks the
   account still half-exists.

Compare with `api/dashboard/modules.js:160-186`, which does cascade through
all secondary indexes on DELETE — that's the pattern this handler should
mirror.

### Why it matters
A coach who churns a client and later wants to re-onboard them with the same
email is silently blocked. The error message ("Email already in use by another
client") is wrong and misleading — there is no other client. Workaround
requires direct Redis access. Storage-wise, every deleted client also leaves
behind dozens of orphan keys (training programs, nutrition logs, daily logs)
that will never be reaped.

### Suggested fix
In the DELETE branch, fetch the client first, normalize the email, and remove
the lookup. Optionally cascade through the per-client data keys. Diff sketch:

```js
if (req.method === 'DELETE') {
  var id = req.query.id || (req.body && req.body.id);
  if (!id) return res.status(400).json({ error: 'ID required' });

  var raw = await redis('GET', 'client:' + id);
  if (raw) {
    var client = JSON.parse(raw);
    if (client.email) {
      var { normalizeEmail } = require('../lib/client-auth');
      var emailKey = 'client_email:' + normalizeEmail(client.email);
      var owner = await redis('GET', emailKey);
      if (String(owner) === String(id)) await redis('DEL', emailKey);
    }
  }
  await redis('DEL', 'client:' + id);
  await redis('ZREM', 'clients_index', String(id));
  // Optional cascade: also DEL client_dailylogs_index, client_545_*, etc.
  return res.status(200).json({ ok: true });
}
```

The `owner === id` guard prevents stomping a lookup that may have been
reassigned to a different client (e.g. an admin manually fixed it).

## BUG-20260509-2: Weekly summary email always shows "Current Streak: 0 days"

- **Severity:** P1
- **File(s):** api/cron/weekly-summary.js:87-90
- **Status:** new

### What's wrong
The streak counter walks the last-7-days array starting at `d = 0` (today,
UTC) and breaks the moment a day has no daily log. The cron runs at
`0 14 * * 1` (vercel.json) — Monday 14:00 UTC, ~9-10am ET. At that hour
virtually no client has submitted today's check-in yet, so `weekData[0][0]`
is empty, the loop breaks immediately, and `streak` stays at 0 — every week,
for every client, regardless of how many consecutive prior days they logged.

### How to reproduce
Code trace:

```js
// weekly-summary.js:46-51 builds dates with d=0 = today (UTC)
for (var d = 0; d < 7; d++) {
  var dateStr = new Date(now - d * 86400000).toISOString().split('T')[0];
  logCmds.push(['GET', 'client_dailylog:' + client.id + ':' + dateStr]);
}
// ...
// weekly-summary.js:87-90 — streak loop starts at d=0
for (var d = 0; d < 7; d++) {
  if (weekData[0][d] && weekData[0][d].result) streak++;
  else break;            // <-- if today is empty, breaks at streak=0
}
```

Manual test: seed `client_dailylog:{id}:{YYYY-MM-DD}` for every day from
yesterday back through 6 days ago (and leave today empty), then invoke the
cron with the correct `Authorization: Bearer ${CRON_SECRET}` header. The
email body renders `Current Streak: 0 days` even though the real streak is
6.

### Why it matters
The Monday recap is one of two automated touchpoints clients receive
(`vercel.json` crons block). It is the visible, motivational artifact of the
whole tracking system. Showing `0 days` to clients who are actively logging
six days a week is demoralizing and erodes trust in the platform. It also
contradicts the dashboard's own streak display, making clients ask "is the
data wrong?".

### Suggested fix
Allow the streak to start from yesterday when today is unlogged (the
dashboard's own todayStr/streak logic uses the same convention). Minimal
patch:

```js
// weekly-summary.js:87-90
var startD = (weekData[0][0] && weekData[0][0].result) ? 0 : 1;
for (var d = startD; d < 7; d++) {
  if (weekData[0][d] && weekData[0][d].result) streak++;
  else break;
}
```

Note: this still under-reports for clients with streaks longer than 6 days
because the cron only fetches 7 days of logs. If accuracy matters, fetch
`ZREVRANGE client_dailylogs_index:{id} 0 30` and walk backwards from the most
recent date.

## BUG-20260509-3: Editing an ad-spend entry's date doesn't update its sort score, breaking monthly totals

- **Severity:** P2
- **File(s):** api/dashboard/adspend.js:84-105, api/dashboard/revenue.js:63-79

### What's wrong
On POST, an ad-spend entry is added to the `adspend_index` ZSET with the
date's epoch as its score (adspend.js:74). On PUT, the entry JSON is updated
in place but the ZSET score is never refreshed (adspend.js:91-100). Any later
call that relies on score-based filtering — notably revenue.js:66
(`ZRANGEBYSCORE adspend_index monthStart +inf` to fetch the current month's
spend) — sees the entry under its original date.

### How to reproduce
1. POST `/api/dashboard/adspend` with `date: "2026-01-15", spend: 500` →
   index entry scored at the Jan 15 epoch.
2. PUT `/api/dashboard/adspend` with `id: <id>, date: "2026-05-15"` → the
   entry's `date` field becomes May 15, but its index score stays at Jan 15.
3. GET `/api/dashboard/revenue` in May → `monthlyAdSpend` excludes this $500
   even though the entry now claims to be a May 15 expense.
4. Symmetric failure: an entry edited from May 15 → Jan 15 still counts in
   May's total.

### Why it matters
Ad spend is one of the dashboard's revenue/CPA inputs. Date-corrections (a
common admin action when reconciling ad invoices with the calendar month)
silently desync the index from the data, producing wrong CPA, wrong monthly
spend, wrong sort order in the Ad Spend table. Severity is P2 only because
it's admin-facing and produces analytical drift rather than a hard break.

### Suggested fix
After updating the entry on PUT, also re-`ZADD` with the new score:

```js
// adspend.js, after entry.updatedAt = ...
await redis('SET', 'adspend:' + body.id, JSON.stringify(entry));
if (body.date !== undefined) {
  var newScore = new Date(entry.date).getTime() || Date.now();
  await redis('ZADD', 'adspend_index', newScore, String(body.id));
}
```

`ZADD` updates the score for an existing member, so no separate `ZREM` is
needed.
