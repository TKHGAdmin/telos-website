## Bug 1 - P1 - Engagement-check cron spams inactive clients daily with no debounce

**ID:** BUG-2026-05-24-01
**Severity:** P1
**Area:** api/cron
**Files:** api/cron/engagement-check.js:36-94

### What's broken
The daily engagement-check cron fires a "We miss you at Telos" email to every active client
whose most recent daily log is 3 or more days old. It does not record when the last reminder
was sent. Because the cron runs every day, an inactive client receives a new reminder email
every single day until they log in.

### Reproduction
1. Provision an active client with `portalEnabled: true` and `emailEnabled !== false`.
2. Skip daily check-ins for 3 days.
3. On day 3: cron triggers `engagement-check.js`. Loop at line 36 finds `lastLogDaysAgo === 3`,
   condition at line 61 passes, email sent.
4. On day 4: still no log. `lastLogDaysAgo === 4`, condition passes, second email sent.
5. On day 5: still no log. Inner loop at lines 45-48 only checks back 5 days, so
   `lastLogDaysAgo === -1` (no log found), `daysSince = 5`, third email sent.
6. Days 6, 7, 8 ... continue forever - one email per day, every day.

Trace of the bug: `engagement-check.js` only reads from `client.id` daily logs. It never reads
or writes any `lastEngagementEmailAt` (grep confirms no such field exists anywhere in
`api/`). After the `fetch` to Resend at line 77, only `sent++` is incremented - the client
record is not updated.

### Impact
Every active client who falls off pace gets bombarded with daily "we miss you" emails. This
hurts the brand, virtually guarantees spam complaints to Resend (which can throttle or
suspend the sender), and trains clients to filter Telos email into junk. A client inactive
for one month receives ~28 reminders.

### Suggested fix
Before sending, check `client.lastEngagementEmailAt`. Skip if set within the last 7 days.
After a successful send, write `client.lastEngagementEmailAt = new Date().toISOString()` back
to `client:{id}`. Roughly:

```js
var lastSent = client.lastEngagementEmailAt ? new Date(client.lastEngagementEmailAt).getTime() : 0;
if (now - lastSent < 7 * 86400000) continue;
// ... existing send code ...
client.lastEngagementEmailAt = new Date().toISOString();
await redis('SET', 'client:' + client.id, JSON.stringify(client));
```

## Bug 2 - P2 - Quiz submit-handler attached on every retake, causing duplicate API submissions

**ID:** BUG-2026-05-24-02
**Severity:** P2
**Area:** js/quiz.js
**Files:** js/quiz.js:215, js/quiz.js:221-276, js/quiz.js:397, js/quiz.js:403

### What's broken
`setupLeadCapture()` calls `submitBtn.addEventListener('click', ...)` (line 225) every time
it is invoked. It is invoked once per quiz completion (line 215). The Retake button resets
state and DOM but does not remove the old listener, so the second time the user reaches the
lead-capture form the submit button has two click listeners attached. Three after the next
retake, four after the next, and so on. The same pattern applies to the retake button
(line 397) and the View-Pricing button (line 403) in `renderResults()`.

### Reproduction
1. Open `index.html` or `pricing.html` in a browser.
2. Click "Start Quiz", answer all 8 questions, fill in name/email/phone, click "See My Score".
   Observe one `POST /api/submit-quiz` and one `Quiz Lead Capture:` console log.
3. Click "Retake Quiz". Answer all 8 questions again. Fill in name/email/phone, click
   "See My Score".
4. Observe two `POST /api/submit-quiz` requests and two console logs from a single click.
5. Repeat retake/complete - the third pass produces three submissions per click, the fourth
   produces four (already over the per-IP rate limit of 10/hr in `api/submit-quiz.js:27`).

The handler in `setupLeadCapture` is the only call site that touches `submitBtn`, so each
attempt simply stacks another anonymous closure on top.

### Impact
- Duplicate quiz submissions corrupt the lead pipeline: a single curious user can show up as
  three or four leads, inflating dashboard counts and Resend follow-ups.
- After the 4th retake (or sooner for any user who clicks impatiently), the user trips the
  IP rate limit and subsequent submissions silently fail (`.catch(function() {})` at
  line 269). The user still sees their result client-side, but the lead is never recorded.
- localStorage `telosQuizCompleted` and `telosQuizData` are also written multiple times per
  click (cheap, but indicative of the wider double-firing problem).

### Suggested fix
Move the listener attachment out of `setupLeadCapture` into the DOMContentLoaded scope,
exactly once - and have it read `quiz.scores`/`quiz.userName` directly from the closure-scoped
`quiz` object. Same treatment for the retake and view-pricing handlers in `renderResults`.
Alternatively, use `submitBtn.onclick = ...` (single-handler assignment) instead of
`addEventListener`, which is the minimum-change fix and matches the inline-handler style
used elsewhere in `shop.js`.
