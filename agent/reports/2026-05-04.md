## Bugs

### BUG-20260504-01 — Quiz double-counts points when user clicks twice during auto-advance

- **Severity:** P2
- **Area:** `js/quiz.js:198-218`
- **Status:** new

**What's wrong**
After answering a question, `handleAnswerSelect` waits 400ms before advancing. During that window the option buttons remain enabled and the click handlers stay attached, so a second click (same option re-clicked, or a different option chosen) adds another `option.points` value to `quiz.scores[pillar]`. Result: inflated pillar scores and incorrect tier assignment.

**Repro / Evidence**
1. Open the homepage quiz (or `/pricing` overlay).
2. Click any answer; immediately click a different answer in the same question (within 400ms).
3. Both selections fire the handler — `quiz.scores[pillar] += option.points` runs twice. By Q8 the user can easily push `totalScore` above the 40 max, lifting them into the wrong `resultTier` and showing wrong "weakest pillar" copy.

The handler at `js/quiz.js:198`:
```js
function handleAnswerSelect(button, option, pillar) {
  button.style.borderColor = '#C9A84C';
  button.style.borderWidth = '2px';
  quiz.scores[pillar] += option.points;
  setTimeout(() => { ... }, 400);
}
```
Nothing disables sibling buttons or removes the listener.

**Suggested fix**
Add `if (quiz.locked) return; quiz.locked = true;` at the top of the handler and reset to `false` inside `renderQuestion()` (or call `optionsContainer.querySelectorAll('.quiz-option').forEach(b => b.disabled = true)` before the timeout).

### BUG-20260504-02 — Quiz fires duplicate /api/submit-quiz POST after every retake

- **Severity:** P2
- **Area:** `js/quiz.js:221-276` (`setupLeadCapture`)
- **Status:** new

**What's wrong**
`setupLeadCapture()` runs every time the user reaches the lead-capture step. It calls `submitBtn.addEventListener('click', ...)` against the same persistent DOM node — the previous listener is never removed. After one retake there are two listeners; after two retakes, three. Each click fires N copies of the fetch to `/api/submit-quiz`, writes to `localStorage` N times, and consumes the per-IP rate limit (10/hr) much faster than expected. The pipeline view in `/thomas` also receives duplicate submissions.

**Repro / Evidence**
1. Complete the quiz, click "See My Score". One POST is sent to `/api/submit-quiz` (verify in Network tab).
2. Click "Retake Quiz", complete again, click "See My Score". Two POSTs are sent for that single click.
3. Repeat — N+1 POSTs after N retakes.

`renderResults()` at `js/quiz.js:397` has the same shape (`retakeBtn.addEventListener('click', resetQuiz)` re-added every time results render), but `resetQuiz` is idempotent so the user-visible damage is limited to the lead-capture listener stacking.

**Suggested fix**
Either (a) clone the button to drop listeners before re-attaching:
```js
var newBtn = submitBtn.cloneNode(true);
submitBtn.parentNode.replaceChild(newBtn, submitBtn);
// then attach to newBtn
```
or (b) attach the listener once at DOMContentLoaded and read state off `quiz.*` rather than re-binding inside `setupLeadCapture`.

### BUG-20260504-03 — Weekly summary email reports streak=0 for clients with active streaks

- **Severity:** P2
- **Area:** `api/cron/weekly-summary.js:86-90`
- **Status:** new

**What's wrong**
The streak counter walks `for (var d = 0; d < 7; d++)` and breaks on the first missing daily-log key. Index `d=0` is **today's** date. The cron runs Monday at 2pm UTC / 9am ET, which is before the vast majority of clients log their Monday check-in. So `weekData[0][0].result` is almost always undefined → `break` → `streak = 0` is sent in the email even when the client logged perfectly Sun/Sat/Fri/etc.

The user-facing line in the email body:
> Current Streak: **0 days**

…directly contradicts what the client sees on their dashboard moments later, and undermines the email's motivational purpose.

**Repro / Evidence**
- File: `api/cron/weekly-summary.js`
- Loop:
  ```js
  for (var d = 0; d < 7; d++) {
    if (weekData[0][d] && weekData[0][d].result) streak++;
    else break;
  }
  ```
- Cron schedule per `CLAUDE.md`: "Monday 2pm UTC (9am ET): sends weekly summary email to each active client". 9am ET is well before the typical daily log time.
- Confirm by setting a test client with daily logs for the previous 6 days but none today; `streak` returned to `buildWeeklySummary()` is 0.

**Suggested fix**
Start the loop at `d = 1` (count consecutive logged days from yesterday backwards). Optional: also include today only if it has a log, e.g. `if (weekData[0][0]?.result) streak++; for (var d = 1; ...)`.
