## BUG-20260514-01: `thomas.jpeg` is 1.9MB / 1206x2136 — displayed at ~350px max

- **Severity**: P1
- **Area**: images
- **File(s)**:
  - `images/thomas.jpeg` (1.9MB, 1206x2136)
  - `index.html:250`
  - `chs.html:946`
- **Status**: new

### What's wrong
`images/thomas.jpeg` is a 1.9MB JPEG with intrinsic dimensions 1206x2136. The CSS rule `.about-photo { width: 100%; max-height: 350px; object-fit: cover; }` (`css/style.css`) means the image is rendered roughly 400-500px wide and at most 350px tall on desktop, and narrower on mobile. The browser still downloads the full 1.9MB regardless. This single asset is ~65% of the total `images/` directory weight (2.9MB) and is loaded on the two most public-facing pages: the homepage (`index.html`) and the Charleston landing page (`chs.html`).

### How to reproduce
1. `du -h images/thomas.jpeg` → 1.9M.
2. `file images/thomas.jpeg` → reports `1206x2136`.
3. Open homepage on a throttled "Slow 3G" profile in DevTools and watch `thomas.jpeg` dominate the resource waterfall.

### Impact
Mobile users on cellular pay for ~1.7MB of wasted bytes per visit to the homepage or Charleston page (the largest possible useful render is ~700px wide, so the file should be <250KB). Charleston is application-gated and currently the primary in-person funnel — slow LCP there directly affects conversion. Homepage LCP and INP also degrade on lower-end devices.

### Suggested fix
Re-export at a sensible web size (1000px wide @ quality 82 should land near 150-200KB), keep the same path so HTML/CSS need no changes. Optionally provide a `<picture>` with a WebP source for further savings. Same treatment for `thomas-speaking-wide.jpeg` (248KB) and `thomas-speaking-close.jpeg` (212KB), which are also displayed in small grid cells.

## BUG-20260514-02: Chart.js loaded render-blocking in client-dashboard `<head>`

- **Severity**: P2
- **Area**: client-dashboard
- **File(s)**: `client-dashboard.html:21`
- **Status**: new

### What's wrong
The Chart.js v4 UMD bundle (`https://cdn.jsdelivr.net/npm/chart.js@4/dist/chart.umd.min.js`, ~200KB minified) is referenced in `<head>` with no `defer` or `async` attribute, so the browser blocks HTML parsing until it downloads and executes. Chart.js is only used to render the weekly nutrition trends chart on the Nutrition tab — it is not needed for the login form, the Home tab, the FAB, or anything above the fold on first paint. There is also no `preconnect` to `cdn.jsdelivr.net`, so the connection setup adds to the blocking time.

### How to reproduce
1. Open `client-dashboard.html` and look at line 21 — bare `<script src="...chart.umd.min.js"></script>` in `<head>`.
2. In a Lighthouse mobile run against `/client-dashboard`, "Eliminate render-blocking resources" will list this script.

### Impact
The client portal is a PWA — first paint on cellular while signing in includes a ~200KB blocking download from a third-party CDN. This affects every client load (logged-in returners and new logins both block on it).

### Suggested fix
Add `defer` to the tag: `<script src="https://cdn.jsdelivr.net/npm/chart.js@4/dist/chart.umd.min.js" defer></script>`. The chart on the Nutrition tab is built inside JS handlers that don't run until after `DOMContentLoaded` anyway, so `defer` is safe. Optionally also add `<link rel="preconnect" href="https://cdn.jsdelivr.net" crossorigin>` next to the existing font preconnect at line 18. Even better: lazy-load Chart.js only when the user opens the Nutrition tab (`import()` or dynamic `<script>` injection on first activation).

## BUG-20260514-03: Images site-wide lack `loading="lazy"` and `width`/`height`

- **Severity**: P2
- **Area**: multiple (public marketing pages)
- **File(s)**:
  - `index.html:35`, `index.html:250`, `index.html:256`, `index.html:259`
  - `chs.html:824`, `chs.html:922`, `chs.html:925`, `chs.html:928`, `chs.html:946`, `chs.html:950`, `chs.html:953`
- **Status**: new

### What's wrong
A repo-wide grep finds 12 `<img>` tags across public HTML and only 1 uses `loading="lazy"` (the hero on `chs.html:824` correctly uses `loading="eager"` plus `fetchpriority="high"`, which is appropriate for an LCP image — so the gap is on the below-the-fold images). Zero `<img>` tags carry explicit `width`/`height` attributes. Below-the-fold candidates that should defer: `index.html:256` and `:259` (secondary about photos), `chs.html:922-928` (events grid), `chs.html:946-953` (founder photos).

### How to reproduce
```
grep -rn 'loading="lazy"' --include="*.html" . | wc -l   # → 1
grep -rn '<img' --include="*.html" . | wc -l             # → 12
grep -rnE '<img[^>]+(width|height)=' --include="*.html" . | wc -l  # → 0
```

### Impact
- Below-the-fold images are downloaded during the initial page load, competing with the LCP hero for bandwidth. On Charleston specifically, four event/founder images load eagerly even though they sit far below the application form.
- Missing intrinsic dimensions cause Cumulative Layout Shift as each image resolves — visible jump on the about section of the homepage. CLS is a Core Web Vitals metric that affects SEO.

### Suggested fix
Add `loading="lazy" decoding="async"` to every `<img>` that is not the hero or above-the-fold LCP candidate. Add `width="..." height="..."` matching the source image's aspect ratio (the browser uses the ratio, not the px values, to reserve space when CSS makes the image fluid). One-pass edit covers all 10 offending tags.

## BUG-20260514-04: Weekly-summary cron sends Resend emails one-at-a-time

- **Severity**: P3
- **Area**: api/cron
- **File(s)**: `api/cron/weekly-summary.js:36-115`
- **Status**: new

### What's wrong
The weekly cron iterates `clients` with a classic `for` loop and `await fetch('https://api.resend.com/emails', ...)` inside the loop body (line 98). Each email send is fully sequential. The Vercel function budget is `maxDuration: 60` (`vercel.json:13-17`). With Resend round-trips typically 400-1500ms each, the function can complete at most ~40-150 sends per invocation before hitting the timeout. Past that, the last clients simply never receive the summary (no retry, no resumption — the cron only runs weekly).

The Redis read for that client's week-data on lines 53-57 is correctly parallelized; only the email send is sequential.

### How to reproduce
1. Read `api/cron/weekly-summary.js` lines 36-115 — `await fetch(...)` is inside the `for` loop over `clients`.
2. Check `vercel.json` — `maxDuration: 60`.
3. Estimate: at ~1s per Resend POST, the function silently truncates the recipient list beyond ~50 clients.

### Impact
Today the active client roster is small enough that this hasn't caused visible breakage, but the failure mode is a silent partial send — there is no log of which clients were skipped, no resume on the next run (the cron is weekly), and the issue scales linearly with client count. Worth fixing before the roster grows past ~30-40 active clients.

### Suggested fix
Replace the per-client `await fetch` with a batched approach. Cheapest: collect a list of email payloads and `Promise.all` them at the end of each batch of ~10 clients (controlled concurrency keeps Resend happy). Cleaner: use Resend's batch endpoint (`POST /emails/batch`) which accepts up to 100 emails in one request. Either change cuts wall-clock time by >10x and removes the truncation risk inside the 60s budget.
