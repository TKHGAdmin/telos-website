# Telos Bug Hunter Report - 2026-04-20

**Focus:** Functional
**Bugs found:** 3
**Summary:** The engagement-check cron will spam any inactive client with a reminder email every day, and two adjacent data-lifecycle bugs (dangling email lookup on client delete, bad streak math in the weekly recap) mean new clients and weekly summaries misbehave in predictable ways.

## Bugs

### [P1] engagement-check cron sends reminder email every day a client is inactive

**ID:** 2026-04-20-01
**Severity:** P1
**Area:** api/cron/engagement-check.js
**Status:** new

**What's wrong:**
The engagement reminder cron runs daily (`"0 15 * * *"` in `vercel.json`) and emails any active, portal-enabled client whose most recent daily log is 3+ days old. There is no throttle or "last reminder sent" state, so once a client becomes inactive, they receive the "We miss you at Telos" email every single day until they log a check-in. For a client who drops off for a week, that is seven identical emails.

**Where:**
- `api/cron/engagement-check.js:61` - the `lastLogDaysAgo >= 3` branch triggers a send with no idempotency gate
- `vercel.json:24-26` - cron is scheduled daily, confirming the spam cadence
- `api/cron/engagement-check.js:36-95` - no read or write of a "last reminder at" field on the client record

**Repro / evidence:**
1. Create an active, portal-enabled client with `emailEnabled !== false`.
2. Do not create any `client_dailylog:{id}:{date}` entries for the last 5 days.
3. Invoke `/api/cron/engagement-check` with `Authorization: Bearer $CRON_SECRET` once per simulated day. Each invocation enters the `lastLogDaysAgo === -1` branch and calls Resend - one email per run, indefinitely.

The handler never writes anything to Redis after sending, so there is no record of a prior reminder to gate on.

**Suggested fix:**
Before sending, read a new field such as `client.lastEngagementEmailAt` (or a dedicated key `client_last_engagement:{id}`) and skip if it was set within the last N days (e.g. 3 or 7). Persist the timestamp on successful send. Alternatively, only send on specific thresholds (day 3, day 7, day 14) by checking `lastLogDaysAgo` values explicitly rather than `>= 3`.

### [P2] Deleting a client leaves the email-to-id lookup behind, blocking email reuse

**ID:** 2026-04-20-02
**Severity:** P2
**Area:** api/dashboard/clients.js
**Status:** new

**What's wrong:**
The admin `DELETE /api/dashboard/clients` handler removes `client:{id}` and the `clients_index` ZSET entry but does not remove the `client_email:{normalizedEmail}` lookup that was created when the portal was enabled. The create path at line 70-73 blocks any POST whose normalized email already exists in `client_email:*`, so once a portal-enabled client is deleted, their email address is effectively permanently reserved - the admin cannot re-add them or add a different client under that address without hand-editing Redis.

**Where:**
- `api/dashboard/clients.js:143-154` - DELETE handler only touches `client:{id}` and `clients_index`; never reads the record first and never DELs `client_email:*`
- `api/dashboard/clients.js:70-73` - POST rejects any reused email, so the orphan causes a real user-facing block
- `api/dashboard/client-portal.js:117` - the orphaned key is created here when portal is enabled, and only cleared on explicit disable (line 123), never on client delete

**Repro / evidence:**
1. Create a client (POST `/api/dashboard/clients`) with `email: "test@example.com"`.
2. Enable portal (POST `/api/dashboard/client-portal` with `enabled: true`). This sets `client_email:test@example.com -> {id}`.
3. Delete the client (DELETE `/api/dashboard/clients?id={id}`).
4. Try to re-create: POST `/api/dashboard/clients` with the same email. The response is 400 `"Email already in use by another client"` even though no such client exists. Login attempts with that email return 401 because `client:{id}` is gone (login.js:29-32), so the lookup is pure orphan state.

**Suggested fix:**
In the DELETE branch, read the client record first, and if `email` and `portalEnabled` are set, `DEL client_email:{normalizeEmail(email)}` in the same step. Suggested shape:

```js
var raw = await redis('GET', 'client:' + id);
if (raw) {
  var { normalizeEmail } = require('../lib/client-auth');
  var c = JSON.parse(raw);
  if (c.email) await redis('DEL', 'client_email:' + normalizeEmail(c.email));
}
await redis('DEL', 'client:' + id);
await redis('ZREM', 'clients_index', String(id));
```

Cleaning up the per-client content keys (training, nutrition, 545, supplements, logs, push sub) is a separate tech-debt consideration and is not required for this bug.

### [P2] Weekly summary streak counts from today, which is almost always empty at 10am Monday

**ID:** 2026-04-20-03
**Severity:** P2
**Area:** api/cron/weekly-summary.js
**Status:** new

**What's wrong:**
The weekly recap email includes a "Current Streak" figure calculated by walking the last seven daily logs starting at `d = 0` (today) and stopping at the first missing day. The cron fires Monday 14:00 UTC (~10am ET in DST), which is before most users have logged their Monday check-in. As a result, almost every recipient's email shows `Current Streak: 0 days` regardless of how consistent they were through the prior week, turning a positive recap into a demotivator.

**Where:**
- `api/cron/weekly-summary.js:87-90` - loop starts at `d = 0` (today) and breaks on the first missing log
- `api/cron/weekly-summary.js:46-47` - `logCmds[0]` corresponds to the send-day itself, not the most recent completed day
- `vercel.json:19-22` - schedule `"0 14 * * 1"` confirms the run time is well before ET morning check-ins

**Repro / evidence:**
1. Seed a client with `client_dailylog:{id}:{date}` entries for each of the previous six days but nothing for today.
2. Trigger `/api/cron/weekly-summary` with the cron secret.
3. Inspect the sent email HTML: `stats.streak` is `0` because `weekData[0][0].result` is falsy, so the loop breaks on the first iteration. The same client would correctly read as a 6-day streak if the loop began at `d = 1`.

**Suggested fix:**
Skip today when today is empty, or equivalently count `d >= 1` if today is unlogged:

```js
var startIdx = (weekData[0][0] && weekData[0][0].result) ? 0 : 1;
for (var d = startIdx; d < 7; d++) {
  if (weekData[0][d] && weekData[0][d].result) streak++;
  else break;
}
```

Same fix applies symmetrically to `daysLogged` / `workouts` if Thomas wants those to report the completed week rather than the rolling 7-day window; current behavior there is defensible, so only streak strictly needs the change.
