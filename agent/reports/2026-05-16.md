## BUG-001: Enabling client portal overwrites existing email lookup, locking out original owner

- **Severity:** P1
- **Area:** Functional / Auth
- **Files:** `api/dashboard/client-portal.js:115-118`, `api/dashboard/clients.js:70-73`
- **Status:** New

### Reproduction (code trace)

1. Admin creates Client A with `email = foo@bar.com` via `POST /api/dashboard/clients`. `clients.js:70` checks `client_email:foo@bar.com` - the key does not exist (no portal-enabled client owns it yet) - so creation succeeds. **No `client_email:` lookup is written for Client A** (clients.js POST never writes it; only the portal-enable path does).
2. Admin creates Client B, also with `email = foo@bar.com`. Same check at `clients.js:70` passes for the same reason. Two clients now share an email.
3. Admin enables portal for Client A via `POST /api/dashboard/client-portal { clientId: A, enabled: true, password: ... }`. `client-portal.js:115-118` runs:
   ```js
   if (client.portalEnabled && client.email) {
     var email = normalizeEmail(client.email);
     await redis('SET', 'client_email:' + email, String(client.id));
   }
   ```
   `client_email:foo@bar.com` now points to A.
4. Admin enables portal for Client B with the same call. The same block runs and **unconditionally overwrites** the lookup, so `client_email:foo@bar.com` now points to B.
5. Client A attempts to log in via `POST /api/client/login`. `login.js:24` does `redis('GET', 'client_email:foo@bar.com')`, which returns B's id. The login flow then loads B's record and tries to verify A's password against B's hash. Login fails with "Invalid email or password" - Client A is permanently locked out from their portal even though their record still exists.

### Impact

Any time the admin re-uses an email across clients (intentional or accidental - e.g., a returning client who was archived and re-added) the second portal enablement silently kicks the first user off the portal. The first client cannot recover by password reset either: `reset-password.js:22` uses the same email lookup, so the reset link goes to B's email-on-file, not A's portal.

### Suggested fix

Before the `SET` in `client-portal.js:115-118`, check existing ownership:

```js
if (client.portalEnabled && client.email) {
  var email = normalizeEmail(client.email);
  var existingOwner = await redis('GET', 'client_email:' + email);
  if (existingOwner && String(existingOwner) !== String(client.id)) {
    return res.status(400).json({ error: 'Another client already owns this email - resolve before enabling portal' });
  }
  await redis('SET', 'client_email:' + email, String(client.id));
}
```

Optionally also write the email lookup at client creation (`clients.js` POST) so the uniqueness check at creation actually catches duplicates regardless of portal state.

## BUG-002: Engagement-check cron re-sends "we miss you" email every day to the same inactive client

- **Severity:** P1
- **Area:** Functional / Email
- **Files:** `api/cron/engagement-check.js:61-93`, `vercel.json:25`
- **Status:** New

### Reproduction (code trace)

The cron schedule in `vercel.json:23-26` runs `engagement-check` daily at 15:00 UTC (`0 15 * * *`).

`engagement-check.js:60-94` sends an email whenever the client has no daily log in the past 3+ days. There is no record of prior sends - no `last_engagement_email:{clientId}` flag, no `engagement_sent:{clientId}:{date}` SET, no cooldown check. So:

- Day 1 of inactivity: no email (under 3-day threshold)
- Day 2 of inactivity: no email
- Day 3 of inactivity: email sent
- Day 4 of inactivity: email sent again
- Day 5+: email sent every day for as long as the client stays inactive

A client on vacation for 10 days returns to 8 identical "We miss you at Telos" emails in their inbox. This is real user-visible spam and a likely complaint/unsubscribe trigger.

The weekly-summary cron (`weekly-summary.js`) does not have this problem - it only fires once a week by schedule, not by client state.

### Impact

Any active-but-inactive client (vacation, illness, travel, paused engagement) receives daily re-engagement spam. Especially bad for clients on the `inactive` status branch (line 38 filters those out, so paused clients are safe) but for `active` clients who simply skip a week, the inbox flood will damage trust in the brand.

### Suggested fix

Add a sent-marker with a cooldown. Minimum viable:

```js
// Before sending
var lastSentRaw = await redis('GET', 'engagement_sent:' + client.id);
var lastSent = lastSentRaw ? parseInt(lastSentRaw, 10) : 0;
var THREE_DAYS_MS = 3 * 86400000;
if (now - lastSent < THREE_DAYS_MS) continue;

// After sending
await redis('SETEX', 'engagement_sent:' + client.id, 14 * 86400, String(now));
```

Or simpler: only send if `lastLogDaysAgo === 3 || lastLogDaysAgo === 7 || lastLogDaysAgo === 14` (specific reminder cadence).

## BUG-003: Quiz auto-advance does not block multi-clicks - rapid clicks inflate pillar scores and skip questions

- **Severity:** P2
- **Area:** Functional / UX
- **File:** `js/quiz.js:198-218`
- **Status:** New

### Reproduction

Open `/` or `/pricing` and start the Execution Score Quiz. On any question:

1. Click answer A (e.g., the 3-point option).
2. Within 400ms, click answer D (the 5-point option) on the same question.

Expected: only one answer is recorded; the next question loads once.

Actual:
- Both clicks call `handleAnswerSelect`, which does `quiz.scores[pillar] += option.points` (line 204). The current question's pillar receives `3 + 5 = 8` points instead of one option's value.
- Both clicks schedule independent `setTimeout(..., 400)` callbacks (lines 207-217). Each increments `quiz.currentQuestion` and calls `renderQuestion()`. The user lands on question N+2 instead of N+1, skipping a question entirely (its options are never rendered and its pillar gets 0 points).

The current code has no `pointer-events: none`, `button.disabled = true`, debounce, or `clearTimeout(prev)` guard.

### Impact

- Inflated total scores - the quiz's purpose is qualifying leads and showing the right pricing tier. A user double-tapping (common on mobile, or just an impatient click) can score ~10 points higher than they should and see "Executing" tier results when they're really "Almost Dialed".
- Skipped questions mean missing pillar data, which produces the wrong "weakest link" callout (`getWeakestPillar()`) and an inaccurate breakdown bar.
- Submission stored in Redis (`/api/submit-quiz`) is wrong, polluting lead data.

### Suggested fix

In `handleAnswerSelect` (line 198), guard against re-entry:

```js
function handleAnswerSelect(button, option, pillar) {
  if (quizQuestionCard.dataset.locked === '1') return;
  quizQuestionCard.dataset.locked = '1';
  button.style.borderColor = '#C9A84C';
  button.style.borderWidth = '2px';
  quiz.scores[pillar] += option.points;
  setTimeout(() => {
    quizQuestionCard.dataset.locked = '0';
    if (quiz.currentQuestion < 7) { ... } else { ... }
  }, 400);
}
```

Or set `optionsContainer.style.pointerEvents = 'none'` at the start of `handleAnswerSelect` and re-enable in `renderQuestion()`.

## BUG-004: Weekly summary email caps current streak at 7 days

- **Severity:** P3
- **Area:** Functional / Email accuracy
- **File:** `api/cron/weekly-summary.js:86-90`
- **Status:** New

### Reproduction (code trace)

The "Current Streak" stat sent to clients in the weekly recap email is computed in lines 86-90:

```js
// Count streak (from today backwards)
for (var d = 0; d < 7; d++) {
  if (weekData[0][d] && weekData[0][d].result) streak++;
  else break;
}
```

The loop is bounded by `d < 7` because `logCmds` only fetched 7 days of daily logs (lines 46-51, `for (var d = 0; d < 7; d++)`). A client who has logged every day for 30 days will see `streak = 7` in their email instead of `30`.

### Impact

Low - the email still correctly recognizes the client logged every day this week, but the streak number undercounts long-term consistency. For Telos's brand of "show clients their execution data", this misreports a motivating metric. A client on a 60-day streak who sees "Current Streak: 7 days" will rightfully think the system is broken.

### Suggested fix

Either:
1. Query the streak from `client_dailylogs_index:{clientId}` ZSET (which holds all logged dates) and count contiguous days backwards from today. This is a single ZREVRANGE call + a linear scan.
2. Fetch a larger window (e.g., 60 or 90 days) of daily logs specifically for the streak count.

Option 1 is cheaper and correct for arbitrary streak lengths.
